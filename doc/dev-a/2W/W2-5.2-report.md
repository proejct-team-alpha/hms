### 2. ReservationService — cancelReservation 반환 타입 변경

취소 완료 페이지에서 예약 정보를 표시하기 위해 `void` → `ReservationCompleteInfo` 반환으로 변경.
취소 처리 전에 예약 정보를 미리 저장하여 반환한다.

```java
@Transactional
public ReservationCompleteInfo cancelReservation(Long id) {
    Reservation reservation = reservationRepository.findById(id)
            .orElseThrow(() -> new IllegalArgumentException("예약을 찾을 수 없습니다."));

    // 취소 전에 정보 저장
    ReservationCompleteInfo info = new ReservationCompleteInfo(
            reservation.getReservationNumber(),
            reservation.getPatient().getName(),
            reservation.getDepartment().getName(),
            reservation.getDoctor().getStaff().getName(),
            reservation.getReservationDate().toString(),
            reservation.getTimeSlot()
    );

    reservation.cancel();
    return info;
}
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: 예약을 취소하기 전에 예약 정보를 먼저 저장해두고, 취소 처리 후 저장해둔 정보를 반환합니다. 취소 완료 화면에서 취소된 예약 정보를 표시하는 데 사용됩니다.
> - **왜 이렇게 썼는지**: `reservation.cancel()`을 호출하면 상태가 CANCELLED로 바뀌어 이후 정보 접근 시 문제가 생길 수 있습니다. 따라서 취소 전에 미리 정보를 DTO에 담아두고 나서 취소를 실행합니다.
> - **쉽게 말하면**: 예약 취소서류를 파쇄하기 전에 복사본을 먼저 만들어두고, 원본을 파쇄한 뒤 복사본을 돌려주는 과정입니다.

### 3. ReservationController — 완료 페이지 엔드포인트 추가 및 PathVariable 수정

#### 문제: PathVariable 바인딩 실패

Spring Boot는 컴파일 시 `-parameters` 플래그 없이는 파라미터 이름을 인식하지 못한다.
`@PathVariable Long id`만으로는 `{id}`와 매핑되지 않아 `IllegalArgumentException` 발생.

#### 해결: @PathVariable에 명시적 이름 지정

```java
// 변경 전
@PostMapping("/modify/{id}")
public String modifyReservation(@PathVariable Long id, ...) { }

// 변경 후
@PostMapping("/modify/{id}")
public String modifyReservation(@PathVariable("id") Long id, ...) { }
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: URL 경로의 `{id}` 값을 메서드 파라미터 `id`에 연결합니다. `@PathVariable("id")`처럼 이름을 명시해야 올바르게 동작합니다.
> - **왜 이렇게 썼는지**: Java 컴파일러는 기본적으로 `-parameters` 플래그 없이 빌드하면 파라미터 이름 정보를 제거합니다. 이름을 명시하지 않으면 Spring이 `{id}`와 `Long id`를 연결하지 못해 오류가 발생합니다.
> - **쉽게 말하면**: URL에 있는 번호판(`{id}`)을 메서드가 받을 변수(`id`)에 "이 번호판이 이 변수야"라고 직접 연결해주는 것입니다.

#### 예약 변경 완료 엔드포인트

```java
@PostMapping("/modify/{id}")
public String modifyReservation(@PathVariable("id") Long id,
                                @ModelAttribute ReservationUpdateForm form,
                                RedirectAttributes redirectAttributes) {
    ReservationCompleteInfo info = reservationService.updateReservation(id, form);
    redirectAttributes.addAttribute("reservationNumber", info.getReservationNumber());
    redirectAttributes.addAttribute("name",       info.getPatientName());
    redirectAttributes.addAttribute("department", info.getDepartmentName());
    redirectAttributes.addAttribute("doctor",     info.getDoctorName());
    redirectAttributes.addAttribute("date",       info.getReservationDate());
    redirectAttributes.addAttribute("time",       info.getTimeSlot());
    return "redirect:/reservation/modify-complete";
}

@GetMapping("/modify-complete")
public String modifyComplete(HttpServletRequest request) {
    request.setAttribute("pageTitle", "예약 변경 완료");
    return "reservation/reservation-modify-complete";
}
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: 예약 변경 폼을 POST로 처리한 뒤 변경 완료 페이지로 리다이렉트합니다. 변경 완료 페이지 접속 시(GET)에는 페이지 제목만 설정하고 화면을 반환합니다.
> - **왜 이렇게 썼는지**: PRG 패턴으로 POST 처리 후 GET 완료 페이지로 이동합니다. `@ModelAttribute`는 폼 데이터를 `ReservationUpdateForm` 객체에 자동 바인딩합니다. `RedirectAttributes`로 완료 정보를 URL 파라미터로 전달합니다.
> - **쉽게 말하면**: 예약을 변경하고 나서 "변경 완료 페이지로 이동하세요"라는 안내장을 주고, 변경된 예약 정보는 안내장 뒤에 메모해서 함께 보냅니다.

#### 예약 취소 완료 엔드포인트

```java
@PostMapping("/cancel/{id}")
public String cancelReservation(@PathVariable("id") Long id,
```

> **💡 입문자 설명**

> - **왜 이렇게 썼는지**: 취소 처리(`cancelReservation()`)에서 반환된 `ReservationCompleteInfo`를 `RedirectAttributes`에 담아 완료 페이지에서 표시할 수 있도록 합니다. PRG 패턴으로 중복 취소 제출을 방지합니다.
> - **쉽게 말하면**: 예약을 취소한 뒤 "취소 완료 페이지로 이동하세요"라는 안내장을 주고, 취소된 예약 정보는 참고용으로 뒤에 적어 보냅니다.

   ```html
   <form id="reservation-form" method="post" action="/reservation/modify/{{id}}" class="space-y-6">

     <input type="hidden" name="{{_csrf.parameterName}}" value="{{_csrf.token}}">
     <input type="hidden" name="patientId" value="{{patientId}}">
     ...
   </form>
   ```

### 5. reservation-cancel.mustache — CSRF 토큰 추가

#### 문제: 403 Forbidden 에러

Spring Security의 CSRF 보호 기능으로 인해 POST 요청 시 토큰이 없으면 403 에러 발생.

#### 해결: CSRF 토큰 추가

```html
<form method="post" action="/reservation/cancel/{{id}}" class="flex-1">
  {{! CSRF 토큰 }}
  <input type="hidden" name="{{_csrf.parameterName}}" value="{{_csrf.token}}">
  <button type="submit" class="w-full py-3 bg-red-500 text-white ...">
    예약 취소
  </button>
</form>
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: 예약 취소 버튼이 있는 폼으로, 클릭 시 해당 예약 ID로 POST 요청을 보냅니다. CSRF 토큰을 포함하여 보안 검증을 통과합니다.
> - **왜 이렇게 썼는지**: 취소 버튼을 `<a>` 태그 대신 `<form>` 안의 `<button type="submit">`으로 만든 것은, 데이터를 변경하는 작업(취소)은 GET이 아닌 POST로 처리해야 하는 HTTP 표준 때문입니다. CSRF 토큰 없이 POST 요청을 보내면 Spring Security가 403(Forbidden) 오류를 반환합니다.
> - **쉽게 말하면**: 취소 버튼을 링크(GET) 대신 폼 제출(POST)로 만들어, 위조 방지 스티커(CSRF 토큰)를 붙여서 서버에 요청하는 것입니다.

### 6. reservation-modify-complete.mustache (신규 생성)

예약 변경 완료 전용 페이지. 예약 완료 페이지와 동일한 구조이지만 **파란색 체크 아이콘**과
"예약이 변경되었습니다!" 메시지로 변경 작업을 강조한다.

#### 주요 특징

- **파란색 체크 아이콘**: 변경 완료를 시각적으로 표현
- **변경된 예약 정보 표시**: 예약번호, 환자명, 진료과/전문의, 예약 일자, 예약 시간
- **버튼**:
  - "메인으로 가기": 홈페이지로 이동
  - "예약 조회": 해당 예약 조회 페이지로 이동

```html
<div class="w-20 h-20 bg-blue-100 rounded-full flex items-center justify-center mx-auto mb-6">
  <i data-feather="check-circle" class="w-10 h-10 text-blue-600"></i>
</div>

<h1 class="text-3xl font-bold text-slate-800 mb-2">예약이 변경되었습니다!</h1>
<p class="text-slate-500 mb-10">
  <span id="res-name-title" class="font-semibold"></span>님의 예약이 성공적으로 변경되었습니다.<br/>
  예약 당일 원무과에서 접수를 진행해주세요.
</p>
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: 예약 변경 완료 페이지의 헤더 영역으로, 파란색 체크 아이콘과 "예약이 변경되었습니다!" 메시지를 표시합니다.
> - **왜 이렇게 썼는지**: `data-feather="check-circle"`은 feather-icons 라이브러리의 체크 원형 아이콘을 사용합니다. Tailwind CSS의 `bg-blue-100`, `text-blue-600`으로 파란색 테마를 표현하여 "변경 완료"를 직관적으로 인식하도록 합니다.
> - **쉽게 말하면**: 변경 완료 화면에 파란색 체크 도장을 찍어 "성공적으로 변경되었습니다"를 시각적으로 표현하는 것입니다.

> **💡 입문자 설명**
> - **이 코드가 하는 일**: Tailwind CSS를 재빌드하여 새 HTML 템플릿에서 사용한 CSS 클래스들이 최종 CSS 파일에 포함되도록 합니다.

> - **쉽게 말하면**: 새 옷(HTML 템플릿)을 만들었으니, 그 옷에 맞는 옷감(CSS)을 새로 짜는 과정입니다.

## 해결한 버그 및 이슈

### 1. PathVariable 바인딩 실패 (IllegalArgumentException)

**증상**:
```
java.lang.IllegalArgumentException: Name for argument of type [java.lang.Long] not specified,
and parameter name information not available via reflection.
Ensure that the compiler uses the '-parameters' flag.
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: Spring이 `@PathVariable` 이름을 자동으로 인식하지 못할 때 발생하는 오류 메시지입니다.
> - **왜 이렇게 썼는지**: 이 오류는 컴파일 시 `-parameters` 플래그가 없어서 파라미터 이름 정보가 삭제되었기 때문에 발생합니다. 해결 방법은 `@PathVariable("id")`처럼 이름을 명시하는 것입니다.
> - **쉽게 말하면**: 이름표가 없는 택배를 받았을 때 "이게 누구 것인지 모르겠다"는 오류입니다. 이름표(`"id"`)를 직접 붙여주면 해결됩니다.

**원인**:
Spring Boot는 기본적으로 `-parameters` 플래그 없이 컴파일되므로 파라미터 이름을 리플렉션으로 얻을 수 없다.
`@PathVariable Long id`만으로는 URL의 `{id}`와 매핑되지 않음.

**해결**:
`@PathVariable("id")`로 명시적 이름 지정.

---

### 2. CSRF 토큰 누락 (403 Forbidden)

**증상**:
```json
{
  "success": false,
  "errorCode": "INTERNAL_ERROR",
  "message": "서버 오류가 발생했습니다.",
  "timestamp": "2026-03-11T06:22:55.794546300Z",
  "path": "/error/403"
}
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: CSRF 토큰이 없는 POST 요청에 대해 서버가 반환하는 403 오류 응답입니다.
> - **왜 이렇게 썼는지**: HTTP 403은 "접근 금지(Forbidden)" 오류로, Spring Security가 CSRF 토큰 없는 POST 요청을 차단할 때 발생합니다. JSON 형태로 응답하는 이유는 이 프로젝트의 전역 오류 처리 설정 때문입니다.
> - **쉽게 말하면**: 위조 방지 스티커(CSRF 토큰) 없이 제출한 서류가 보안 검사에서 거부된 것입니다.

**원인**:
Spring Security의 CSRF 보호 기능이 활성화되어 있지만,
`reservation-cancel.mustache`와 `reservation-modify.mustache` 폼에 CSRF 토큰이 없음.

**해결**:
모든 POST 폼에 CSRF 토큰 hidden input 추가.

```html
<input type="hidden" name="{{_csrf.parameterName}}" value="{{_csrf.token}}">
```

> **💡 입문자 설명**
> - **이 코드가 하는 일**: 모든 POST 폼에 CSRF 보안 토큰을 숨겨진 필드로 추가합니다.
> - **왜 이렇게 썼는지**: Spring Security는 POST 요청 시 이 토큰을 확인하여 요청이 실제 사이트에서 온 것인지 검증합니다. `{{_csrf.parameterName}}`과 `{{_csrf.token}}`은 Mustache에서 Spring Security가 제공하는 CSRF 정보를 자동으로 삽입합니다.
> - **쉽게 말하면**: 폼 제출 시 "이건 우리 사이트에서 보내는 진짜 요청입니다"라는 비밀 코드를 함께 보내는 것입니다.

---

### 3. Tailwind CSS 클래스 미적용 (투명한 버튼)

**증상**:
예약 조회 페이지의 "조회하기" 버튼이 투명하게 표시됨.

**원인**:
Tailwind CSS가 재빌드되지 않아 새로운 HTML의 클래스가 CSS 파일에 포함되지 않음.

**해결**:
`npm run build:css` 실행하여 Tailwind CSS 재생성.
