## W3-1: AI 증상 추천 — 키워드 매핑 (더미)

### 배운 개념

**더미(Dummy) 구현**: 실제 AI API 없이 키워드 테이블로 흉내 내기

```javascript
// SYMPTOM_MAP: 증상 키워드 → 진료과 매핑 테이블
const SYMPTOM_MAP = [
    {
        dept: '내과',
        doctorName: '이민준',
        keywords: ['열', '기침', '복통', '구토'],
        recTime: '09:00'
    },
    {
        dept: '정형외과',
        doctorName: '박서준',
        keywords: ['허리', '무릎', '어깨', '골절'],
        recTime: '14:00'
    }
    // ...
];

function analyzeSymptom(text) {
    for (const entry of SYMPTOM_MAP) {
        if (entry.keywords.some(kw => text.includes(kw))) {
            return entry;  // 첫 번째 매칭 진료과 반환
        }
    }
    return null;
}
```

> **왜 더미로 먼저 만드나?**
> 실제 AI API 연결(W4)이 완성되기 전에 화면 흐름을 먼저 구성하기 위해.
> "진짜 대신 가짜로 동작하는지 먼저 확인" → 나중에 진짜로 교체

## W3-2: 비동기 처리 — async/await, Promise

### 배운 개념

**동기 vs 비동기:**

```
동기 방식: 서버 응답이 올 때까지 화면이 멈춤 (사용자가 기다려야 함)
비동기 방식: 서버에 요청하고 기다리는 동안 다른 것 처리 → 응답 오면 처리
```

**Promise**: "나중에 값이 생길 것"을 나타내는 객체

```javascript
// 기존 방식 (setTimeout으로 흉내)
function callSymptomApi(symptomText) {
    return new Promise((resolve) => {
        setTimeout(() => {
            resolve(analyzeSymptom(symptomText));  // 1.5초 후 결과 반환
        }, 1500);
    });
}

// 호출 측에서 async/await으로 사용
async function onAnalyzeClick() {
    const result = await callSymptomApi(symptomInput.value);
    // result가 올 때까지 기다렸다가 처리
    showResult(result);
}
```

> **왜 이 구조를 만드나?**
> W4에서 실제 AI API로 교체할 때 `callSymptomApi()` 내부만 바꾸면 됨.
> 호출하는 코드는 그대로 유지. → **변경에 강한 구조**

## W3-3: 추천 시간 표시 — UI 구성

### 배운 개념

## W3-4: 추천 결과 → 예약 폼 자동 채우기

### 배운 개념

**URL 파라미터로 데이터 전달:**

```javascript
// AI 추천 결과를 URL에 담아서 예약 폼 페이지로 이동
function proceedToDirect(result) {
    const params = new URLSearchParams({
        dept: result.dept,
        doctor: result.doctorName,
        time: result.recTime
    });
    window.location.href = '/reservation/direct-reservation?' + params;
}
// 결과: /reservation/direct-reservation?dept=내과&doctor=이민준&time=09:00
```

**예약 폼 페이지에서 URL 파라미터 읽어서 자동 선택:**

```javascript
const params = new URLSearchParams(window.location.search);
const recDept = params.get('dept');   // "내과"
const recDoctor = params.get('doctor'); // "이민준"
const recTime = params.get('time');   // "09:00"

// 진료과 자동 선택
departmentSelect.value = DEPT_ID_MAP[recDept];  // 이름 → ID 변환 필요

// 의사 목록을 API로 가져온 후 해당 의사 자동 선택
const doctors = await fetchDoctors(deptId);
const matched = doctors.find(d => d.name === recDoctor);
if (matched) doctorSelect.value = matched.id;
```

> **DEPT_ID_MAP이 필요한 이유:**
> URL에는 "내과"라는 이름이 있지만, `<select>` 드롭다운의 value는 숫자 ID.
> 이름 → ID 변환 테이블이 없으면 자동 선택이 불가능.

## W3-5: 에러 처리 — Toast UI, 폴백

### 배운 개념

**Toast 알림**: 화면 모서리에 잠깐 나타났다 사라지는 알림 메시지

```javascript
async function onAnalyzeClick() {
    try {
        const result = await callSymptomApi(text);
        showResult(result);
    } catch (error) {
        // API 실패 시
        showToast('분석에 실패했습니다. 직접 예약으로 이동합니다.');
        setTimeout(() => {
            window.location.href = '/reservation/direct-reservation';
        }, 3000);  // 3초 후 자동 이동
    }
}
```

**폴백(Fallback)**: 주 기능이 실패했을 때 대안을 제공
→ AI 추천 실패해도 사용자가 직접 예약 페이지로 이동할 수 있게

## W3-6: AI 면책 문구 — 법적/윤리적 고려

### 배운 개념

**AI 추천은 의학적 진단이 아님을 명시해야 함**

```html
<!-- 추천 결과 아래에 면책 문구 표시 -->
<div class="disclaimer">
    <i data-feather="info"></i>
    AI 추천은 참고용이며, 의학적 진단이 아닙니다. 정확한 진단은 의사에게 문의하세요.
</div>
```

> **왜 중요한가?**
> 소프트웨어가 의료 행위를 대신할 수 없음. 사용자에게 오해를 주지 않도록 명시 필요.
> 법적 책임 소재를 분명히 하기 위한 요구사항.

## W3-7: 코드 리뷰 반영 — 보안 8건 + 예약 불가 슬롯

### 배운 개념

#### 1. IDOR 공격 방어 (H-01)

**IDOR (Insecure Direct Object Reference)**: 다른 사람의 예약 번호를 알면 그 예약을 취소/변경할 수 있는 취약점

```java
// 취약한 코드: 예약 번호만 확인 (다른 사람 예약도 취소 가능!)
public void cancelReservation(String reservationNumber) {
    Reservation r = findByNumber(reservationNumber);
    r.cancel();
}

// 수정된 코드: 전화번호로 소유자 확인
public void cancelReservation(String reservationNumber, String phone) {
    Reservation r = findByNumber(reservationNumber);
    if (!r.getPatient().getPhone().equals(normalize(phone))) {
        throw new CustomException("예약 소유자가 아닙니다.");
    }
    r.cancel();
}
```

#### 2. TOCTOU 방어 — 비관적 락 (H-03)

**TOCTOU (Time-of-Check-Time-of-Use)**: 중복 체크 후 저장 사이에 다른 사용자가 끼어드는 문제

```
A: "09:00 예약 가능 확인" → (이 순간 B도 동시에 확인) → A: "저장"
                                                           → B: "저장" (중복!)
```

```java
// @Lock으로 비관적 락 적용: 먼저 가져간 쪽이 끝날 때까지 다른 쪽 대기
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT r FROM Reservation r WHERE r.id = :id")
Optional<Reservation> findByIdForUpdate(@Param("id") Long id);
```

#### 3. 예약 불가 슬롯 비활성화

```
새 API 추가: GET /api/reservation/booked-slots?doctorId=1&date=2026-03-17
→ 응답: ["09:00", "10:30"]  (이미 예약된 시간 목록)

프론트엔드에서 이 목록을 받아 해당 시간 옵션을 disabled 처리
```

```java
// Repository에 쿼리 추가
@Query("SELECT r.timeSlot FROM Reservation r " +
       "WHERE r.doctor.id = :doctorId AND r.reservationDate = :date " +
       "AND r.status <> :excluded")
List<String> findBookedTimeSlots(...);
```

#### 4. @Pattern 검증 (M-03)

```java
@Pattern(regexp = "^$|^01[0-9]-\\d{3,4}-\\d{4}$",
         message = "올바른 전화번호 형식이 아닙니다.")
private String phone;
```

> **`^$|`를 앞에 붙이는 이유**: `@NotBlank`가 빈 값을 이미 잡아주므로, `@Pattern`은 빈 값을 허용(`^$`)하고 형식만 검사. 이렇게 해야 에러 메시지가 하나만 나옴.

## W3-9: 직원 기능 개선 — 진료중 상태 / 실시간 갱신 / 스케쥴 제한

### 배운 개념

#### 1. enum switch 표현식 완전성

```java
// Java 14+ switch 표현식: 모든 case를 반드시 처리해야 컴파일됨
String statusText = switch (status) {
    case RESERVED    -> "예약완료";
    case RECEIVED    -> "접수됨";
    case IN_TREATMENT -> "진료중";   // 빠뜨리면 컴파일 에러!
    case COMPLETED   -> "진료완료";
    case CANCELLED   -> "취소됨";
};
```

#### 2. AJAX 응답에 여러 데이터 담기

```java
// 물품 출고 후 재고 수량 + 오늘 출고 내역 목록을 한 번에 반환
@PostMapping("/use")
@ResponseBody
public ResponseEntity<?> useItem(@RequestParam Long id, ...) {
    int newQuantity = itemManagerService.useItem(id, amount);
    List<ItemUsageLogDto> logs = itemManagerService.getTodayUsageLogs();
    return ResponseEntity.ok(Map.of("quantity", newQuantity, "logs", logs));
}
```

```javascript
// 프론트엔드: 응답받아서 재고와 목록을 동시에 갱신
fetch('/staff/item/use', { method: 'POST', body: formData })
    .then(res => res.json())
    .then(data => {
        document.getElementById('quantity').textContent = data.quantity;
        refreshTodayLog(data.logs);  // 출고 내역 목록 새로 그리기
    });
```

#### 3. HTML5 form 속성으로 중첩 폼 해결

```html
<!-- 문제: <form> 안에 <form>을 넣을 수 없음 (브라우저가 무시) -->

<!-- 해결: form 속성으로 폼 밖의 버튼과 폼을 연결 -->
<form id="complete-form" action="/doctor/treatment/complete" method="post">
    <!-- 진료 완료 폼 내용 -->
</form>

<!-- 이 버튼은 complete-form 밖에 있어도 제출 가능 -->
<button type="submit" form="complete-form">진료 완료</button>
```

#### 4. Flatpickr를 의사 스케쥴에 연동 (방문접수/전화예약)

```javascript
// 의사를 선택하면 그 의사의 근무 요일만 달력에서 선택 가능
docSelect.addEventListener('change', function() {
    const opt = this.options[this.selectedIndex];
    const days = opt.dataset.availableDays;  // "MON,WED,FRI"

    const allowed = days.split(',').map(d => DAY_MAP[d]);  // [1, 3, 5]

    datePicker.set('disable', [
        date => !allowed.includes(date.getDay())  // 해당 요일 아닌 날 비활성화
    ]);
});
```

## 전체 3주 학습 흐름

```
1주차: URL → 화면 연결 (Spring MVC 기초)
  ↓
2주차: 화면 → 데이터 저장 (Form, JPA, Service, 테스트)
  ↓
3주차: AI 추천 + 보안 + 실시간 기능 (고급 패턴, 보안 원칙)
```

**핵심 아키텍처:**

```
브라우저 (Mustache + JS)
    ↕ HTTP
Controller (@Controller / @RestController)
    ↕ 메서드 호출
Service (@Service, @Transactional)
    ↕ JPA
Repository (JpaRepository)
    ↕ SQL
Database (H2 / MySQL)
```
