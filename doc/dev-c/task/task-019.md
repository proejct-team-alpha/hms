## 메모
- [x] 직원 등록/수정은 기존 `@Valid + BindingResult` 골격이 이미 있어서 DTO 필수값 보강과 same-view 재렌더링 패턴 정리에 집중했다.
- [x] `staff-form.mustache`는 등록/수정 화면을 함께 쓰고 있으므로 `nameError`, `departmentIdError`, `passwordError` 같은 필드 에러를 공통 패턴으로 보여주도록 맞췄다.
- [x] 직원 등록/수정 검증 실패 테스트는 same-view 렌더링, 에러 attribute, 기존 입력값 유지까지 함께 확인하도록 보강했다.
- [x] 진료과 등록은 기존 `@RequestParam` 기반이라 `@Valid`를 자연스럽게 적용하기 위해 등록 전용 입력 객체를 추가했다.
- [x] 진료과 목록 인라인 등록 검증 실패 시에는 목록 화면 same-view 재렌더링을 사용하고, `createName`, `createActive`, `openCreateModal`, `nameError`를 함께 넘겨서 모달이 다시 열리도록 정리했다.
- [x] 진료과 등록 성공 경로는 기존 `RedirectView`를 유지해 공통 레이아웃 모델 값이 쿼리스트링에 섞이지 않도록 했다.
- [x] 진료과 수정도 등록과 같은 패턴으로 입력 DTO를 분리하고, 상세 화면 same-view 재렌더링 시 `editName`과 `nameError`를 사용해 입력값과 필드 에러를 유지하도록 맞췄다.
- [x] 진료과 수정 성공은 PRG 리다이렉트, DTO 검증 실패는 same-view 재렌더링, 서비스 검증 실패(예: 중복 이름)는 flash 메시지 리다이렉트로 역할을 분리했다.
- [ ] 다음 단계에서는 직원/진료과 폼의 에러 메시지 위치와 문구를 더 통일하고, 범위 테스트와 전체 테스트 상태를 함께 정리하면 된다.
- [ ] 이번 단계에서는 `@NotBlank`, `@NotNull` 중심의 기본 검증만 다루고, 길이 제한이나 패턴 검증은 후속 단계로 미룬다.
