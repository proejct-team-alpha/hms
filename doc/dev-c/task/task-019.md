## 메모
- [x] 직원 등록 DTO에는 이미 로그인 아이디/비밀번호/이름/사번/역할 검증이 일부 있었고, 이번 단계에서 departmentId에 @NotNull을 추가해 등록 필수값 기준을 맞췄다.
- [x] 직원 등록 검증 실패 시 컨트롤러가 BindingResult의 필드 에러를 usernameError, departmentIdError 같은 request attribute로 풀어 Mustache에서 바로 사용할 수 있게 정리했다.
- [x] staff-form.mustache에는 등록 폼 기준 로그인 아이디, 비밀번호, 이름, 사번, 역할, 부서 필드 아래 에러 메시지 영역을 추가했고 기존 입력값은 model 재주입 구조를 그대로 활용했다.
- [x] 컨트롤러 테스트는 부서 미선택 시 same-view 재렌더링, departmentIdError 노출, 입력값 유지(로그인 아이디/사번)까지 확인하도록 보강했다.
- [x] 직원 등록/수정은 이미 CreateAdminStaffRequest, UpdateAdminStaffRequest와 @Valid + BindingResult 골격이 있으며, 등록은 로그인 아이디/비밀번호 필수 검증이 일부 들어가 있지만 부서 필수 검증은 아직 없다.
- [x] 직원 수정은 이름 필수만 DTO에 반영돼 있고, 부서 필수 검증과 필드별 에러 메시지 노출은 아직 미적용 상태다.
- [x] 진료과 등록/수정은 아직 전부 @RequestParam 기반이라 @Valid를 적용하려면 등록용/수정용 입력 객체를 새로 두는 쪽이 자연스럽다.
- [x] 진료과 등록 실패는 목록 화면(admin/department-list), 진료과 수정 실패는 상세 화면(admin/department-detail) same-view 재렌더링으로 가야 입력값 유지가 가능하다.
- [x] 직원 폼은 adminStaffService.getCreateForm(request), getEditForm(request)가 있어 입력값 재주입 기반이 이미 있고, 진료과 쪽은 폼 값을 다시 담아줄 별도 모델 필드나 입력 DTO 응답 구조가 추가로 필요하다.
- [ ] `workflow-019`는 SSR 입력 DTO의 기본 유효성 검증과 에러 UX 정리 명세이고, `task-019`는 실제 구현 순서로 분해한 실행 문서로 사용한다.
- [ ] 이번 단계에서는 `@NotBlank`, `@NotNull` 중심의 기본 검증만 다루고 길이 제한/패턴 검증은 다음 단계로 미룬다.
- [ ] 검증 실패 시에는 가능하면 redirect보다 same-view 재렌더링을 우선 적용한다.
- [ ] 직원 등록과 직원 수정은 비밀번호 필수 여부가 다르므로 동일 DTO 강제 재사용 여부를 주의해서 정리한다.
- [ ] 진료과 등록은 목록 인라인 폼, 진료과 수정은 상세/수정 통합 페이지라는 화면 차이를 반영해 검증 UX를 분리한다.
