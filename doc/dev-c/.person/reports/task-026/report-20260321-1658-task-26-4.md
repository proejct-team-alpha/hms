## 1. 전체 작업 흐름 (Workflow)

1. 먼저 `Task 26-4`의 목표를 다시 확인해서, 이번 단계가 “새 `/admin/rule/new` 흐름을 유지한 채 예전 `/admin/rule/form` 경로를 어떻게 정리할지”를 다루는 정리 단계라는 점을 분명히 했다.
2. 그다음 `AdminRuleController`, `rule-list.mustache`, `AdminRuleControllerTest`를 다시 읽으면서 현재 남아 있는 `/admin/rule/form` 사용 지점을 찾았다.
3. `GET /admin/rule/form`은 사용자 인터뷰와 워크플로우 결정대로 `/admin/rule/new`로 직접 리다이렉트하도록 바꾸고, `POST /admin/rule/form`은 기존 북마크나 오래된 폼 제출이 깨지지 않도록 새 저장 로직을 그대로 타는 legacy 호환 엔드포인트로 유지했다.
4. 목록 화면의 등록 버튼 링크도 `/admin/rule/new`로 바꿔서, 이제 UI에서 새 규칙 등록으로 들어가는 기본 동선이 완전히 새 경로를 사용하도록 맞췄다.
5. 마지막으로 컨트롤러 테스트를 보강해서 legacy GET 리다이렉트와 목록 버튼 링크 변경을 검증하고, rule 대상 테스트를 다시 실행해 회귀가 없는지 확인했다.

## 2. 핵심 코드 (Core Logic)

```java
@GetMapping("/form")
public RedirectView legacyForm() {
    // 옛 등록 화면 경로는 새 표준 경로로 바로 보낸다.
    return redirectTo("/admin/rule/new");
}

@PostMapping("/new")

    return handleCreate(request, bindingResult, redirectAttributes, model, req);
```

이번 단계의 핵심은 “예전 경로를 무조건 없애는 것”이 아니라, 사용자 동선은 `/new`로 통일하면서도 legacy POST는 안전하게 호환시키는 균형을 잡은 점이다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 이번 작업은 예전 건물 입구 표지판을 새 입구로 바꾸는 것과 비슷하다.
- 사람은 이제 새 입구(`/admin/rule/new`)로만 들어가게 안내하지만, 혹시 예전 안내문을 보고 찾아온 사람(`/admin/rule/form`)이 있으면 “입구는 저쪽입니다” 하고 새 입구로 보내 준다.
- 다만 우편물처럼 이미 손에 들고 온 제출물(POST 요청)은 현관에서 버리면 안 되니까, 같은 접수 창구로 안전하게 넘겨서 처리하도록 둔 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **GET redirect와 POST legacy alias를 다르게 처리한 이유**
  - GET 요청은 단순히 화면 진입 경로이기 때문에, `RedirectView`로 새 표준 경로로 보내는 것이 가장 깔끔하다.
  - 하지만 POST 요청은 이미 사용자가 입력한 폼 데이터를 들고 오기 때문에, 단순 redirect를 해 버리면 입력값이 사라지거나 브라우저/서버 동작이 어색해질 수 있다.
  - 그래서 이번에는 GET은 redirect, POST는 같은 저장 로직을 재사용하는 호환 alias로 두는 쪽이 더 안전한 선택이었다.

- **동선 통일과 회귀 방지**
  - 컨트롤러만 바꿔서는 부족하고, 실제 UI에서 누르는 링크도 새 경로를 써야 동선이 통일된다.
  - 그래서 `rule-list.mustache`의 등록 버튼을 `/admin/rule/new`로 바꾸고, 테스트에서 HTML에 이 링크가 실제로 들어가는지까지 확인했다.
  - 이렇게 해야 이후 `Task 26-5`나 후속 문서 정리 때도 “사용자 진입 경로는 `/new`”라는 기준이 흔들리지 않는다.
