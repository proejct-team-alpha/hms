2. 그다음 `AdminRuleController`, `AdminRuleControllerTest`, 관리자 템플릿 문서, 관리자 헤더/사이드바 스크립트를 다시 훑으면서 아직 남아 있는 `rule-form` 템플릿명 참조가 어디에 남아 있는지 검색으로 찾았다.

```java
    return redirectTo("/admin/rule/new");

    return handleCreate(request, bindingResult, redirectAttributes, model, req);
}

private String renderForm(HttpServletRequest req, Model model, CreateAdminRuleRequest request) {
    // 새 등록 화면 이름으로 렌더링을 최종 통일한다.
    return "admin/rule-new";

```

```java
@Test
@DisplayName("legacy form post validation failure rerenders new form and does not call service")
void createRule_withLegacyPathValidationFailure_rerendersNewForm() throws Exception {
    // 예전 POST 경로도 실패 시 새 등록 화면으로 다시 보여줘야 한다.
    mockMvc.perform(post("/admin/rule/form")
                    .param("title", " ")
                    .param("content", "")
                    .with(user("admin01").roles("ADMIN"))
                    .with(csrf()))

            .andExpect(view().name("admin/rule-new"));

```

이번 단계의 핵심은 URL을 새것으로 바꾸는 것보다, 내부 SSR 뷰 이름과 테스트 기대값을 모두 같은 기준으로 맞춰서 이후 작업이 더 이상 `rule-form`과 `rule-new` 사이에서 흔들리지 않게 만든 점이다.

## 3. 초등학생도 이해하는 비유 (Easy Analogy)

- 이번 작업은 학교 교실 이름표를 모두 새 이름으로 바꾸는 작업과 비슷하다.
- 전에는 같은 교실을 어떤 안내판은 `1반`, 어떤 안내판은 `새 1반`이라고 부르고 있어서 헷갈릴 수 있었다.
- 이제는 실제 교실 이름을 전부 `새 1반(rule-new)`으로 통일했고, 예전 이름표를 보고 찾아오는 친구가 있어도 교실 자체는 같은 곳으로 안내되게 만든 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 뷰 이름 통일**
  - Spring MVC + Mustache 환경에서는 같은 화면이라도 컨트롤러 반환 문자열, 템플릿 파일명, 테스트 기대값이 서로 다르면 유지보수 중에 쉽게 어긋난다.
  - 이번 작업에서는 렌더링 기준을 `admin/rule-new` 하나로 고정해, 새 경로 `/admin/rule/new`와 내부 뷰 이름이 같은 방향을 보도록 맞췄다.
  - 이 덕분에 이후 Task 26-6 이후 단계에서 폼 UX를 더 손봐도 화면 이름 때문에 생기는 혼선을 줄일 수 있다.

- **legacy POST를 redirect가 아니라 alias로 유지한 이유**
  - GET 경로는 단순 진입이므로 새 경로로 redirect해도 문제가 없다.
  - 하지만 POST는 사용자가 이미 적어 둔 입력값과 검증 결과를 포함하고 있어서, 무조건 redirect하면 실패 시 입력 유실이나 흐름 왜곡이 생길 수 있다.
  - 그래서 `POST /admin/rule/form`은 삭제하지 않고 새 저장 로직을 재사용하는 alias로 남겨 두고, 실패 시에도 `admin/rule-new` same-view를 직접 렌더링하게 유지했다.

- **참조 제거 범위 제어**
  - 이번에는 운영 코드와 현재 기준 문서에서만 구 템플릿명 참조를 제거했다.
  - 과거 리포트, worklog, 코드 리뷰 문서는 당시 상태를 보존해야 하므로 수정하지 않았다.
  - 이렇게 범위를 나눠야 현재 기준은 깔끔하게 유지하면서도, 과거 의사결정 흔적은 잃지 않을 수 있다.
