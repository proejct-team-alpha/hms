## 2. 핵심 코드 (Core Logic)

```java
public record AdminRuleCreateRequest(
        // 제목은 비어 있으면 안 되고, 엔티티 제한인 200자를 넘기면 안 된다.
        @NotBlank(message = "제목은 필수입니다.")
        @Size(max = 200, message = "제목은 200자 이하로 입력해 주세요.")
        String title,

        // 내용은 문서 기준 최대 3000자로 제한한다.
        @NotBlank(message = "내용은 필수입니다.")
        @Size(max = 3000, message = "내용은 3000자 이하로 입력해 주세요.")
        String content,

        // 카테고리는 문자열이 아니라 enum으로 직접 받아 바인딩한다.
        @NotNull(message = "카테고리를 선택해 주세요.")
        HospitalRuleCategory category,

        // 체크박스가 해제되면 값이 아예 안 올 수 있어서 Boolean으로 받는다.
        Boolean active
) {
    public static AdminRuleCreateRequest defaultForm() {
        return new AdminRuleCreateRequest(null, null, null, Boolean.TRUE);
    }

    public boolean isActiveChecked() {
        return Boolean.TRUE.equals(active);
    }
}
```

핵심은 "입력값을 컨트롤러에서 흩어 받아서 나중에 맞추는 방식"에서 "등록 요청 자체를 하나의 검증 가능한 모델로 먼저 고정하는 방식"으로 바꾼 점이다. 이렇게 해 두면 다음 단계에서 `@Valid`, `BindingResult`, 실패 재렌더링, 체크박스 기본값 처리가 자연스럽게 이어진다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **Record 기반 요청 DTO**
  - 이 프로젝트 규칙은 입력 DTO에 record 사용을 우선한다.
  - `AdminRuleCreateRequest`를 record로 만들면 불변 구조가 기본이 되고, 컨트롤러가 받는 입력 모델의 의도가 더 선명해진다.
  - 다음 단계에서 `@ModelAttribute`나 `@Valid`와 함께 사용할 때도 "등록 요청"이라는 책임이 분명하게 유지된다.

- **Bean Validation 선반영**
  - 이번 단계에서 가장 중요한 것은 검증 규칙을 컨트롤러 로직이 아니라 DTO 선언부에 올려놓는 것이다.
  - `@NotBlank`, `@Size`, `@NotNull`을 DTO에 두면 제목/내용/카테고리 검증 기준이 한 파일에 모이고, 이후 화면 오류 메시지와 테스트도 이 기준을 그대로 따라갈 수 있다.
  - 특히 `category`를 `HospitalRuleCategory` enum으로 직접 받으면 문자열 비교나 수동 변환을 줄일 수 있고, 유효하지 않은 값은 바인딩 단계에서 더 일찍 걸러진다.

- **체크박스와 Boolean 처리**
  - HTML 체크박스는 체크 해제 상태일 때 값이 아예 전송되지 않는 경우가 많다.
  - 그래서 `boolean` 대신 `Boolean`으로 받아 `null`을 허용하면 "미전송"과 "체크됨"을 구분할 수 있다.
  - 여기에 `defaultForm()`으로 초기 진입 시 기본 활성값을 `true`로 주고, `isActiveChecked()`로 저장 직전 안전하게 `boolean`으로 변환하면 UI 기본값과 서버 해석을 함께 맞출 수 있다.
