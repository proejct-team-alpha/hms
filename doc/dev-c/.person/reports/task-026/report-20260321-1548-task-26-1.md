## 2. 핵심 코드/구조 (Core Logic)

```java
@GetMapping("/form")
public String form(Model model) {
    model.addAttribute("pageTitle", RULE_FORM_TITLE);
    return "admin/rule-form";
}

@PostMapping("/form")
public String create(
        @RequestParam("title") String title,
        @RequestParam("content") String content,
        @RequestParam("category") String category) {
    adminRuleService.createRule(title, content, category);
    return "redirect:/admin/rule/list";
}
```

```java
@Transactional
public void createRule(String title, String content, String category) {
    hospitalRuleRepository.save(HospitalRule.create(title, content, HospitalRuleCategory.valueOf(category)));
}

public static HospitalRule create(String title, String content, HospitalRuleCategory category) {
    HospitalRule r = new HospitalRule();
    r.title = title;
    r.content = content;
    r.category = category;
    return r;
}
```

핵심은 현재 등록 흐름이 아직 "옛 경로 + 단순 저장" 상태라는 점이다.  
컨트롤러는 `/new`가 아니라 `/form`을 쓰고, 서비스는 `active` 없이 저장하며, 엔티티의 `create(...)`도 `active`를 명시적으로 받지 않는다. 그래서 다음 단계에서는 단순 URL 치환만이 아니라 DTO, 검증, 체크박스 입력, 실패 재렌더링까지 한 번에 구조를 바꿔야 한다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 이번 작업은 같은 가게를 가리키는 표지판이 세 군데에서 서로 다른 방향을 가리키는지 먼저 확인하는 단계와 비슷했다.
- 실제 매장 입구 표지판은 `/admin/rule/form`을 가리키고 있고, 안내 책자(PRD)는 `/admin/rule/new`라고 적혀 있고, 운영 매뉴얼(API)은 화면은 `/new`, 저장은 `/create`라고 적혀 있는 상태다.
- 바로 공사를 시작하면 나중에 사람들이 다른 문으로 들어오게 될 수 있으니, 먼저 어느 문을 정문으로 쓸지 확정하고 메모해 둔 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **URL 계약 차이**
  - 현재 구현: `GET/POST /admin/rule/form`
  - PRD: `S31 -> /admin/rule/new`
  - API: 화면 `GET /admin/rule/new`, 처리 `POST /admin/rule/create`
  - 이번 구현 기준은 사용자 결정과 PRD를 우선해 `/admin/rule/new`로 통일하는 쪽으로 고정했다.

- **엔티티 제한 vs 문서 제한**
  - `HospitalRule.title`은 엔티티에서 `length = 200`이므로 로컬 코드 기준이 분명하다.
  - `content`는 엔티티상 `TEXT`라 길이 제한이 없지만, PRD와 API에는 3000자 제한이 적혀 있다.
  - 따라서 다음 단계 DTO 검증에서는 `title 200자`, `content 3000자`를 가져와야 한다.

- **SSR 등록 폼 구조**
  - 현재 컨트롤러는 `@RequestParam`만 받고 있어 검증 실패 시 `BindingResult` 기반 재렌더링 구조가 없다.
  - 성공 후에도 플래시 메시지 없이 단순 리다이렉트만 수행한다.
  - 즉, `POST -> redirect -> GET`라는 PRG 뼈대는 있지만, 사용자 경험 관점의 성공/실패 처리는 아직 비어 있다.

- **템플릿 재사용 포인트**
  - 등록 화면 템플릿은 이미 `rule-form.mustache`로 존재하므로 완전 신규 제작보다는 rename 또는 내용 이동이 유리하다.
  - 다만 현재 폼에는 `active` 체크박스, 오류 메시지, 입력값 유지 바인딩이 없어 실질적으로는 "재사용 + 구조 보강"이 필요하다.
