## 2. 핵심 코드 (Core Logic)

```java
@PostMapping("/new")
public Object createNew(
        @Valid @ModelAttribute("model") CreateAdminRuleRequest request,
        BindingResult bindingResult,
        RedirectAttributes redirectAttributes,
        Model model,
        HttpServletRequest req) {
    return create(request, bindingResult, redirectAttributes, model, req);
}

@PostMapping("/form")
public Object create(
        @Valid @ModelAttribute("model") CreateAdminRuleRequest request,
        BindingResult bindingResult,
        RedirectAttributes redirectAttributes,
        Model model,
        HttpServletRequest req) {
    if (bindingResult.hasErrors()) {
        // SSR 필드 오류를 request attribute로 심어 같은 폼을 다시 렌더링한다.
        SsrValidationViewSupport.applyErrors(req, bindingResult);
        return renderForm(req, model, request);
    }

    String successMessage = adminRuleService.createRule(request);
    redirectAttributes.addFlashAttribute("successMessage", successMessage);
    return redirectTo("/admin/rule/list");
}
```

```java
@Transactional
public String createRule(CreateAdminRuleRequest request) {
    HospitalRule rule = HospitalRule.create(
            request.normalizedTitle(),
            request.normalizedContent(),
            request.normalizedCategory()
    );

    // 엔티티 기본값은 active=true 이므로, 체크 해제된 경우에만 뒤집어 비활성으로 맞춘다.
    if (!request.isActiveChecked()) {
        rule.toggleActive();
    }

    hospitalRuleRepository.save(rule);
    return "규칙이 등록되었습니다.";
}
```

핵심은 “등록 화면 경로만 새로 추가한 것”이 아니라, 컨트롤러 검증 흐름, 서비스 저장 계약, 템플릿 바인딩, 테스트를 한 덩어리로 묶어 `/admin/rule/new`를 실제 동작하는 기준 경로로 만든 점이다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 이번 작업은 새 접수 창구를 여는 일과 비슷하다.
- 이전에는 예전 창구(`/admin/rule/form`)에서 종이 몇 장만 받아서 바로 저장하던 상태였다면, 이제는 새 창구(`/admin/rule/new`)에서 정식 신청서(`CreateAdminRuleRequest`)를 받고, 빈칸이 있으면 바로 다시 작성하게 하고, 접수가 끝나면 목록 창구로 안내 메시지와 함께 돌려보내는 흐름을 만든 셈이다.
- 그리고 예전 창구도 당장 폐쇄하지 않고 같은 뒤쪽 접수 절차를 공유하게 해서, 지금 쓰던 동선이 갑자기 끊기지 않게 해 둔 상태다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 검증 실패 재렌더링**
  - 이 프로젝트는 AJAX보다 Mustache SSR과 PRG 흐름을 기본으로 쓴다.
  - 그래서 검증 실패 시 JSON 응답이 아니라 같은 템플릿을 다시 렌더링해야 하고, 이때 `SsrValidationViewSupport.applyErrors(...)`로 `errorMessage`, `titleError`, `contentError`, `categoryError`를 request attribute에 넣어준다.
  - 이렇게 하면 서버 검증 기준과 화면 메시지 표시 기준이 분리되지 않고 자연스럽게 연결된다.

- **RedirectView와 exposeModelAttributes 제어**
  - 공통 레이아웃 인터셉터가 여러 request/model 값을 넣기 때문에, 단순 `"redirect:/..."` 문자열을 반환하면 그 값들이 쿼리스트링에 섞여 나갈 수 있다.
  - 이번 작업에서는 `RedirectView#setExposeModelAttributes(false)`를 사용해 이 문제를 차단했다.
  - 덕분에 성공 리다이렉트 URL이 `/admin/rule/list`로 깔끔하게 유지되고, 플래시 메시지만 필요한 범위로 전달된다.

- **체크박스 기반 active 처리**
  - HTML 체크박스는 해제 시 값이 아예 전송되지 않을 수 있으므로, DTO에서 `Boolean active`로 받고 `isActiveChecked()`로 해석한다.
  - 엔티티의 `HospitalRule.create(...)`는 기본적으로 활성 규칙을 만들기 때문에, 체크 해제 상태일 때만 `toggleActive()`를 호출하는 방식으로 최소 수정으로 요구사항을 반영했다.
