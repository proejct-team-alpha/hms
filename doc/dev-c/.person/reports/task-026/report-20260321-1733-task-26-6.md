# 이번 작업 보고서 - 규칙 저장 규칙 도메인 확정과 카테고리 오류 공통화 (Task 26-6)

## 1. 전체 작업 흐름 (Workflow)

1. 먼저 `AdminRuleService`, `HospitalRule`, `AdminRuleController`, rule 테스트 코드를 다시 확인해서, 현재 규칙 등록이 이미 동작은 하지만 `active`를 생성 후 토글로 뒤집고 있다는 점과 invalid category 처리가 화면 친화적으로 고정되지 않았다는 점을 확인했다.
2. 그다음 규칙 저장 책임을 더 자연스럽게 만들기 위해 `HospitalRule.create(...)`가 생성 시점에 `active`를 직접 받도록 바꾸고, 서비스에서는 trim된 값과 체크박스 결과만 도메인 팩토리에 넘기도록 정리했다.
3. 이어서 컨트롤러에는 `BindingResult` 오류 처리를 감싸는 공통 메서드를 추가해서, 기본 필드 오류 세팅 뒤 `category`의 enum 바인딩 실패만 따로 감지해 친절한 메시지로 덮어쓰도록 만들었다.
4. 이 공통 메서드를 `POST /admin/rule/new`와 legacy `POST /admin/rule/form`이 모두 재사용하도록 연결해서, 두 경로가 invalid category 입력에서도 완전히 같은 SSR 재렌더링 규칙을 따르게 맞췄다.
5. 마지막으로 컨트롤러/서비스 테스트를 보강해서 new/legacy POST 모두 invalid category same-view를 검증하고, rule 대상 테스트 묶음을 다시 실행해 회귀가 없는지 확인했다.

## 2. 핵심 동작 코드 (Core Logic)

```java
public static HospitalRule create(String title, String content, HospitalRuleCategory category, boolean active) {
    HospitalRule rule = new HospitalRule();
    rule.title = title;
    rule.content = content;
    rule.category = category;
    rule.active = active; // 생성 시점에 최종 상태를 직접 확정한다.
    return rule;
}
```

```java
private void applyFormErrors(HttpServletRequest req, BindingResult bindingResult) {
    SsrValidationViewSupport.applyErrors(req, bindingResult);

    FieldError categoryError = bindingResult.getFieldError("category");
    if (categoryError != null && isTypeMismatchError(categoryError)) {
        req.setAttribute("categoryError", "올바른 카테고리를 선택해 주세요.");
    }
}
```

이번 단계의 핵심은 서비스가 더 이상 생성 후 상태를 뒤집지 않는다는 점과, 잘못된 카테고리 입력을 컨트롤러 한 곳에서 정규화해서 new/legacy POST가 같은 실패 UX를 갖게 만든 점이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **도메인 팩토리에서 상태를 직접 확정한 이유**
  - 이전 방식은 `HospitalRule.create(...)`로 기본 활성 규칙을 만든 뒤, 비활성 요청이면 서비스에서 `toggle`로 상태를 뒤집는 구조였다.
  - 이 방식도 동작은 하지만 “생성 결과가 요청값 그대로인가”가 도메인 팩토리 시그니처만 봐서는 드러나지 않고, 상태 전환이 후처리에 숨어 있게 된다.
  - 이번에는 팩토리 입력에 `active`를 포함시켜 생성 결과가 요청값과 1:1로 대응되게 만들었고, 서비스는 조립자 역할만 하도록 단순화했다.

- **invalid category 보정을 컨트롤러 공통 메서드로 묶은 이유**
  - `CreateAdminRuleRequest.category`를 enum으로 유지하면 타입 안전성은 좋아지지만, 잘못된 문자열 입력은 일반 `@NotNull` 누락과 다른 종류의 바인딩 오류가 된다.
  - 이 차이를 그냥 두면 new/legacy POST에서 각자 다른 보정을 하거나, 누락과 잘못된 값이 같은 메시지로 섞일 위험이 있다.
  - 그래서 `BindingResult`를 읽는 공통 메서드 하나에서 `typeMismatch` 계열만 감지해 `올바른 카테고리를 선택해 주세요.`로 바꿨고, 두 POST 경로가 같은 실패 규칙을 재사용하게 만들었다.
