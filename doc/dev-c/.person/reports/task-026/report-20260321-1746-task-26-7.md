## 2. 🧩 핵심 코드 (Core Logic)

```java
mockMvc.perform(post("/admin/rule/new")
                .param("title", "응급 지침")
                .param("content", "응급실 대응 안내")
                .param("category", "WRONG") // 잘못된 카테고리 입력
                .param("active", "true")
                .with(user("admin01").roles("ADMIN"))
                .with(csrf()))
        .andExpect(status().isOk())
        .andExpect(view().name("admin/rule-new")) // 같은 화면으로 다시 보여주고
        .andExpect(request().attribute("categoryError", "올바른 카테고리를 선택해 주세요."));
```

이번 단계의 핵심은 위 같은 테스트가 이미 존재한다는 사실을 기준으로 문서를 맞춘 것이다. 즉, 테스트를 새로 “발명”했다기보다, 실제 커버리지가 체크리스트와 1:1로 대응되도록 정리한 작업이었다.

## 4. 📚 기술 딥다이브 (Technical Deep-dive)

- **MockMvc 기반 SSR 테스트**: Spring MVC 화면 흐름을 실제 브라우저 없이 검증하는 방식이다. 이번 작업에서는 `/admin/rule/new`, `/admin/rule/form`의 렌더링, 리다이렉트, flash 메시지, request attribute를 다시 확인하는 기준이 됐다.
- **BindingResult 기반 검증 흐름**: 폼 입력이 잘못됐을 때 서비스로 내려가지 않고 같은 화면을 다시 보여주는 구조다. Task 26-7 동기화에서는 missing category와 invalid category가 서로 다른 메시지로 노출되는 기존 테스트를 체크리스트에 맞춰 문서화했다.
- **도메인 팩토리 + 서비스 역할 분리**: `HospitalRule.create(..., active)`가 상태를 직접 결정하고, 서비스는 요청 DTO를 조립해 넘기는 구조다. Task 26-6 구현과 Task 26-7 검증이 연결되는 핵심 지점이라 문서 메모에 함께 정리했다.
