## 2. 핵심 전달 코드 (Core Logic)

```java
public interface AdminDepartmentRepository extends JpaRepository<Department, Long> {

    // 관리자 진료과 목록은 admin.department 소유 리포지토리에서 이름순으로 조회한다.
    List<Department> findAllByOrderByNameAsc();
}

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class AdminDepartmentService {

    private final AdminDepartmentRepository adminDepartmentRepository;

    public List<AdminDepartmentDto> getDepartmentList() {
        return adminDepartmentRepository.findAllByOrderByNameAsc()
                .stream()
                // 조회 결과를 화면 전용 DTO로 바로 변환해 SSR 템플릿이 그대로 쓰게 한다.
                .map(AdminDepartmentDto::new)
                .collect(Collectors.toList());
    }
}
```

## 4. 기술 딥다이브 (Technical Deep-dive)

- **모듈 소유권 정리**: `AdminDepartmentService`가 `reservation` 모듈의 `DepartmentRepository`를 참조하고 있던 구조는 기능 소유권이 흐려지는 전형적인 신호다. 이번 작업에서는 관리자 진료과 목록의 조회 책임을 `admin.department` 전용 리포지토리로 이동시켜 모듈 경계를 분명히 했다.
- **조회 메서드 명시화**: `findAll()`에 서비스 쪽 정렬 로직을 얹는 대신 `findAllByOrderByNameAsc()`를 리포지토리에 선언했다. 이렇게 하면 “목록 조회는 이름 오름차순”이라는 의도가 메서드 이름에 직접 드러나고, 후속 페이징 작업에서도 기준이 더 선명해진다.
- **서비스 책임 유지**: 서비스는 여전히 `@Transactional(readOnly = true)` 범위에서 동작하고, 조회 결과를 `AdminDepartmentDto`로 변환하는 역할만 담당한다. 즉 이번 리팩토링은 화면 렌더링 계약을 건드리지 않고 내부 의존성만 바로잡은 셈이다.
- **회귀 방지 테스트**: `AdminDepartmentServiceTest`는 실제 DB 테스트가 아니라 Mockito 기반 단위 테스트로 작성했다. 이 테스트는 “admin 리포지토리의 정렬 메서드를 호출한다”와 “결과 DTO의 이름/상태 라벨이 유지된다”를 동시에 잠가두기 때문에, 다음 단계에서 페이징을 붙일 때도 안전망으로 쓸 수 있다.
