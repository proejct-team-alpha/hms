## 2. 핵심 전달 코드 (Core Logic)

```mustache
<div class="text-sm text-slate-500 text-right">
  <div>총 <span class="font-semibold text-slate-700">{{model.totalCount}}</span>개</div>
  {{#model.totalPages}}
    <div>{{model.currentPage}} / {{model.totalPages}}페이지</div>
  {{/model.totalPages}}
  {{^model.totalPages}}
    <div>0 / 0페이지</div>
  {{/model.totalPages}}
</div>

<div class="flex items-center justify-center gap-2">
  {{#model.hasPrevious}}
    <a href="{{model.previousUrl}}">이전</a>
  {{/model.hasPrevious}}
  {{^model.hasPrevious}}
    <span>이전</span>
  {{/model.hasPrevious}}

  {{#model.pageLinks}}
    <a href="{{url}}">{{page}}</a>
  {{/model.pageLinks}}

  {{#model.hasNext}}
    <a href="{{model.nextUrl}}">다음</a>
  {{/model.hasNext}}
  {{^model.hasNext}}
    <span>다음</span>
  {{/model.hasNext}}
</div>
```

위 코드는 이번 작업의 핵심을 보여준다. 백엔드가 이미 준비한 `model` 안의 페이징 정보를 화면이 그대로 읽어서, 현재 페이지 상태와 이동 링크를 사용자에게 보여주는 구조다. 중요한 점은 템플릿이 자체적으로 페이지를 계산하지 않고, 이미 계산된 값을 `그대로 꺼내 쓰는 역할`만 하도록 유지했다는 점이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 템플릿과 ViewModel 계약**: 이번 작업의 핵심은 서버에서 만든 응답 모델(`AdminDepartmentListResponse`)과 Mustache 템플릿 사이의 계약을 안정적으로 유지하는 것이다. 템플릿이 `{{model.totalCount}}`, `{{model.currentPage}}`, `{{model.pageLinks}}`처럼 `model.xxx`만 읽도록 하면, 페이지 계산 책임은 서비스에 남고 템플릿은 표현만 담당하게 된다. 이렇게 역할이 분리되면 다음 단계에서 정렬 기준이나 페이지 계산 로직이 바뀌더라도 화면은 계약만 유지하면 된다.
- **HTML 이스케이프와 테스트 안정성**: Mustache는 링크 안의 `&`를 `&amp;`로 렌더링할 수 있다. 처음에는 테스트에서 URL 전체 문자열을 그대로 비교하려 했는데, 이 방식은 템플릿 엔진의 표현 방식에 따라 쉽게 흔들린다. 그래서 이번에는 `2 / 3페이지` 같은 화면 정보와 `>1</a>`, `>3</a>` 같은 번호 링크가 실제로 보이는지를 확인하는 방향으로 테스트를 바꿨다. 덕분에 테스트가 “링크가 존재하는지”라는 본래 목적에 더 집중하게 됐다.
- **UI 단계별 분리 작업**: 이번 작업은 백엔드 페이징 계산(Task 17-3)과 화면 렌더링(Task 17-4)을 분리해서 진행한 덕분에 훨씬 안전했다. 먼저 백엔드가 값을 정확히 만들고, 그 다음 템플릿이 그 값을 소비하게 나누니 문제 생겼을 때도 어디가 원인인지 빠르게 좁힐 수 있었다.
