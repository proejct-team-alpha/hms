## 2. 핵심 전달 코드 (Core Logic)

```java
@PostMapping("/form")
public RedirectView create(
        @RequestParam String name,
        @RequestParam(defaultValue = "false") boolean active) {
    adminDepartmentService.createDepartment(name, active);

    RedirectView redirectView = new RedirectView("/admin/department/list");
    redirectView.setExposeModelAttributes(false);
    return redirectView;
}
```

이 코드는 이번 작업의 핵심이다. 화면이 보내는 `name`, `active`를 서버가 실제로 그대로 받도록 맞췄고, 등록 후에는 다른 모델 값이 URL에 붙지 않도록 리다이렉트 방식도 함께 정리했다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **HTML 체크박스와 서버 바인딩**: HTML 체크박스는 체크됐을 때만 요청에 값이 포함되고, 체크되지 않으면 아예 파라미터가 오지 않는다. 그래서 서버 쪽에서 `@RequestParam(defaultValue = "false") boolean active`처럼 기본값을 지정해 주는 방식이 가장 단순하고 안정적이다. 이 패턴을 쓰면 템플릿은 그대로 두고도 “체크 안 됨 = false” 계약이 자연스럽게 완성된다.
- **SSR 등록 계약 정리**: 이번 작업은 DTO를 새로 만들기보다 `name`, `active` 두 값만 정확히 전달하는 데 집중했다. 입력 필드가 적고 로직도 단순할 때는 `@RequestParam` 기반이 오히려 읽기 쉽고, 다음 단계까지 부담을 덜 준다. 대신 계약은 테스트로 확실하게 고정했다.
- **RedirectView와 모델 누출 방지**: 테스트에서 발견한 것처럼, 일반 `redirect:` 문자열은 상황에 따라 모델 값이 리다이렉트 URL 쿼리스트링으로 노출될 수 있다. 이번에는 `RedirectView`와 `setExposeModelAttributes(false)`를 사용해 목록 복귀 URL을 항상 일정하게 유지했다. 덕분에 등록 후 이동 동작이 더 예측 가능해졌고, 공통 레이아웃 모델이 URL에 섞이는 문제도 사라졌다.
