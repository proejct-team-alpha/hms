## 2. 핵심 전달 코드 (Core Logic)

```java
@GetMapping("/detail")
public String detail(
        @RequestParam("departmentId") Long departmentId,
        HttpServletRequest req,
        HttpServletResponse response) {
    try {
        // 서비스가 상세 화면용 응답 모델을 조회해 준다.
        req.setAttribute("model", adminDepartmentService.getDepartmentDetail(departmentId));
        req.setAttribute("pageTitle", "진료과 상세");
        return "admin/department-detail";
    } catch (CustomException ex) {
        // SSR에서는 여기서 직접 404 상태와 에러 화면을 내려준다.
        response.setStatus(HttpServletResponse.SC_NOT_FOUND);
        req.setAttribute("pageTitle", "페이지를 찾을 수 없습니다");
        req.setAttribute("errorMessage", ex.getMessage());
        req.setAttribute("path", req.getRequestURI());
        return "error/404";
    }
}
```

이 코드의 핵심은 “상세 화면 정상 렌더링”과 “존재하지 않는 진료과 요청의 404 처리”를 한 메서드 안에서 SSR 흐름에 맞게 분기했다는 점이다. 이전에는 `CustomException`을 던지면 공통 SSR 예외 처리기가 모두 500 화면으로 보내고 있었기 때문에, 상세 not found는 컨트롤러 레벨에서 분기해 주는 것이 가장 안전했다.

## 3. 초등학생도 이해할 수 있는 비유를 들어서 (Easy Analogy)

- 이번 작업은 학교 사물함 카드에 비유할 수 있다.
- 사물함 번호를 누르면 그 학생 카드가 열리고, 카드 안에서 이름을 고치거나 상태를 바꿀 준비를 해둔 것이다.
- 그런데 없는 번호를 눌렀을 때는 “카드를 찾을 수 없어요”라고 404 안내판을 보여줘야 한다.
- 예전 구조는 없는 번호를 눌러도 무조건 “큰 오류가 났어요(500)”라고만 말하고 있었기 때문에, 이번 작업에서 그 상황만 따로 구분해서 더 정확한 안내판을 달아준 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 상세/수정 통합 화면 구성**: 이번 작업은 진료과 상세와 수정을 같은 페이지에서 처리하는 구조를 실제 템플릿으로 풀어낸 단계다. `AdminDepartmentDetailResponse`가 `departmentId`, `name`, 상태 라벨, 배지 클래스, 액션 URL까지 모두 갖고 있기 때문에 템플릿은 `{{model.xxx}}`만으로 화면을 그릴 수 있다. 이런 구조는 컨트롤러를 가볍게 유지하고, 이후 `Task 18-4`, `Task 18-5`에서 POST 액션을 붙일 때도 화면 계약을 흔들지 않게 해준다.
- **SSR의 404 처리 방식**: 공통 [SsrExceptionHandler.java](/c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/common/exception/SsrExceptionHandler.java)는 현재 `CustomException`을 모두 `error/500`으로 보내고 있다. 그래서 상세 조회 not found를 그대로 예외로만 던지면 테스트 기준으로도 404가 아니라 500 흐름이 되어버린다. 이번 작업에서는 공통 핸들러를 건드리지 않고, 상세 컨트롤러에서만 404 상태와 `error/404` 템플릿을 직접 반환해 기능 요구사항을 맞췄다. 공유 예외 처리기를 무리하게 바꾸지 않고, 필요한 화면 흐름만 국소적으로 정리했다는 점이 중요하다.
- **컨트롤러 테스트의 역할**: [AdminDepartmentControllerTest.java](/c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/department/AdminDepartmentControllerTest.java)에서는 단순히 view 이름만 확인한 것이 아니라, 실제로 `내과`, `이름 수정`, `비활성화` 같은 문자열이 HTML에 들어가는지까지 확인했다. 덕분에 템플릿 추가 후에도 화면 계약이 깨지지 않았는지 확인할 수 있었고, 404 흐름도 상태 코드와 뷰 이름, 에러 메시지까지 같이 검증할 수 있었다.
