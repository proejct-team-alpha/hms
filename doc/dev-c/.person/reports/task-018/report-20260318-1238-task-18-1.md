## 4. 기술 딥다이브 (Technical Deep-dive)

- **URL 계약 정리의 중요성**: SSR 화면은 컨트롤러, 템플릿, 테스트가 모두 같은 URL 계약을 바라봐야 안정적이다. 지금은 문서상 목표가 `/admin/department/create`인데 실제 코드는 `/admin/department/form`을 사용 중이어서, 상세/수정 기능을 붙이기 전에 이 계약을 먼저 통일해야 한다.
- **도메인 책임 분리**: 현재 `Department.update(name, active)`는 편해 보이지만, 이름 수정과 상태 전이는 성격이 다르다. 이름 수정은 일반 편집이고, 활성/비활성은 상태 전이다. 이 둘을 분리해두면 컨트롤러와 서비스가 의도를 더 명확하게 표현할 수 있고, 예외 처리도 훨씬 자연스러워진다.
- **목록 DTO와 상세 DTO의 역할 차이**: 목록 DTO는 테이블 렌더링에 필요한 최소 정보만 담으면 충분하지만, 상세/수정 화면은 버튼 노출 조건, 상태 배지, 수정 폼 바인딩 값까지 함께 필요하다. 그래서 기존 `AdminDepartmentDto`를 무리하게 재활용하기보다 상세 화면 전용 응답 모델을 별도로 두는 편이 구조적으로 더 안전하다.
