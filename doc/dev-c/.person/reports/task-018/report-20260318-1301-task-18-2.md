## 2. 핵심 전달 코드 (Core Logic)

```java
public record AdminDepartmentDetailResponse(
        Long departmentId,
        String name,
        boolean active,
        String activeText,
        String activeBadgeClass,
        boolean activatable,
        boolean deactivatable,
        String updateAction,
        String activateAction,
        String deactivateAction
) {
    public static AdminDepartmentDetailResponse from(Department department) {
        boolean active = department.isActive();

        return new AdminDepartmentDetailResponse(
                department.getId(),
                department.getName(),
                active,
                active ? "운영 중" : "비운영",
                active ? "초록 배지 클래스" : "빨강 배지 클래스",
                !active,   // 비운영일 때만 활성화 버튼 노출
                active,    // 운영 중일 때만 비활성화 버튼 노출
                "/admin/department/update",
                "/admin/department/activate",
                "/admin/department/deactivate"
        );
    }
}
```

이번 단계의 핵심은 “상세 화면이 필요한 정보를 여기저기서 따로 계산하지 않게 하고, 한 번 만든 응답 모델 안에 표현용 상태와 액션 조건을 같이 담는다”는 점이다. 이렇게 해두면 다음 단계의 컨트롤러와 템플릿은 `pageTitle + model` 패턴으로 이 모델만 넘겨받아 화면을 구성하면 된다.

## 3. 초등학생도 이해할 수 있는 비유를 들어서 (Easy Analogy)

- 이번 작업은 발표 준비를 하기 전에, 발표자가 들고 올라갈 종이 한 장에 필요한 내용을 미리 다 적어두는 것과 비슷하다.
- 이름, 현재 상태, 어떤 버튼을 보여줄지 같은 정보를 한 종이에 정리해 두면, 발표자는 그 종이를 보기만 하면 된다.
- 반대로 이런 정리 없이 올라가면, 발표 도중에 “지금 이 버튼을 보여줘야 하나?”를 계속 현장에서 다시 계산해야 해서 훨씬 헷갈린다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **Record 기반 화면 응답 모델**: `AdminDepartmentDetailResponse`를 record로 만든 이유는 “이 화면이 정확히 어떤 데이터를 받는지”를 선언적으로 보여주기 위해서다. 필드 목록만 봐도 상세/수정 통합 화면 계약이 거의 드러나기 때문에, 다음 단계 구현자가 구조를 빠르게 이해할 수 있다.
- **도메인 상태와 화면 상태의 분리**: 도메인에는 `active`라는 원시 상태만 있지만, 화면은 `운영 중/비운영` 문구, 색상 배지 클래스, 활성화/비활성화 버튼 노출 조건까지 필요하다. 이번 단계에서는 이 표현 책임을 응답 모델의 `from()` 팩토리에서 흡수해, 템플릿이 숫자나 boolean의 의미를 다시 해석하지 않도록 만들었다.
- **상태 전이 액션 분리**: `workflow-018`은 이름 수정과 상태 변경을 한 폼에서 섞지 말라고 요구한다. 그래서 응답 모델도 `name`과 `updateAction`은 수정 폼용으로, `activatable`/`deactivatable`과 각 action URL은 상태 전이용으로 분리해 담도록 설계했다. 덕분에 다음 단계에서 컨트롤러와 템플릿이 책임을 섞지 않고 구현할 수 있다.
