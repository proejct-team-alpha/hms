## 2. 핵심 전달 코드 (Core Logic)

```java
@PostMapping("/update")
public RedirectView update(
        @RequestParam Long departmentId,
        @RequestParam(defaultValue = "") String name,
        RedirectAttributes redirectAttributes) {
    try {
        // 서비스에서 이름 검증과 실제 변경을 모두 처리한다.
        String successMessage = adminDepartmentService.updateDepartmentName(departmentId, name);
        // 성공하면 상세 페이지에서 바로 결과를 볼 수 있도록 flash 메시지를 남긴다.
        redirectAttributes.addFlashAttribute("successMessage", successMessage);
        return redirectToDetail(departmentId);
    } catch (CustomException ex) {
        // 실패 메시지도 같은 방식으로 전달해 사용자가 이유를 놓치지 않게 한다.
        redirectAttributes.addFlashAttribute("errorMessage", ex.getMessage());
        // 없는 진료과라면 상세로 가도 404가 나므로 목록으로 돌려보낸다.
        if (ex.getHttpStatus() == HttpStatus.NOT_FOUND) {
            return redirectTo("/admin/department/list");
        }
        // 그 외 검증 실패는 다시 상세 페이지로 보내 재입력하게 한다.
        return redirectToDetail(departmentId);
    }
}
```

이 코드는 이번 작업의 핵심 흐름을 가장 잘 보여준다. 컨트롤러는 입력값을 받고, 실제 비즈니스 판단은 서비스에 위임하고, 사용자가 다음 화면에서 무엇을 보게 될지만 책임진다. 그래서 역할이 섞이지 않고, 테스트도 컨트롤러와 서비스로 나누어 안정적으로 작성할 수 있었다.

## 3. 초등학생도 이해할 수 있는 비유를 들어서 (Easy Analogy)

- 이번 작업은 학교 게시판에서 반 이름표를 바꾸는 것과 비슷하다.
- 게시판에는 반마다 이름표가 하나씩 붙어 있는데, 이름표를 바꿀 때는 아무 글자나 적으면 안 된다.
- 이름이 비어 있으면 안 되고, 다른 반이 이미 쓰고 있는 이름도 쓰면 안 된다.
- 그리고 없는 반 번호를 들고 와서 이름을 바꾸겠다고 하면, 그건 수정 화면이 아니라 반 목록으로 다시 안내해주는 게 더 자연스럽다.
- 이번 구현은 바로 이런 규칙을 코드로 옮긴 작업이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **PRG와 flash 메시지 설계**: 이름 수정은 POST 요청이기 때문에 성공 후 바로 화면을 렌더링하지 않고 redirect로 넘기는 PRG 패턴을 유지했다. 이렇게 하면 새로고침 시 폼 재전송을 막을 수 있고, `successMessage`와 `errorMessage`는 flash attribute로 넘겨 한 번만 보여줄 수 있다. SSR 관리 화면에서 가장 안전한 기본 패턴이다.
- **도메인 메서드 분리**: 기존 `Department.update(name, active)`는 이름 수정과 상태 변경이 한 메서드 안에 같이 들어갈 여지가 있었다. 이번 단계에서는 이름 수정만 구현하는 게 목표였기 때문에 `rename(...)`을 따로 두어 의도를 명확히 했다. 이 분리는 다음 단계에서 활성화/비활성화 상태 전이를 붙일 때도 도움이 된다.
- **중복 이름 검증 위치**: 중복 이름 검증은 컨트롤러가 아니라 서비스에서 처리했다. 이렇게 하면 SSR 화면 외에 이후 REST API나 다른 입력 경로가 생겨도 같은 검증 규칙을 재사용할 수 있다. 또한 `existsByNameIgnoreCaseAndIdNot(...)` 같은 리포지토리 메서드를 통해 자기 자신을 제외한 중복만 잡을 수 있어 수정 시나리오와 잘 맞는다.
- **에러 흐름 분기 전략**: 검증 실패와 리소스 없음은 사용자 경험이 다르다. 공백 이름과 중복 이름은 사용자가 같은 상세 화면에서 바로 고치면 되므로 다시 상세 페이지로 보내는 것이 자연스럽다. 반대로 존재하지 않는 진료과는 상세 페이지 자체가 성립하지 않으므로 목록으로 돌려보내는 편이 운영 흐름에 맞았다. 이 차이를 컨트롤러 리다이렉트 정책으로 분리한 것이 이번 작업의 중요한 정리 포인트다.
