## 2. 핵심 전달 코드 (Core Logic)

```java
@PostMapping("/create")
public RedirectView create(
        @RequestParam String name,
        @RequestParam(defaultValue = "false") boolean active) {
    // 목록 화면 인라인 폼이 보내는 등록 요청을 단일 URL로 받는다.
    adminDepartmentService.createDepartment(name, active);
    return redirectTo("/admin/department/list");
}
```

이 코드는 이번 작업의 핵심을 가장 잘 보여준다. 등록 처리 진입점을 `/create` 하나로 고정함으로써, 문서와 코드와 테스트가 모두 같은 계약을 바라보게 됐다. 별도 `/form` 등록 페이지를 두지 않으니 사용자는 목록 화면에서만 등록하고, 서버도 그 흐름만 책임지게 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **URL 계약 일원화**: SSR에서는 컨트롤러, 템플릿, 테스트가 모두 같은 URL 계약을 바라봐야 회귀가 적다. 이번 작업 전에는 문서상 등록 URL과 실제 구현 URL이 달라 유지보수 리스크가 있었다. `/create`로 통일하면서 문서-코드 간 불일치를 해소했다.
- **비노출 경로의 테스트 고정**: “없는 기능”도 계약이다. `GET /admin/department/form`을 단순히 안 쓰는 것에서 끝내지 않고, 실제로 404가 나는지 테스트에 넣어 두면 이후 누군가 별도 등록 페이지를 다시 살리는 변경을 넣었을 때 바로 감지할 수 있다.
- **PRG 일관성 유지**: 등록 처리는 여전히 PRG(Post-Redirect-Get)를 따르며 목록으로 복귀한다. 이번 작업은 비즈니스 로직을 바꾸지 않고, 입력 진입점만 정리했기 때문에 기존 서비스 로직과 리다이렉트 흐름을 안전하게 유지할 수 있었다.
- **정리 대상 파일 제거**: 사용하지 않는 `department-form.mustache`를 남겨두면 “죽은 코드”가 되고, 나중에 화면 계약을 오해하게 만든다. 템플릿 삭제는 단순한 정리처럼 보이지만, 실제로는 SSR 화면 흐름을 명확히 하는 중요한 마무리 작업이다.
