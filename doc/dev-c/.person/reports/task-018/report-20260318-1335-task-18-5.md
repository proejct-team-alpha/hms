## 2. 핵심 전달 코드 (Core Logic)

```java
@Transactional
public String deactivateDepartment(Long departmentId) {
    Department department = findDepartment(departmentId);
    if (!department.isActive()) {
        // 이미 비활성 상태라면 중복 상태 전이 요청을 거부한다.
        throw CustomException.invalidStatusTransition(ALREADY_INACTIVE_MESSAGE);
    }

    // 실제 상태 변경은 엔티티 메서드가 맡아 의도를 분명하게 한다.
    department.deactivate();
    adminDepartmentRepository.save(department);
    return DEPARTMENT_DEACTIVATED_MESSAGE;
}
```

이 코드는 이번 작업의 핵심을 잘 보여준다. 컨트롤러는 요청을 받고 리다이렉트만 정리하고, 실제 상태 전이 규칙은 서비스가 책임진다. 또 상태값을 직접 필드에 대입하지 않고 `Department.deactivate()`를 통해 변경해서, 나중에 상태 전이 규칙이 더 복잡해져도 도메인 안에서 확장할 수 있게 만들었다.

## 3. 초등학생도 이해할 수 있는 비유를 들어서 (Easy Analogy)

- 이번 작업은 교실 불을 켜고 끄는 스위치를 다루는 것과 비슷하다.

- 반대로 불이 꺼져 있을 때는 "켜기"만 의미가 있고, 이미 켜져 있는데 또 "켜기"를 누르면 그것도 중복 행동이 된다.
- 이번 구현은 바로 이런 규칙을 코드로 옮겨서, 스위치를 잘못 눌렀을 때는 "이미 그런 상태예요"라고 알려주고, 정상 동작이면 다시 상세 화면으로 돌아오게 만든 것이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **상태 전이 분리**: 이번 작업의 핵심은 이름 수정과 상태 변경을 완전히 다른 액션으로 나눈 것이다. 이름은 `rename()`, 상태는 `activate()/deactivate()`로 분리하면서, 수정 화면이 상태 select를 직접 다루지 않아도 운영 흐름을 자연스럽게 유지할 수 있게 되었다.
- **PRG와 flash 메시지 일관성**: 이름 수정에서 이미 쓰고 있던 PRG 패턴을 상태 전이에도 그대로 적용했다. 성공 시에는 `successMessage`, 실패 시에는 `errorMessage`를 flash attribute로 넘겨서 새로고침 재전송 문제를 피하면서도 사용자가 결과를 놓치지 않도록 했다.
- **중복 상태 전이 방어**: 단순히 `active` 값을 true/false로 덮어쓰는 대신, 현재 상태를 보고 중복 요청을 막았다. 이 방식은 운영 화면에서 버튼을 여러 번 누르거나 브라우저가 중복 요청을 보냈을 때도 의미 없는 상태 변경을 허용하지 않는다는 점에서 중요하다.
- **SSR 테스트 강화**: 컨트롤러 테스트는 단순히 status code만 보는 수준을 넘어서, 어떤 URL로 리다이렉트되는지와 flash 메시지가 무엇인지까지 함께 검증했다. 서비스 테스트는 실제 active 값이 바뀌는지, 그리고 중복 상태 요청이 예외로 막히는지 확인했다. 덕분에 상태 전이 규칙이 화면과 서비스 양쪽에서 모두 고정되었다.
