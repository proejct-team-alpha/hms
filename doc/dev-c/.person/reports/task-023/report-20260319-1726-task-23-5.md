## 2. 핵심 코드/구조 (Core Logic)

```java
@WebMvcTest(AdminPatientApiController.class)
@Import(AdminControllerTestSecurityConfig.class)
class AdminPatientApiControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockitoBean
    private AdminPatientService adminPatientService;

    @Test
    void updatePatient_success() throws Exception {
        UpdateAdminPatientApiResponse response = new UpdateAdminPatientApiResponse(
                10L,
                "김수정",
                "010-9999-8888",
                "메모 수정",
                "환자 정보가 수정되었습니다."
        );
        given(adminPatientService.updatePatient(10L, "김수정", "010-9999-8888", "메모 수정"))
                .willReturn(response);

        mockMvc.perform(post("/admin/api/patients/10/update")
                        .with(user("admin").roles("ADMIN"))
                        .with(csrf())
                        .contentType("application/json")
                        .content("""
                                {
                                  "name": "김수정",
                                  "phone": "010-9999-8888",
                                  "note": "메모 수정"
                                }
                                """))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.body.patientId").value(10))
                .andExpect(jsonPath("$.body.message").value("환자 정보가 수정되었습니다."));
    }
}
```

이번 단계의 핵심은 서비스 로직을 더 건드리는 것이 아니라, 이미 만들어진 서비스/DTO/API 연결이 실제로 관리자 API 규약에 맞게 작동하는지를 테스트 계층에서 닫는 것이다. 특히 `Resp.ok(...)`의 success body 구조와 `GlobalExceptionHandler`의 validation/business error 응답 형식이 동시에 보장되는지가 중요했다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 `@WebMvcTest`로 API 테스트를 닫았는가**: 이번 범위는 서비스 알고리즘이 아니라 HTTP 계층 계약 검증이 핵심이기 때문이다. 서비스는 이미 별도 테스트([AdminPatientServiceTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/patient/AdminPatientServiceTest.java))로 검증되어 있어서, API 테스트에서는 JSON 요청 바인딩, 보안, 예외 변환, 응답 wrapper에 집중하는 것이 맞다.
- **권한 테스트를 왜 꼭 넣었는가**: URL이 `/admin/api/**`이기 때문에, 실제 운영에서는 SSR보다도 권한 분기 누락이 더 위험하다. 그래서 성공 케이스와 같은 무게로 `ROLE_STAFF` 차단 시나리오를 확인했다.
- **왜 `AdminPatientControllerTest`까지 같이 정리했는가**: 새 API 테스트 자체는 바로 통과했지만, 같은 패키지 범위를 다시 돌리는 순간 기존 컨트롤러 테스트에 남아 있던 인코딩 손상이 드러났다. 이걸 그냥 두면 다음 단계마다 `admin.patient.*` 범위 확인이 흔들리기 때문에, 이번에 같이 정리하는 편이 전체 흐름상 더 안전했다.
- **문구 검증을 줄이고 계약 검증을 남긴 이유**: Mustache 화면 테스트에서 한글 문자열 전체에 강하게 의존하면 인코딩 이슈에 과민해질 수 있다. 그래서 컨트롤러 테스트는 `view/model/핵심 데이터 렌더링` 중심으로 바꾸고, 템플릿 문구 자체는 필요한 최소한만 본다.
