## 2. 핵심 코드/구조 (Core Logic)

```java
public void updateInfo(String name, String phone, String email, String address, String note) {
    this.name = name;
    this.phone = phone;
    this.email = email;
    this.address = address;
    this.note = note;
}
```

```java
public interface AdminPatientRepository extends JpaRepository<Patient, Long> {

    @Query("""
            SELECT p
            FROM Patient p
            WHERE (:nameKeyword = '' OR LOWER(p.name) LIKE LOWER(CONCAT('%', :nameKeyword, '%')))
              AND (:contactKeyword = '' OR REPLACE(p.phone, '-', '') LIKE CONCAT('%', :contactKeyword, '%'))
            ORDER BY p.createdAt DESC, p.id DESC
            """)
    Page<Patient> search(...);
}
```

이번 점검의 핵심은 “이미 있는 구조를 그대로 API에 붙일 수 있는가?”를 확인하는 것이었다. 결론은 일부는 재사용 가능하지만, 그대로 쓰면 안 되는 부분도 분명했다. `Patient.updateInfo(...)`는 부분 수정용 메서드가 아니고, `AdminPatientRepository`도 아직 전화번호 중복 검사 책임을 갖고 있지 않다. 그래서 다음 단계에서는 DTO와 서비스 로직으로 보호막을 먼저 세워야 한다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **부분 수정과 엔티티 메서드의 거리**: 엔티티의 `updateInfo(...)`는 한 번에 많은 필드를 바꾸는 메서드다. 부분 수정 API를 그대로 연결하면 요청에 없는 필드까지 덮어쓸 위험이 있다. 따라서 서비스에서 기존 이메일/주소를 유지한 채 필요한 값만 바꾸는 조합이 필요하다.
- **전화번호 중복 방지의 중요성**: 현재 시스템은 환자를 사실상 전화번호로 찾아 재사용한다. 즉 전화번호 중복 허용은 단순한 표시 문제가 아니라 환자 동일성 판단 규칙을 흔드는 문제다. 이번 API에서 중복 전화번호를 차단해야 하는 이유가 여기에 있다.
- **관리자 API 패턴 재사용**: `admin.staff`, `admin.reservation`을 보면 관리자 API는 `/admin/api/**` 아래에 별도 컨트롤러를 두고, 입력은 `@Valid @RequestBody`, 출력은 `Resp.ok(...)`로 통일한다. `admin.patient`도 이 패턴을 따르면 새 기능이어도 프로젝트 전체 흐름과 잘 맞는다.
- **예외 처리 책임 분리**: 공백 입력은 Bean Validation이 자동으로 막고, 없는 `patientId`나 전화번호 중복은 서비스에서 `CustomException`을 던지는 방식이 가장 자연스럽다. 이렇게 나누면 컨트롤러는 얇게 유지되고, 테스트도 검증 실패와 비즈니스 예외를 구분해서 쓸 수 있다.
