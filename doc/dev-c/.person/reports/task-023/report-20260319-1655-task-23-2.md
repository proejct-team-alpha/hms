## 2. 핵심 코드/구조 (Core Logic)

```java
public record UpdateAdminPatientApiRequest(
        @NotBlank(message = "이름은 필수입니다.")
        String name,

        @NotBlank(message = "연락처는 필수입니다.")
        String phone,

        String note
) {
}
```

```java
public record UpdateAdminPatientApiResponse(
        Long patientId,
        String name,
        String phone,
        String note,
        String message
) {
}
```

핵심은 수정 API의 입력과 출력을 “필요한 만큼만” 정의하는 것이다. 요청은 검증 가능한 최소 필드만 받고, 응답은 수정 성공 후 클라이언트가 다시 보여주기 좋은 정보만 돌려준다. 이렇게 하면 이후 서비스와 컨트롤러 구현이 훨씬 단순해진다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 이번 DTO 설계는 택배 송장과 같다.
- 보내는 쪽에는 꼭 필요한 정보만 적고, 받는 쪽에도 확인에 필요한 정보만 적어야 읽기 쉽고 실수도 줄어든다.
- 이름, 연락처, 메모만 바꾸는 API인데 환자 상세 화면 전체 정보를 실어 나르면, 작은 택배에 이사 짐을 같이 넣는 것처럼 과해진다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **요청 DTO의 역할**: `UpdateAdminPatientApiRequest`는 단순히 필드를 모으는 구조가 아니라, 공백 입력을 미리 막아주는 검증 계약이다. `@Valid @RequestBody`와 붙이면 컨트롤러에서 별도 if문 없이도 기본 입력 오류를 공통 예외 처리로 넘길 수 있다.
- **응답 DTO를 상세 DTO와 분리한 이유**: `AdminPatientDetailResponse`는 SSR 상세 화면 전용 모델이라 이메일, 주소, 예약 이력까지 담는다. 수정 API 응답까지 이 DTO를 쓰면 계약이 무거워지고, 나중에 상세 화면 변경이 API 응답까지 불필요하게 흔들릴 수 있다. 그래서 수정 API는 별도 응답 DTO로 작게 분리하는 쪽이 더 안전하다.
- **`note` 정책을 지금 확정하지 않은 이유**: `note`는 선택 입력이라 `null`과 빈 문자열을 모두 허용할 수 있다. 하지만 DB 저장 시 둘 중 무엇으로 정규화할지는 서비스 레벨의 책임에 더 가깝다. DTO 단계에서 성급히 제한하지 않고, 서비스에서 실제 저장 정책을 한 번 더 정리하기로 미룬 건 의도적인 결정이다.
