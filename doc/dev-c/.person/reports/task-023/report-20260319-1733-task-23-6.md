## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 workflow/task 문서를 같이 갱신했는가**: `workflow`는 요구사항과 수용 기준을 담고, `task`는 실제 작업 분할과 완료 상태를 담는다. 둘 중 하나만 갱신하면 구현 상태와 문서 상태가 어긋난다. 그래서 둘 다 현재 코드 기준으로 맞추는 게 중요했다.
- **왜 범위 테스트와 전체 테스트를 둘 다 다시 확인했는가**: `admin.patient` 범위 테스트는 이번 변경의 직접 영향 영역을 확인하는 데 적합하고, 전체 `./gradlew test`는 브랜치 전체 안정성을 보는 데 필요하다. 두 층을 다 확인해야 "이 기능은 붙었고, 전체 프로젝트도 깨지지 않았다"는 말을 근거 있게 할 수 있다.
- **왜 PR 리뷰 포인트를 문서에 남겼는가**: 이번 API는 URL 설계, 전화번호 중복 방지 규칙, `note` 정규화 정책처럼 리뷰어가 의사결정 포인트를 확인해야 하는 부분이 있다. 이런 내용을 작업 문서에 미리 적어두면 PR 본문을 쓸 때도 일관성이 생기고, 리뷰가 더 빨라진다.
- **문서 마감 단계의 의미**: 코드만 맞고 문서가 뒤처진 상태는 결국 팀 커뮤니케이션 비용을 늘린다. 이번 단계는 개발자가 구현한 것을 팀이 이해 가능한 형태로 정리한 과정이라고 보면 된다.
