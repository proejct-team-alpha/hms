## 2. 핵심 코드/구조 (Core Logic)

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/admin/api/patients")
public class AdminPatientApiController {

    private final AdminPatientService adminPatientService;

    @PostMapping("/{id}/update")
    public ResponseEntity<Resp<UpdateAdminPatientApiResponse>> updatePatient(
            @PathVariable("id") Long patientId,
            @Valid @RequestBody UpdateAdminPatientApiRequest request) {
        UpdateAdminPatientApiResponse response = adminPatientService.updatePatient(
                patientId,
                request.name(),
                request.phone(),
                request.note()
        );

        return Resp.ok(response);
    }
}
```

이번 단계의 핵심은 컨트롤러가 비즈니스 규칙을 새로 만들지 않고, 서비스가 이미 정해둔 규칙을 HTTP/JSON 형태로 전달하는 역할만 하게 만든 것이다. 그래서 컨트롤러는 얇고, 예외 흐름도 기존 공통 처리와 자연스럽게 이어진다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 API 컨트롤러를 따로 두는가**: 현재 프로젝트는 SSR 컨트롤러와 API 컨트롤러를 분리하는 패턴을 쓰고 있다. 환자 목록/상세 SSR은 `AdminPatientController`가 맡고, 수정 API는 `AdminPatientApiController`가 맡도록 나누는 편이 URL 의미와 책임이 모두 더 깔끔하다.
- **컨트롤러를 얇게 유지한 이유**: 입력 검증은 `@Valid`, 비즈니스 규칙은 서비스, 예외 응답 포맷은 `GlobalExceptionHandler`가 담당하면 각 레이어의 책임이 분명해진다. 컨트롤러에서 중복 전화번호나 not found 판단까지 같이 하기 시작하면 이후 테스트도 훨씬 복잡해진다.
- **왜 이번 단계는 compile만 돌렸는가**: 서비스 로직은 이미 테스트로 닫혀 있었고, 이번 단계는 HTTP 엔드포인트 연결이 핵심이었다. 그래서 우선 컴파일이 깨지지 않는지 빠르게 확인하고, 다음 Task에서 API 테스트를 붙여 입력/응답/예외 시나리오를 한 번에 검증하는 쪽이 더 효율적이었다.
