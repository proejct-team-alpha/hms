4. 프론트 동작은 별도 JS 파일로 크게 벌리지 않고, 상세 템플릿 안에 작은 스크립트로 붙였다. `const`와 `async/await` 중심으로 `fetch`를 사용해 기존 환자 수정 API를 호출하고, 성공하면 화면의 이름/연락처/메모를 즉시 갱신한 뒤 수정 모드를 닫도록 했다. 실패 시에는 공통 API 에러 포맷을 기준으로 `VALIDATION_ERROR`, `DUPLICATE_PATIENT_PHONE`, `RESOURCE_NOT_FOUND`를 나눠 처리해서, 필드 아래 에러 또는 카드 상단 메시지로 사용자에게 바로 보이게 했다.
5. 마지막으로 [AdminPatientControllerTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/patient/AdminPatientControllerTest.java)에 상세 화면에서 수정 버튼, 수정 폼, API URL이 실제로 렌더링되는지 검증을 추가했다. 그 뒤 `./gradlew test --tests 'com.smartclinic.hms.admin.patient.*'`와 전체 `./gradlew test`를 순차로 다시 돌려서, 이번 연동이 `admin.patient` 범위뿐 아니라 전체 브랜치 안정성에도 영향을 주지 않는지 확인했다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 목록이 아니라 상세 화면에 붙였는가**: 환자 목록은 검색 인덱스 역할이 핵심이고, 수정 UI까지 넣으면 화면이 금방 무거워진다. 반면 상세 화면은 이미 환자 한 명에 집중하는 구조라 수정까지 흡수해도 책임이 자연스럽다.
- **왜 인라인 편집을 택했는가**: 수정 필드가 3개뿐이라 모달을 띄우기보다 같은 카드 안에서 읽기 모드와 수정 모드를 바꾸는 편이 훨씬 단순했다. 사용자도 현재 값과 수정값을 쉽게 비교할 수 있고, 예약 이력 맥락도 잃지 않는다.
- **왜 `noteValue`를 분리했는가**: 상세 화면에서는 빈 메모를 `-`로 보여주는 것이 읽기 UX에는 좋지만, 편집할 때 `-`가 textarea 안에 들어가면 잘못된 저장값이 될 수 있다. 표시값과 편집 원본값을 분리해서 이런 문제를 막았다.
- **왜 에러를 문자열 파싱으로 처리했는가**: 현재 공통 API 에러 응답은 `Resp.fail(...)`의 `msg` 문자열에 에러 코드와 메시지를 같이 담는 구조다. 이번에는 기존 공통 포맷을 건드리지 않고 화면에서 필요한 수준만 분기하는 것이 가장 안전했다. 나중에 공통 에러 body 구조가 생기면 이 부분은 더 깔끔하게 바꿀 수 있다.
- **테스트 범위 선택 이유**: API 자체는 이미 `task-023`에서 충분히 검증됐기 때문에, 이번에는 상세 화면이 실제로 수정 UI를 노출하는지와 전체 `admin.patient` 범위가 깨지지 않는지에 초점을 맞췄다.
