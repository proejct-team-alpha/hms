## 1. 전체 작업 흐름

1. 먼저 [task-029.md](/c:/Users/HSystem/hms/doc/dev-c/task/task-029.md)와 [workflow-029.md](/c:/Users/HSystem/hms/doc/dev-c/workflow/workflow-029.md)를 기준으로, 규칙 목록에서 필요한 UX가 “행 전체 클릭 상세 이동 + 행별 삭제 버튼 + 목록 유지형 삭제”라는 점을 다시 고정했다.
2. 기존 [rule-list.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/rule-list.mustache)를 점검해 제목 anchor 중심 구조를 행 전체 클릭 구조로 바꾸고, 관리 컬럼과 삭제 버튼, toast 영역을 추가했다.
3. 목록 행에는 `data-rule-row`, `data-rule-detail-url`을 부여해 `GET /admin/rule/detail?ruleId={id}`로 이동하게 만들고, 삭제 버튼에는 `data-rule-delete-url`을 부여해 기존 `POST /admin/api/rules/{id}` API를 그대로 호출하도록 연결했다.
4. 삭제 버튼 클릭 시 `confirm()` 확인, `event.stopPropagation()`으로 행 클릭과 분리, AJAX 삭제 성공 시 해당 행 제거, 마지막 행이면 empty state 생성, 성공/실패 toast 표시까지 한 흐름으로 묶었다.
5. 마지막으로 [AdminRuleControllerTest.java](/c:/Users/HSystem/hms/src/test/java/com/smartclinic/hms/admin/rule/AdminRuleControllerTest.java) 기대값을 갱신하고, `AdminRuleControllerTest`와 `AdminRuleApiControllerTest`를 함께 실행해 렌더링과 삭제 API 계약을 재검증했다.

## 2. 핵심 결과

```text
GET /admin/rule/list
-> 각 행 전체 클릭
-> GET /admin/rule/detail?ruleId={id}

GET /admin/rule/list
-> 행별 삭제 버튼 클릭
-> confirm()
-> POST /admin/api/rules/{id}
-> 성공 시 해당 행만 제거 + 성공 toast
-> 실패 시 실패 toast
```

## 3. 쉬운 비유

- 이전에는 목록에서 규칙 하나를 열려면 제목 텍스트라는 작은 손잡이만 잡아야 하는 구조였다.
- 이번 작업은 그 손잡이를 “행 전체”로 넓힌 것이다. 그래서 사용자는 목록에서 더 자연스럽게 상세로 들어갈 수 있다.
- 삭제도 별도 버튼으로 분리해, “행을 눌렀더니 상세로 가는 동작”과 “정말 삭제하는 동작”이 서로 섞이지 않게 정리했다.

## 4. 기술 딥다이브

- **행 전체 클릭 구조 전환**: [rule-list.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/rule-list.mustache)에 `data-rule-row`, `data-rule-detail-url`, `tabindex="0"`를 추가해 클릭과 키보드 Enter로 상세 이동이 가능하게 했다.
- **삭제 버튼과 이벤트 분리**: 각 행 우측에 `관리` 컬럼과 `삭제` 버튼을 추가하고, 삭제 버튼에서는 `preventDefault()` + `stopPropagation()`을 사용해 행 전체 클릭과 충돌하지 않게 했다.
- **기존 삭제 API 재사용**: 삭제는 새 API를 만들지 않고 기존 [AdminRuleApiController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleApiController.java)의 `POST /admin/api/rules/{id}`를 그대로 호출했다.
- **목록 유지형 삭제 UX**: 성공 시 전체 리로드 대신 삭제된 `<tr>`만 제거하고, 남은 데이터가 0건이면 JS에서 empty state row를 동적으로 추가하도록 만들었다.
- **toast 피드백**: 고정형 toast 영역을 추가해 성공/실패 메시지를 페이지 내에서 보여주고 자동으로 숨기도록 처리했다.
- **비JSON 실패 응답 대비**: 응답이 JSON이 아닐 가능성까지 고려해 `content-type`을 보고 파싱하는 보조 로직을 넣어, 실패 시에도 toast로 안전하게 안내되도록 했다.
