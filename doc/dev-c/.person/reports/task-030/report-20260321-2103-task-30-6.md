## 2. 핵심 확인 포인트

```mustache
{{#categoryError}}
  <p class="mt-2 text-sm font-medium text-red-600">{{categoryError}}</p>
{{/categoryError}}

{{#titleError}}
  <p class="mt-2 text-sm font-medium text-red-600">{{titleError}}</p>
{{/titleError}}

{{#contentError}}
  <p class="mt-2 text-sm font-medium text-red-600">{{contentError}}</p>
{{/contentError}}
```

```mustache
{{#errorMessage}}
  <div class="rounded-xl border border-red-200 bg-red-50 px-4 py-3 text-sm text-red-700">
    {{errorMessage}}
  </div>
{{/errorMessage}}
```

- 첫 번째 블록은 [_rule-form.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/_rule-form.mustache)에 이미 들어 있는 공통 필드 오류 출력이다.
- 두 번째 블록은 [rule-new.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/rule-new.mustache)와 [rule-detail.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/rule-detail.mustache)에 공통으로 존재하는 상단 오류 안내 영역이다.
- 이번 단계 결론은 "템플릿 구조상 이미 등록/수정이 같은 오류 키와 같은 문구 계약을 공유하고 있으므로 추가 템플릿 수정은 불필요"다.

## 3. 쉬운 비유

- 이번 단계는 새 문장을 쓰는 작업이라기보다, 등록 화면과 수정 화면이 같은 안내판을 같이 쓰고 있는지 확인하는 점검 작업에 가까웠다.
- 실제로는 두 화면이 같은 폼 partial을 같이 보고 있었기 때문에, 안내판을 하나만 고쳐도 둘 다 같은 문구를 보게 되는 구조가 이미 완성돼 있었다.

## 4. 기술 메모

- `rule-new.mustache`는 등록 전용 wrapper이지만 실제 입력 필드와 오류 메시지는 [_rule-form.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/_rule-form.mustache)에 위임한다.
- `rule-detail.mustache`도 수정 전용 wrapper일 뿐 동일 partial을 재사용하므로, DTO 메시지 계약이 바뀌어도 템플릿 쪽은 같은 키 이름만 유지하면 된다.
- 이번 단계에서는 production 코드 변경이 없어야 자연스럽다고 판단했고, 대신 컨트롤러 테스트를 재실행해 same-view 렌더링 회귀가 없는지만 확인했다.
