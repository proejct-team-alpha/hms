## 4. 기술 딥다이브 (Technical Deep-dive)

- **DTO 어노테이션 검증**: 등록 DTO와 수정 DTO 모두 이미 `@NotBlank`, `@Size`, `@NotNull`을 사용하고 있다. 즉 이번 작업은 검증 인프라를 새로 만드는 것이 아니라, 문서 계약과 현재 어노테이션 메시지 문구를 정렬하는 작업에 더 가깝다.
- **SSR same-view 재렌더링 패턴**: [AdminRuleController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleController.java)는 등록 실패 시 `renderForm(...)`, 수정 실패 시 `renderDetail(...)`로 되돌리는 구조를 이미 사용한다. 그래서 등록은 `admin/rule-new`, 수정은 `admin/rule-detail` same-view 계약이 코드상으로도 성립해 있다.
- **오류 속성 바인딩 구조**: [SsrValidationViewSupport.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/common/util/SsrValidationViewSupport.java)가 `field + "Error"` 키로 요청 속성을 주입하고, [admin/_rule-form.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/_rule-form.mustache)가 `titleError`, `contentError`, `categoryError`를 출력한다. 즉 화면 쪽 연결선도 이미 이어져 있다.
- **옵션 데이터 재세팅 경로**: 컨트롤러의 `populateFormAttributes(...)`는 카테고리 선택 상태와 `activeChecked`를 다시 세팅한다. 수정 화면은 상세 정보 모델과 폼 모델을 동시에 다루므로, 이후 단계에서는 "문구 정렬" 외에도 이 재세팅이 길이 초과 케이스에서도 유지되는지 테스트로 고정하면 된다.
