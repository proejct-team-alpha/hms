private Object renderUpdateValidationFailure(
        HttpServletRequest req,
        Model model,
        UpdateAdminRuleRequest request,
        BindingResult bindingResult,
        RedirectAttributes redirectAttributes) {
    try {
        AdminRuleDetailResponse detail = adminRuleService.getRuleDetail(request.ruleId());
        applyFormErrors(req, bindingResult);
        return renderDetail(req, model, detail, request);
    } catch (CustomException ex) {
        redirectAttributes.addFlashAttribute("errorMessage", ex.getMessage());
        return redirectTo("/admin/rule/list");
    }
}

## 3. 쉬운 비유 (Easy Analogy)

- 이번 단계는 수정 화면에서 실수했을 때 사용자가 쓰던 종이를 잃어버리지 않는지 확인하는 점검과 비슷하다.
- 상세 화면은 옆에 참고용 원본 문서가 놓여 있는 책상이고, 수정 폼은 그 위에 덧쓰는 메모지라고 볼 수 있다.
- 이번 테스트는 "메모를 잘못 써도 원본 문서는 그대로 보이고, 메모에 적던 내용도 안 사라지는가?"를 직접 확인한 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **상세 모델과 폼 모델 공존 검증**: 수정 화면은 [rule-detail.mustache](/c:/Users/HSystem/hms/src/main/resources/templates/admin/rule-detail.mustache)가 `rule` 상세 정보와 `model` 폼 입력을 같이 사용한다. 그래서 테스트에서도 `request().attribute("rule", detail)`와 `request().attribute("model", invalidRequest)`를 동시에 검증해 두 모델이 섞이지 않는다는 점을 고정했다.
- **same-view 재렌더링 + 선택값 유지**: `renderDetail(...)`는 `populateFormAttributes(...)`를 통해 category selected 상태와 `activeChecked`를 다시 세팅한다. 이번 단계의 title 길이 초과 테스트는 이 재세팅이 실제로 동작하는지 HTML의 `selected` 상태까지 포함해 검증한 것이다.
- **체크박스 바인딩 특성 확인**: HTML 체크박스는 체크 해제 시 아예 파라미터가 오지 않기 때문에, 수정 요청 DTO의 `active`는 `false`가 아니라 `null`로 바인딩된다. 컨트롤러는 이를 `Boolean.TRUE.equals(active)` 기반으로 해석해 화면에서는 `activeChecked=false`로 처리한다. 이번 단계에서 이 차이를 테스트로 확인한 덕분에 이후 회귀를 더 안정적으로 막을 수 있다.
