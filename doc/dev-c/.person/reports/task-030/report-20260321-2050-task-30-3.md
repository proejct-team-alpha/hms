## 3. 쉬운 비유 (Easy Analogy)

- 이번 단계는 등록 화면과 수정 화면에 붙어 있던 안내 표지판 문구를 같은 기준으로 통일한 작업과 비슷하다.

- 그래서 이제 사용자는 "등록할 때는 이렇게 말하고, 수정할 때는 저렇게 말하는" 어색한 차이를 덜 겪게 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **등록/수정 DTO 메시지 계약 통일**: [UpdateAdminRuleRequest.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/dto/UpdateAdminRuleRequest.java)의 `title`, `content`, `category` 메시지를 등록 DTO와 동일한 기준으로 맞췄다. 이번 작업의 핵심은 검증 규칙 추가가 아니라 "문서 계약과 사용자 메시지 일관성" 확보다.
- **수정 전용 식별자 필드 정리**: 수정 DTO는 등록 DTO와 달리 `ruleId`를 반드시 가져야 하므로, 이 필드의 `@NotNull` 메시지도 읽기 가능한 형태로 함께 정리했다. 이건 이번 인터뷰 핵심 계약은 아니지만, 수정 요청 DTO를 만지는 김에 함께 정돈한 보조 개선이다.
- **SSR 회귀 테스트 동기화**: 수정 실패 same-view 기대값을 검증하는 [AdminRuleControllerTest.java](/c:/Users/HSystem/hms/src/test/java/com/smartclinic/hms/admin/rule/AdminRuleControllerTest.java)가 문자열 비교를 하고 있어서, DTO 메시지 변경에 맞춰 테스트 기대값도 같이 업데이트했다. 이렇게 해야 메시지 계약 변경이 실제 렌더링에도 반영됐는지 바로 확인할 수 있다.
