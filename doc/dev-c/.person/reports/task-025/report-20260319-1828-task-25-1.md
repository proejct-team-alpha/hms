## 2. 핵심 코드/구조 (Core Logic)

```java
@GetMapping("/list")
public String list(Model model) {
    List<AdminRuleDto> rules = adminRuleService.getRuleList();
    model.addAttribute("rules", rules);
    model.addAttribute("hasRules", !rules.isEmpty());
    model.addAttribute("pageTitle", "병원 규칙 관리");
    return "admin/rule-list";
}
```

```java
public List<AdminRuleDto> getRuleList() {
    return hospitalRuleRepository.findAllByOrderByCreatedAtDesc()
            .stream()
            .map(AdminRuleDto::new)
            .collect(Collectors.toList());
}
```

현재 구조는 “전체 조회 + 카드 렌더링”에 맞춘 단순한 목록 구조다. 검색형 목록으로 완성하려면 컨트롤러 파라미터, 서비스 응답 구조, 리포지토리 조회 책임, 템플릿 UI, 테스트 범위를 함께 확장해야 한다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **Controller 관점**: 현재 `Model` 기반 SSR 렌더링은 이미 맞지만, 검색형 목록에 필요한 쿼리 파라미터 구조가 없다. 다음 단계에서 `page`, `size`, `category`, `active`, `keyword`를 받아 서비스에 위임하는 형태로 바뀌어야 한다.
- **Service 관점**: `List<AdminRuleDto>` 반환은 페이지네이션과 필터 상태 유지에 불리하다. 따라서 다음 단계에서는 환자/직원 목록처럼 페이지 링크, 현재 필터, 총 건수까지 포함하는 응답 DTO가 필요하다.
- **Repository 관점**: 현재는 `findAllByOrderByCreatedAtDesc()` 하나뿐이라 선택지가 명확하다. 파생 메서드 몇 개로 커버할지, 아니면 조합 필터를 위해 `@Query`를 도입할지 다음 단계에서 결정하면 된다. 제목 검색이 `LIKE` 조건을 요구하므로, 조합이 많아질 경우 `@Query`가 더 깔끔할 가능성이 높다.
- **DTO 관점**: `AdminRuleDto`의 `categoryText`, `activeText`, `activeBadgeClass`는 이미 유용하다. 규칙 목록 완성 작업은 이 DTO를 없애는 것이 아니라, 목록 응답 바깥쪽에 페이지네이션과 필터 상태를 감싸는 응답 DTO를 추가하는 방향이 적절하다.
- **Template 관점**: 현재 `rule-list.mustache`는 인코딩 손상까지 있어 단순 UI 추가보다 전체 정리가 더 안전하다. 검색 바와 필터 UI를 붙일 때 기존 구조를 부분 수정하다가 더 꼬일 수 있으니, 다음 단계에서는 화면을 한 번에 다시 정리하는 편이 좋다.
- **테스트 관점**: 기존 테스트는 전체 조회 기준만 확인하므로, 다음 단계에서 파라미터 전달, 필터 적용, 빈 결과, 페이지 이동 유지 같은 시나리오를 새로 채워야 한다. 이번 점검 덕분에 어떤 테스트가 부족한지 범위가 분명해졌다.
