## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 `AdminRuleDto`를 유지했는가**: 이미 카테고리 한글 라벨, 활성 상태 텍스트, 배지 클래스까지 포함하고 있어서 row 단위 데이터로 충분히 유용하다. 이 DTO를 다시 쪼개거나 지우기보다는, 바깥쪽에 페이지/필터 상태를 담는 response를 추가하는 편이 훨씬 단순하고 안전하다.
- **왜 outer response가 필요한가**: 검색형 목록 화면은 단순 `List<T>`만으로는 부족하다. 현재 선택된 카테고리, 활성 필터, 키워드, 총 건수, 이전/다음 링크, 페이지 링크 리스트가 모두 필요하다. 이런 상태를 하나의 response로 감싸야 컨트롤러와 Mustache가 훨씬 단순해진다.
- **왜 페이지 링크 DTO를 따로 두었는가**: 페이지네이션 UI는 결국 `(page, url, active)` 조합만 있으면 된다. 이 구조를 작은 DTO로 분리하면 다른 목록 화면과 패턴을 맞추기 쉽고, Mustache에서도 단순 반복 렌더링으로 처리할 수 있다.
- **정렬 기준을 `createdAt desc, id desc`로 잡은 이유**: `HospitalRule`은 이미 `createdAt`을 가지고 있고, 관리자 목록 화면에서 최신 등록순은 가장 자연스럽다. `id desc`를 보조 정렬로 두면 동일 시각 데이터에서도 결과가 안정적이다.
- **기본값을 이 단계에서 고정한 이유**: 구현 단계에서 기본값이 흔들리면 컨트롤러, 서비스, 테스트가 동시에 수정된다. 그래서 `page=1`, `size=10`, `category=ALL`, `active=ALL`, `keyword=''`를 지금 고정해두는 편이 훨씬 좋다.
