## 1. 전체 작업 흐름
1. 먼저 [AdminStaffController.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffController.java)와 [AdminStaffService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffService.java)를 다시 확인해 보니, API 경로에는 본인 계정 차단 규칙이 들어가 있었지만 SSR 저장 경로는 여전히 `updateStaff(request)` 구버전 시그니처를 타고 있었다.
2. 그래서 이번 단계의 핵심을 `SSR 저장도 API와 같은 서비스 규칙을 타게 맞추는 것`으로 잡고, 컨트롤러에서 현재 로그인 사용자명을 서비스까지 넘기도록 정리했다.
3. 이어서 `퇴사 일시`와 `active` 관계를 등록/수정 모두 같은 기준으로 맞췄다. 즉 `retiredAt`이 과거면 저장 즉시 비활성화되고, 그렇지 않으면 사용자가 요청한 `active` 상태를 유지하는 쪽으로 규칙을 통합했다.
4. 마지막으로 [AdminStaffServiceTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/staff/AdminStaffServiceTest.java)를 clean UTF-8 기준으로 다시 정리하면서, Task 35-5가 요구하는 핵심 시나리오를 테스트로 닫았다.

## 2. 핵심 전달 코드
```java
@Transactional
public String createStaff(CreateAdminStaffRequest request) {
    ...
    LocalDateTime retiredAt = request.retiredAt();
    boolean active = resolveActiveState(request.active(), retiredAt);
    staff.update(staff.getName(), null, active);
    staff.updateRetiredAt(retiredAt);
    ...
}

@Transactional
public String updateStaff(UpdateAdminStaffRequest request, String currentUsername) {
    ...
    validateSelfAccountUpdate(staff, request, currentUsername);
    validateReactivation(staff, request.active());

    LocalDateTime retiredAt = request.retiredAt();
    boolean active = resolveActiveState(request.active(), retiredAt);

    staff.update(request.name().trim(), null, active);
    staff.updateRetiredAt(retiredAt);
    ...
}
```

```java
@PostMapping("/update")
public String update(..., Authentication authentication, HttpServletRequest req) {
    ...
    String successMessage = adminStaffService.updateStaff(request, authentication.getName());
    ...

```

- `resolveActiveState(...)`를 공통으로 써서 등록/수정 모두 같은 기준으로 `retiredAt`과 `active` 관계를 계산하게 했다.
- `AdminStaffController.update(...)`는 이제 SSR 저장에서도 현재 로그인 사용자명을 서비스까지 넘긴다.
- 즉 본인 계정 차단 규칙이 SSR/API 양쪽 모두 서비스 기준으로 통일됐다.

## 3. 쉬운 비유
- 이번 작업은 출입문이 두 개 있는 사무실에 같은 출입 규칙을 붙이는 일과 비슷했다.
- API 문은 이미 `본인 계정은 스스로 잠글 수 없다`는 규칙이 붙어 있었는데, SSR 저장 문은 아직 예전 규칙을 쓰고 있었다.
- 그래서 두 문이 모두 같은 경비실([AdminStaffService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffService.java))을 통하게 바꿨고, 거기서 `본인 계정 차단`, `재활성화 불가`, `과거 퇴사 일시 즉시 비활성화`를 한 번에 판단하도록 정리한 셈이다.

## 4. 기술 딥다이브
- **SSR 저장 경로 통합**: [AdminStaffController.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffController.java)의 `POST /admin/staff/update`가 이제 `authentication.getName()`을 서비스에 함께 넘긴다. 이 덕분에 SSR 저장도 API와 동일하게 `validateSelfAccountUpdate(...)`를 탄다.
- **등록/수정 규칙 일관화**: 기존에는 `updateStaff(...)`만 과거 `retiredAt` 즉시 비활성화를 처리하고 있었고, `createStaff(...)`는 `retiredAt`을 실제 저장 상태에 반영하지 못하고 있었다. 이번에 `resolveActiveState(...)`를 공통으로 두면서 등록/수정 둘 다 같은 규칙을 쓰게 됐다.
- **재활성화 차단 유지**: `validateReactivation(...)`는 그대로 유지하되, SSR 저장도 같은 경로를 타게 바꿔서 비활성 직원의 `active=false -> true` 시도가 UI 저장/API 모두에서 막히는 기반을 맞췄다.
- **테스트 재정비**: [AdminStaffServiceTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/staff/AdminStaffServiceTest.java)는 인코딩 흔들림 때문에 부분 patch보다 전체 정리가 안전하다고 판단했다. 이번에 아래 시나리오를 명시적으로 검증하게 바꿨다.
  - 과거 `retiredAt` 등록 시 즉시 비활성화
  - 과거 `retiredAt` 수정 시 즉시 비활성화
  - 본인 계정 비활성화 차단
  - 본인 계정 `retiredAt` 변경 차단
  - 비활성 직원 재활성화 차단
