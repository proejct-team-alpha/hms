## 1. 전체 작업 흐름
1. 먼저 [SecurityConfig.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/config/SecurityConfig.java), [WebMvcConfig.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/config/WebMvcConfig.java), [AuthController.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/auth/AuthController.java), [LayoutModelInterceptor.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/common/interceptor/LayoutModelInterceptor.java)를 확인해서 자동 비활성화 후 어느 지점에서 세션을 정리하는 게 가장 안전한지 먼저 판단했다.
2. 점검 결과 로그인 자체는 이미 `findByUsernameAndActiveTrue(...)` 기준이라 막혀 있었고, 부족한 부분은 `이미 로그인 중인 계정`의 다음 요청 처리뿐이었다. 그래서 `LayoutModelInterceptor`에 책임을 섞지 않고, 비활성 계정만 전담하는 별도 인터셉터를 두는 방향으로 정리했다.
3. 그다음 [AdminStaffRepository.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffRepository.java), [AdminStaffService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffService.java), [StaffRetirementScheduler.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/StaffRetirementScheduler.java)를 연결해서 `retiredAt <= now`인 활성 직원만 1시간마다 비활성화되도록 묶었다.
4. 마지막으로 [AuthController.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/auth/AuthController.java)와 [login.mustache](c:/workspace/Team/hms/src/main/resources/templates/auth/login.mustache)에 자동 로그아웃 안내 문구를 반영하고, 서비스/인터셉터/로그인 화면 테스트까지 같이 닫았다.

## 2. 핵심 전달 코드
```java
@Transactional
public int deactivateExpiredStaffs() {
    LocalDateTime now = LocalDateTime.now();
    List<Staff> expiredStaffs = adminStaffRepository
            .findAllByActiveTrueAndRetiredAtLessThanEqual(now);

    expiredStaffs.forEach(staff -> staff.update(staff.getName(), staff.getDepartment(), false));
    return expiredStaffs.size();
}
```

```java
@Scheduled(fixedRate = 3_600_000L)
public void deactivateExpiredStaffs() {
    adminStaffService.deactivateExpiredStaffs();
}
```

- 스케줄러는 1시간마다 만료된 활성 직원만 서비스에 넘긴다.
- 서비스는 실제 비활성화 규칙만 담당한다.
- 인터셉터는 이미 로그인 중인 비활성 계정을 다음 요청에서 정리하고, 로그인 페이지로 안내한다.

## 3. 쉬운 비유
- 이번 작업은 `퇴사 예약 시간이 되면 자동으로 출입카드를 정지시키고, 이미 건물 안에 있는 사람은 다음 출입문 통과 시점에 안내 후 밖으로 보내는 흐름`을 시스템으로 옮긴 것에 가깝다.
- 스케줄러는 일정 시간마다 출입카드 만료 목록을 확인하는 관리실이고,
- `InactiveStaffLogoutInterceptor`는 문 앞에서 카드 상태를 한 번 더 확인하는 경비원 역할이다.
- 로그인 페이지 문구는 왜 입장이 막혔는지 알려주는 안내판에 해당한다.

## 4. 기술 딥다이브
- **자동 비활성화 서비스화**: `AdminStaffService.deactivateExpiredStaffs()`를 추가해, 스케줄러가 직접 엔티티를 만지지 않고 서비스 규칙을 재사용하게 했다. 덕분에 스케줄러 테스트와 비즈니스 테스트를 분리할 수 있게 됐다.
- **리포지토리 범위 명확화**: `AdminStaffRepository.findAllByActiveTrueAndRetiredAtLessThanEqual(...)`는 이미 비활성인 직원은 건드리지 않고, 만료된 활성 직원만 찾도록 범위를 명확히 했다.
- **인터셉터 분리**: `LayoutModelInterceptor`는 화면 공통 모델 주입이라는 역할이 선명해서, 자동 로그아웃 책임을 얹지 않는 편이 유지보수에 안전했다. 그래서 `InactiveStaffLogoutInterceptor`를 따로 두고 `preHandle`에서만 세션을 정리하게 했다.
- **테스트 컨텍스트 유연화**: WebMvc 테스트가 새 인터셉터 때문에 `StaffRepository` 빈을 요구하지 않도록, [WebMvcConfig.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/config/WebMvcConfig.java)의 인터셉터 빈 등록을 `@ConditionalOnBean(StaffRepository.class)` 기준으로 정리했다. 이걸 안 하면 `admin.staff` WebMvc 테스트들이 전부 컨텍스트 로딩에서 실패했다.
- **로그인 안내 문구 연결**: `AuthController`는 `deactivated` query param을 받아 모델에 담고, `login.mustache`는 그 값을 기준으로 `계정이 비활성화되어 자동 로그아웃되었습니다.` 문구를 렌더링한다.
