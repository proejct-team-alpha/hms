## 2. 핵심 전달 코드 (Core Logic)

```java
// AdminStaffController.update(...)
try {
    String successMessage = adminStaffService.updateStaff(request);
    redirectAttributes.addFlashAttribute("successMessage", successMessage);
    return "redirect:/admin/staff/list";
} catch (CustomException ex) {
    req.setAttribute("errorMessage", ex.getMessage());
    req.setAttribute("model", adminStaffService.getEditForm(request));
    return "admin/staff-form";
}
```

위 흐름에서 중요한 점은, SSR 수정 저장이 현재 로그인 사용자 이름을 서비스에 넘기지 않는다는 점이다. 그래서 "본인 계정은 비활성화할 수 없습니다." 규칙을 넣으려면 컨트롤러 시그니처와 서비스 호출을 함께 바꿔야 한다.

```java
// AdminStaffApiController.updateStaff(...)
UpdateAdminStaffRequest serviceRequest = new UpdateAdminStaffRequest(
        staffId,
        request.name(),
        request.departmentId(),
        request.password(),
        request.active(),
        request.availableDays()
);

String successMessage = adminStaffService.updateStaff(serviceRequest);
```

API도 같은 구조다. 지금은 `staffId`와 수정 값만 서비스로 내려가고, "누가 수정하고 있는지" 정보는 서비스에서 모른다. 즉 API 차단 규칙도 현재 사용자 정보 전달이 먼저 필요하다.

```java
// CustomUserDetailsService.loadUserByUsername(...)
Staff staff = staffRepository.findByUsernameAndActiveTrue(username)
        .orElseThrow(() -> new UsernameNotFoundException("아이디 또는 비밀번호가 올바르지 않습니다."));
```

인증은 이미 `active=true` 조건으로 동작한다. 그래서 직원이 비활성화되면 다음 로그인부터는 막힌다. 이번 작업의 핵심은 여기에 더해 "이미 로그인 중인 세션을 다음 요청에서 어떻게 정리할지"를 붙이는 것이다.

## 3. 초등학생도 이해할 수 있게 비유 (Easy Analogy)

- 이번 점검은 학교 출입카드를 관리하는 규칙을 점검하는 일과 비슷하다.
- 지금 시스템은 "카드가 비활성 상태면 새로 출입할 수 없다"는 규칙은 이미 있다.
- 그런데 아직 부족한 부분은 두 가지다.
  - 본인이 자기 카드에 스스로 "출입 금지"를 걸 수 있으면 안 된다.
  - 퇴사 시간이 지나서 카드가 비활성화되면, 이미 학교 안에 있는 사람도 다음 문을 통과할 때는 막아야 한다.
- 그래서 다음 단계에서는 카드 상태를 바꾸는 창구(SSR/API), 카드 검사기(인증), 자동 만료 체크(스케줄러)를 한 번에 맞춰야 한다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **서비스 공통 규칙 필요**: `AdminStaffService.updateStaff(...)`가 현재는 `request.active()`를 그대로 `staff.update(...)`에 전달한다. 본인 계정 차단, 과거 퇴사 일시 즉시 비활성화, 재활성화 불가 규칙은 컨트롤러가 아니라 서비스의 공통 비즈니스 규칙으로 들어가야 한다.
- **인증의 현재 기준점**: `CustomUserDetailsService`와 `SecurityUtils`가 둘 다 `findByUsernameAndActiveTrue(...)`를 사용한다. 이 말은 비활성 계정이 되면 "다음 인증"은 이미 막혀 있다는 뜻이다. 하지만 현재 `SecurityConfig`에는 비활성화된 상태를 감지해 다음 요청에서 자동 로그아웃시키는 별도 처리 지점이 없어서, 그 부분을 새로 설계해야 한다.
- **UI와 DTO 동시 수정 필요**: `staff-form.mustache`, `CreateAdminStaffRequest`, `UpdateAdminStaffRequest`, `UpdateAdminStaffApiRequest` 어디에도 아직 `퇴사 일시`가 없다. 즉 화면만 먼저 바꾸는 접근은 위험하고, 다음 단계는 DTO/서비스 계약을 먼저 고정한 뒤 SSR/API를 함께 수정하는 순서가 더 안전하다.
- **현재 보호 규칙의 빈 구멍**: 본인 계정 비활성화 차단은 현재 `deactivateStaff(staffId, currentUsername)`에만 들어 있다. 즉 "직원 수정 저장"과 "직원 수정 API"는 아직 같은 보호 규칙을 공유하지 않는다. 이번 작업의 첫 번째 기술 포인트는 이 규칙을 모든 수정 경로로 끌어올리는 일이다.
