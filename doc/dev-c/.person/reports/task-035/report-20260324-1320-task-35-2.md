## 3. 초등학생도 이해할 수 있게 비유 (Easy Analogy)

- 이번 설계는 직원 계정을 출입증으로 생각하면 이해가 쉽다.
- `active`는 지금 출입증이 켜져 있는지 꺼져 있는지다.
- `퇴사 일시`는 "이 시간부터 이 출입증은 자동으로 꺼진다"는 예약이다.
- 중요한 규칙은 두 가지다.
  - 본인은 자기 출입증을 스스로 끄거나, 자동 종료 예약을 걸 수 없다.
  - 다른 관리자는 다른 사람 출입증에 종료 시간을 넣을 수 있다.
- 그리고 종료 시간이 지나면, 그 사람이 이미 건물 안에 있더라도 다음 문을 통과하려고 할 때는 자동으로 막히고 로그인 화면으로 보내는 구조다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **저장 위치 선택**: `퇴사 일시`는 `Doctor`, `Nurse` 같은 역할별 상세가 아니라 `Staff`에 둬야 한다. 이유는 비활성화와 로그인 가능 여부가 역할별 부가정보가 아니라 계정 상태 그 자체이기 때문이다. 현재 `active`도 `Staff`에 있으니, `retiredAt`도 같은 경계 안에 두는 편이 규칙 응집도가 높다.
- **서비스 공통 규칙화**: 본인 계정 차단은 SSR 컨트롤러 한 군데나 API 한 군데에만 넣으면 구멍이 생긴다. 그래서 `AdminStaffService.updateStaff(...)`에 현재 로그인 사용자 정보를 함께 받고, 그 안에서 본인 계정 여부와 `active`, `retiredAt` 변경 여부를 모두 비교하는 공통 규칙으로 올리는 것이 맞다.
- **즉시 비활성화와 자동 비활성화의 관계**: 다른 관리자가 이미 지난 `퇴사 일시`를 저장하면 저장 즉시 비활성화하고, 미래 시각이면 스케줄러가 1시간마다 확인해 시점 도달 후 비활성화하는 구조로 정리했다. 이렇게 하면 즉시 반영과 예약 반영을 하나의 `retiredAt <= now` 규칙으로 통일할 수 있다.
- **다음 요청 자동 로그아웃 위치**: `LayoutModelInterceptor`는 공통 레이아웃 모델 주입 책임만 가지고 있고, 메모리 규칙상 함부로 건드리면 안 되는 보호 영역에 가깝다. 그래서 이번 설계에서는 별도 인증 상태 인터셉터를 추가해 `preHandle`에서 활성 직원 존재 여부를 확인하고, 비활성 상태면 세션 무효화와 `SecurityContext` 정리를 수행한 뒤 `/login?deactivated=true`로 보내는 방향이 가장 안전하다고 정리했다.
