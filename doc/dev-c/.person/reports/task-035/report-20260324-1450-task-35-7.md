## 4. 기술 딥다이브
- **SSR 경로와 API 경로의 규칙 닫기**: 서비스 규칙이 들어갔다고 끝이 아니라, SSR과 API 각각에서 사용자가 실제로 보게 되는 실패 형태가 다르다. SSR은 same-view 에러 속성(`activeError`)을, API는 HTTP 400과 공통 예외 포맷을 확인해야 해서 둘을 분리해 테스트하는 게 맞았다.
- **스케줄러는 얇게 유지**: 자동 비활성화 스케줄러는 단순해 보여도, 이 테스트가 없으면 나중에 스케줄러 안에 비즈니스 규칙이 섞이기 쉽다. [StaffRetirementSchedulerTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/staff/StaffRetirementSchedulerTest.java)로 “서비스 위임만 한다”는 계약을 고정해뒀다.
- **인코딩 안정화**: PowerShell `Set-Content -Encoding UTF8`은 환경에 따라 BOM이 들어가면서 Java 소스 첫 줄에서 `illegal character: '\ufeff'`를 만들 수 있다. 이번에는 두 테스트 파일을 UTF-8 no BOM으로 다시 작성해, 이후 같은 종류의 테스트 보강이 들어와도 컴파일 단계에서 흔들리지 않게 만들었다.
- **Task 문서와 실제 테스트 범위 맞추기**: Task 35-7 체크리스트에는 서비스/SSR/API/스케줄러/인증 흐름까지 넓게 적혀 있었지만, 실코드 기준으로는 이미 커버된 부분이 많았다. 이번 단계는 그 사실을 문서 메모에 같이 남겨서, 다음 사람이 Task 35-8에서 “왜 여기서는 새 테스트가 네 개만 추가됐는지”를 바로 이해할 수 있게 했다.
