## 3. 쉬운 비유
- 이번 작업은 이미 지어진 건물에 대해 마지막 준공 점검표를 붙이는 과정과 비슷하다.
- 출입 통제 장치, 자동 잠금 장치, 안내 표지판은 앞 단계에서 이미 설치돼 있었고, 이번 단계는 그 장치들이 실제로 다 동작하는지 다시 확인한 뒤 체크리스트에 서명하는 단계였다.
- 그래서 겉보기엔 문서 수정처럼 보이지만, 실제로는 “이제 이 규칙을 팀이 믿고 써도 된다”는 상태를 만드는 마감 작업에 가깝다.

## 4. 기술 딥다이브
- **문서와 코드 동기화**: workflow/task 문서가 실제 구현 상태보다 늦게 따라오면, 다음 작업자는 이미 구현된 규칙도 다시 의심하게 된다. 이번 단계에서는 수용 기준을 하나씩 코드와 테스트에 대조해 문서 신뢰도를 회복하는 데 집중했다.
- **검증 결과를 문서에 남기는 이유**: Task 35는 SSR, API, 서비스, 스케줄러, 인터셉터, 로그인 화면까지 범위가 넓다. 이런 작업은 “테스트 통과”를 문서에 남겨두지 않으면 다음 사람이 변경 영향 범위를 읽기 어렵다. 그래서 범위 테스트와 전체 테스트를 모두 다시 실행한 뒤 결과를 명시했다.
- **최종 완료 기준의 의미**: 본인 계정 보호, 즉시 비활성화, 자동 비활성화, 자동 로그아웃은 서로 다른 레이어에 퍼져 있다. 전체 완료 기준을 모두 체크 처리한 것은 단순 체크가 아니라, 이 레이어들이 함께 동작하는지 확인한 결과를 남긴 것이다.
