## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 이번 작업은 선생님이 숙제를 다시 제출하라고 할 때, 이름표는 원래 붙어 있는 채로 두고, 틀린 답만 다시 고쳐 쓰게 하는 것과 비슷하다.
- 수정 화면에서는 로그인 아이디나 사번처럼 이미 정해진 정보는 그대로 보여주고, 이름이나 부서처럼 바꿀 수 있는 것만 다시 고치게 해야 한다.
- 그래서 “부서를 안 썼어요” 같은 빨간 메모는 보여주되, 사용자가 방금 입력한 이름은 그대로 남겨서 다시 입력하지 않게 만든 것이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **수정 DTO 검증의 의미**: 등록 때만 필수값을 검사하고 수정은 느슨하게 두면, 운영자가 편집 화면에서 빈 부서 상태를 만들어버릴 수 있다. 이번 작업은 수정 DTO에도 최소한의 구조적 유효성 기준을 맞춰 등록/수정 간 기준 차이를 줄였다.
- **읽기 전용 값과 수정 가능 값의 공존**: 수정 폼은 `username`, `employeeNumber`처럼 바뀌지 않는 값과 `name`, `departmentId`처럼 바꿀 수 있는 값이 함께 그려진다. 검증 실패 시 이 두 종류의 정보가 동시에 자연스럽게 남아 있어야 사용자 경험이 흔들리지 않는다.
- **비밀번호 선택 입력 정책 유지**: 수정 폼에서 비밀번호는 선택값이기 때문에 DTO 검증을 과하게 걸면 기존 정책과 충돌한다. 그래서 이번 단계는 이름/부서 필수 검증만 강화하고, 비밀번호는 기존 서비스의 “입력 시만 변경” 정책을 그대로 유지했다.
- **same-view 재렌더링의 장점**: 수정 실패 시 redirect로 보내버리면 어떤 값을 바꾸다가 실패했는지 맥락이 끊긴다. 반대로 same-view 재렌더링은 같은 화면 안에서 바로 에러와 기존 입력값을 함께 보여주기 때문에 관리 화면에 더 적합하다.
