## 2. 핵심 전달 코드 (Core Logic)

```java
public class CreateAdminDepartmentRequest {

    @NotBlank(message = "진료과명은 필수입니다.")
    private String name;

    private boolean active;
}
```

```java
@PostMapping("/create")
public Object create(
        @Valid @ModelAttribute CreateAdminDepartmentRequest request,
        BindingResult bindingResult,
        HttpServletRequest req) {
    if (bindingResult.hasErrors()) {
        return renderCreateValidationFailure(req, request, bindingResult);
    }

    adminDepartmentService.createDepartment(request.getName(), request.isActive());
    return redirectTo("/admin/department/list");
}
```

```java
private String renderCreateValidationFailure(
        HttpServletRequest req,
        CreateAdminDepartmentRequest request,
        BindingResult bindingResult) {
    req.setAttribute("model", adminDepartmentService.getDepartmentList(1, 10));
    req.setAttribute("pageTitle", "진료과 관리");
    req.setAttribute("createName", request.getName());
    req.setAttribute("createActive", request.isActive());
    req.setAttribute("openCreateModal", true);
    req.setAttribute("nameError", getFieldError(bindingResult, "name"));
    return "admin/department-list";
}
```

이 코드의 핵심은 진료과 등록을 단순 POST 처리에서 끝내지 않고, 검증 실패 시 같은 화면 안에서 사용자가 바로 수정할 수 있는 SSR 흐름으로 바꿨다는 점이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **`@RequestParam`에서 입력 DTO로 바꾼 이유**: 진료과 등록은 원래 `name`, `active`를 컨트롤러 파라미터로 바로 받는 구조였다. 이 방식은 빠르게 구현하기에는 좋지만 `BindingResult`와 필드별 에러 처리를 일관되게 붙이기 어렵다. 이번 단계에서는 직원 쪽과 같은 검증 UX를 맞추는 게 목표였기 때문에, 등록 전용 입력 객체를 두고 `@Valid @ModelAttribute` 패턴으로 올리는 것이 가장 자연스러웠다.
- **same-view 재렌더링의 의미**: 검증 실패 후 redirect로 보내면 에러 메시지와 입력값을 다시 조합해야 하고, 모달 화면에서는 사용자가 왜 실패했는지 맥락을 잃기 쉽다. 같은 목록 화면을 다시 렌더링하면 목록 데이터는 그대로 유지하면서도, 인라인 등록 모달만 다시 열어 문제 필드를 바로 보여줄 수 있다.
- **모달 폼의 입력값 유지 전략**: Mustache는 SPA처럼 상태를 기억하지 않기 때문에, 서버가 `createName`, `createActive`, `nameError`, `openCreateModal` 같은 값을 직접 넣어줘야 한다. 이번 작업은 “입력값과 에러 상태를 서버가 명시적으로 다시 그려준다”는 SSR 방식의 전형적인 패턴을 진료과 쪽에도 동일하게 적용한 단계다.
- **PRG와 검증 실패 처리의 균형**: 성공 시에는 기존처럼 `RedirectView`를 유지해서 공통 레이아웃 모델 값이 쿼리스트링에 섞이지 않도록 했다. 반면 실패 시에는 same-view 렌더링을 사용했다. 즉 “성공은 PRG, 실패는 same-view”라는 실무적으로 가장 편한 조합을 그대로 쓴 셈이다.
