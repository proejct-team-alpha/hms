5. 반대로 서비스 검증 실패는 기존 책임을 유지했다. 예를 들어 중복 이름처럼 도메인/서비스 규칙에 해당하는 오류는 flash 메시지 + redirect로 남겨 두고, DTO 검증 실패와 역할을 분리했다. 덕분에 “폼 입력 문제”와 “비즈니스 규칙 문제”가 사용자 경험에서도 구분된다.

## 2. 핵심 전달 코드 (Core Logic)

```java
public class UpdateAdminDepartmentRequest {

    @NotNull(message = "진료과 ID는 필수입니다.")
    private Long departmentId;

    @NotBlank(message = "진료과명은 필수입니다.")
    private String name;
}
```

```java
@PostMapping("/update")
public Object update(
        @Valid @ModelAttribute UpdateAdminDepartmentRequest request,
        BindingResult bindingResult,
        HttpServletRequest req,
        RedirectAttributes redirectAttributes) {
    if (bindingResult.hasErrors()) {
        return renderUpdateValidationFailure(req, request, redirectAttributes, bindingResult);
    }

    String successMessage = adminDepartmentService.updateDepartmentName(
            request.getDepartmentId(),
            request.getName());
    redirectAttributes.addFlashAttribute("successMessage", successMessage);
    return redirectToDetail(request.getDepartmentId());
}
```

```java
private Object renderUpdateValidationFailure(
        HttpServletRequest req,
        UpdateAdminDepartmentRequest request,
        RedirectAttributes redirectAttributes,
        BindingResult bindingResult) {
    if (request.getDepartmentId() == null) {
        redirectAttributes.addFlashAttribute("errorMessage", "입력값을 확인해 주세요.");
        return redirectTo("/admin/department/list");
    }

    AdminDepartmentDetailResponse model = adminDepartmentService.getDepartmentDetail(request.getDepartmentId());
    req.setAttribute("nameError", getFieldError(bindingResult, "name"));
    return renderDetailPage(req, model, request.getName());
}
```

이 코드의 핵심은 수정 폼의 기본 검증 오류를 서비스 예외와 섞지 않고, 상세 화면 안에서 바로 수정할 수 있는 same-view 흐름으로 올린 점이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **DTO 검증과 서비스 검증의 분리**: 빈 값이나 공백 입력은 화면 입력 단계에서 바로 잡는 것이 맞고, 중복 이름처럼 이미 저장된 데이터와 비교해야 하는 규칙은 서비스가 책임지는 것이 더 자연스럽다. 이번 단계는 이 둘을 분리해서, DTO 검증 실패는 same-view 재렌더링, 서비스 검증 실패는 flash + redirect로 나눴다.
- **same-view 재렌더링의 장점**: 상세/수정 페이지는 목록 인라인 등록보다 더 문맥이 강한 화면이다. 사용자는 “지금 어느 진료과를 수정하는지”를 보고 있는 상태라서, 검증 실패 시 다른 페이지로 튕기는 것보다 같은 상세 페이지 안에서 바로 수정하는 편이 훨씬 이해하기 쉽다.
- **`editName` 분리의 의미**: 상세 응답 모델 `model.name`은 현재 저장된 값이고, 검증 실패 후 사용자가 방금 입력한 값은 그와 다를 수 있다. 그래서 템플릿이 `{{model.name}}` 대신 `{{editName}}`을 보게 하면, 저장된 원본 데이터와 “방금 폼에 입력한 값”을 서버가 분리해서 제어할 수 있다.
- **실패 경로의 안전장치**: `departmentId` 자체가 비어 있으면 상세 페이지를 다시 그릴 기준이 없기 때문에, 그 경우는 목록으로 돌려보내고 공통 에러 메시지를 띄우는 쪽이 안전하다. 반대로 `departmentId`가 있으면 상세 응답을 다시 조회해 same-view 재렌더링을 할 수 있다. 이 분기 덕분에 잘못된 요청에도 화면 흐름이 망가지지 않는다.
