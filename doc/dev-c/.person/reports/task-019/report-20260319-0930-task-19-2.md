    req.setAttribute("model", adminStaffService.getCreateForm(request));

```java
private void applyFieldErrors(HttpServletRequest req, BindingResult bindingResult) {
    for (FieldError fieldError : bindingResult.getFieldErrors()) {
        req.setAttribute(fieldError.getField() + "Error", fieldError.getDefaultMessage());
    }
}
```

이번 단계의 핵심은 “검증 규칙 추가”보다도 “검증 실패 정보를 템플릿이 쓰기 쉬운 형태로 바꿔준다”는 점이다. Mustache는 복잡한 폼 상태 객체를 다루기보다 단순한 값 바인딩에 강하므로, 필드 에러를 request attribute로 푸는 방식이 SSR 흐름에 잘 맞았다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **Bean Validation의 역할**: `@NotNull`, `@NotBlank`는 컨트롤러에 들어오기 전 입력값의 최소 조건을 선언적으로 보장한다. 이번에는 `departmentId`를 등록 필수로 올려서 직원 등록 폼의 기준을 더 분명하게 만들었다.
- **`BindingResult`를 SSR UX로 연결하는 방식**: 검증 실패가 발생하면 단순히 `bindingResult.hasErrors()`만 보는 것으로 끝나지 않고, 각 필드 에러를 템플릿이 바로 읽을 수 있게 풀어줘야 사용자가 어느 입력을 수정해야 하는지 알 수 있다.
- **Mustache 친화적인 에러 모델링**: Spring Form 태그 대신 Mustache SSR을 쓰고 있기 때문에, 템플릿에서는 `{{#departmentIdError}}...{{/departmentIdError}}`처럼 단순한 블록 조건이 가장 안정적이다. 그래서 필드 에러를 복잡한 객체가 아니라 request attribute로 노출하는 쪽이 유지보수에 유리하다.
- **입력값 유지 전략**: 직원 폼은 이미 `adminStaffService.getCreateForm(request)`가 있어 사용자가 적었던 값을 다시 모델에 실어 보낼 수 있다. 이번 단계는 이 구조를 재사용하면서 검증 UX만 강화한 작업이라고 볼 수 있다.
