## 2. 핵심 전달 코드 (Core Logic)

```java
@PostMapping("/create")
public String create(
        @Valid @ModelAttribute CreateAdminStaffRequest request,
        BindingResult bindingResult,
        RedirectAttributes redirectAttributes,
        HttpServletRequest req) {
    // 직원 등록은 이미 @Valid + BindingResult 골격이 있다.
    if (bindingResult.hasErrors()) {
        req.setAttribute("errorMessage", adminStaffService.getInputCheckMessage());
        req.setAttribute("model", adminStaffService.getCreateForm(request));
        return "admin/staff-form";
    }
    ...
}
```

```java
@PostMapping("/create")
public RedirectView create(
        @RequestParam String name,
        @RequestParam(defaultValue = "false") boolean active) {
    // 진료과 등록은 아직 @RequestParam 기반이라 입력 DTO가 없다.
    adminDepartmentService.createDepartment(name, active);
    return redirectTo("/admin/department/list");
}
```

핵심은 두 모듈의 현재 출발점이 다르다는 점이다. 직원 쪽은 검증 훅은 이미 있고 규칙을 보강하면 되는 상태이고, 진료과 쪽은 검증 객체와 same-view 재렌더링 모델부터 설계해야 한다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **`@Valid + BindingResult`의 의미**: SSR 폼 검증에서 가장 중요한 건 실패했을 때 사용자를 같은 화면으로 돌려보내며 에러와 입력값을 함께 보여주는 것이다. 직원 컨트롤러는 이 골격이 이미 있어서 검증 어노테이션을 보강하면 흐름을 재사용할 수 있다.
- **same-view 재렌더링과 입력값 유지**: redirect를 써버리면 사용자가 방금 입력한 값이 사라지기 쉽다. 그래서 이번 구조 점검에서는 어떤 엔드포인트가 redirect가 아니라 same-view 재렌더링으로 가야 하는지 먼저 구분했다.
- **입력 DTO의 존재 여부**: 진료과는 아직 `@RequestParam` 기반이라 필드 단위 Bean Validation을 일관되게 쓰기 어렵다. 이 경우 등록용/수정용 입력 객체를 새로 두면 컨트롤러 시그니처, 에러 처리, 템플릿 바인딩이 함께 정리된다.
- **서비스 재주입 헬퍼의 가치**: 직원 쪽의 `adminStaffService.getCreateForm(request)`와 `getEditForm(request)`는 검증 실패 시 입력값을 폼 모델로 다시 만들어주는 역할을 한다. 이 구조가 이미 있다는 건 직원 폼 검증 UX를 빠르게 정리할 수 있다는 뜻이다.
