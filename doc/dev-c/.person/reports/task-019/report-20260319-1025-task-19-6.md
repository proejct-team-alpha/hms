## 2. 핵심 전달 코드 (Core Logic)

```java
public final class SsrValidationViewSupport {

    public static final String INPUT_CHECK_MESSAGE = "입력값을 확인해 주세요.";

    public static void applyErrors(HttpServletRequest req, BindingResult bindingResult) {
        req.setAttribute("errorMessage", INPUT_CHECK_MESSAGE);

        for (FieldError fieldError : bindingResult.getFieldErrors()) {
            req.setAttribute(fieldError.getField() + "Error", fieldError.getDefaultMessage());
        }
    }
}
```

```java
if (bindingResult.hasErrors()) {
    SsrValidationViewSupport.applyErrors(req, bindingResult);
    req.setAttribute("model", adminStaffService.getCreateForm(request));
    return "admin/staff-form";
}
```

```java
if (bindingResult.hasErrors()) {
    return renderCreateValidationFailure(req, request, bindingResult);
}
```

```java
private String renderCreateValidationFailure(
        HttpServletRequest req,
        CreateAdminDepartmentRequest request,
        BindingResult bindingResult) {
    req.setAttribute("model", adminDepartmentService.getDepartmentList(1, 10));
    req.setAttribute("pageTitle", "진료과 관리");
    req.setAttribute("createName", request.getName());
    req.setAttribute("createActive", request.isActive());
    req.setAttribute("openCreateModal", true);
    SsrValidationViewSupport.applyErrors(req, bindingResult);
    return "admin/department-list";
}
```

이 코드는 “검증 실패 시 어떤 문구를 보여주고 어떤 필드 에러를 주입할지”를 컨트롤러마다 따로 흩뿌리지 않고, 같은 패턴으로 맞춘 핵심이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 검증 UX의 공통화**: API는 보통 JSON 에러 포맷으로 통일하지만, SSR은 화면마다 request attribute 이름이 흩어지기 쉽다. 이번엔 `errorMessage`와 `fieldName + Error` 규칙을 공통 유틸로 고정해서 템플릿 계약을 단순하게 만들었다.
- **DTO 검증과 서비스 호출 분리**: `@NotBlank`, `@NotNull` 같은 DTO 검증은 컨트롤러 진입 직후 걸러지는 단계다. 이 단계에서 서비스 메서드까지 타면 중복 예외나 불필요한 조회가 생긴다. 그래서 테스트에서 `should(never())`로 서비스 호출 차단을 확인한 것이 중요하다.
- **공백과 빈 문자열 회귀 방지**: 실무에서는 `""`만 막고 `"   "`를 놓치는 경우가 꽤 많다. 이번엔 직원/진료과 양쪽에서 빈 문자열과 공백 입력 시나리오를 나눠서 테스트해, Bean Validation이 기대대로 동작하는지 명시적으로 확인했다.
- **폼별 same-view 전략 유지**: 직원 폼은 `staff-form.mustache`, 진료과 등록은 목록 same-view + 모달 재오픈, 진료과 수정은 상세 same-view라는 기존 UX를 유지한 채 검증 패턴만 통일했다. 공통화가 UI 흐름을 깨지 않게 한 점이 이번 단계의 핵심이다.
