## 4. 기술 딥다이브 (Technical Deep-dive)

- **서비스 테스트의 가치**: 컨트롤러 테스트는 요청/응답 흐름을 보장하지만, 실제 비즈니스 규칙이 서비스 레이어에 있다면 핵심 회귀는 서비스 테스트가 막아준다. 이번 점검도 “직원 생성/수정/비활성화, 진료과 수정/상태 전이”가 서비스에서 얼마나 보장되는지 보는 데 초점을 맞췄다.
- **커버리지보다 경계 시나리오**: 이번 점검에서 중요한 건 테스트 개수보다 어떤 경계가 비어 있는지였다. 직원 쪽은 `employeeNumber` 중복 방지, 진료과 쪽은 생성 중복 방지와 활성화 시 없는 ID 차단처럼, 운영에서 바로 사고로 이어질 수 있는 경계가 빠져 있었다.
- **읽기 쉬운 배치의 중요성**: 테스트가 많이 있어도 생성/수정/상태 전이가 섞여 있으면 유지보수성이 떨어진다. 그래서 `Task 20-6`에 섹션 재배치와 이름 정리를 별도 작업으로 둔 것이 의미가 있다. 이번 단계는 기능 보강 전에 테스트 파일 자체의 독해성을 점검한 단계라고 볼 수 있다.
