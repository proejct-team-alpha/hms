## 2. 핵심 코드 (Core Logic)

```java
@Test
@DisplayName("중복 사번으로 직원 등록 시 예외가 발생한다")
void createStaff_duplicateEmployeeNumber_throwsException() {
    // given
    Department department = persistDepartment("정형외과");
    persistStaff("existing-user", "S-001", "기존직원", StaffRole.STAFF, department, true);
    entityManager.flush();
    entityManager.clear();

    CreateAdminStaffRequest request = new CreateAdminStaffRequest(
            "new-user",
            "password123",
            "신규직원",
            "S-001",
            "STAFF",
            department.getId(),
            true,
            null,
            List.of()
    );

    // when
    // then
    assertThatThrownBy(() -> adminStaffService.createStaff(request))
            .isInstanceOf(CustomException.class)
            .hasMessage("이미 사용 중인 사번입니다.");
}
```

이 테스트는 직원 생성 시 `username`뿐 아니라 `employeeNumber`도 고유해야 한다는 서비스 규칙을 고정한다. 이후 서비스 구현이 바뀌더라도 사번 중복 차단이 빠지면 바로 테스트가 깨지도록 만드는 역할을 한다.

## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 직원 등록은 새 학생을 반 명단에 추가하는 것과 비슷하다.
- 로그인 아이디는 학교 계정 번호 같은 것이고, 사번은 출석번호 같은 것이다.
- 계정 번호만 겹치면 안 되는 게 아니라 출석번호도 겹치면 안 된다.
- 이번 작업은 “출석번호가 같은 학생을 또 넣으려고 하면 선생님이 바로 막아준다”는 규칙을 시험지로 만들어 둔 것과 같다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 서비스 수정이 아니라 테스트 추가만 했는가**: [AdminStaffService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffService.java)에는 이미 `validateDuplicateEmployeeNumber(...)`가 존재하고, `createStaff(...)` 내부에서 호출되고 있었다. 즉 구현 누락이 아니라 테스트 공백이 문제였다.
- **왜 이 시나리오가 중요한가**: `username` 중복은 흔히 먼저 떠올리지만, 실제 운영에서는 사번도 외부 시스템 연계나 내부 식별 기준으로 자주 사용된다. 사번 중복이 허용되면 직원 검색/동기화/감사 로그에서 혼선이 생길 수 있다.
- **인코딩 이슈를 함께 정리한 이유**: 테스트를 추가하는 순간 파일 자체가 컴파일되지 않으면, 새 테스트가 맞는지 틀린지조차 확인할 수 없다. 그래서 이번 단계는 “새 테스트 추가”와 “테스트 파일 안정화”를 함께 처리하는 것이 더 안전했다.
- **범위를 왜 좁게 유지했는가**: Task 20-2는 생성/중복 방지/암호화 범위인데, 이미 암호화와 `DOCTOR` 생성 시나리오는 갖춰져 있었다. 그래서 이번에는 정말 비어 있는 시나리오 하나만 채워서 다음 단계(Task 20-3)로 자연스럽게 넘어갈 수 있게 했다.
