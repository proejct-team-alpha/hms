## 2. 핵심 코드 (Core Logic)

```java
@Test
@DisplayName("update staff keeps existing password when password input is blank")
void updateStaff_blankPassword_keepsExistingPassword() {
    // given
    Department originalDepartment = persistDepartment("general-admin");
    Department changedDepartment = persistDepartment("ops-admin");
    Staff staff = persistStaff("staff-keep-password", "S-011", "existing-staff", StaffRole.STAFF, originalDepartment, true);
    String originalPassword = staff.getPassword();
    entityManager.flush();
    entityManager.clear();

    UpdateAdminStaffRequest request = new UpdateAdminStaffRequest(
            staff.getId(),
            "updated-staff",
            changedDepartment.getId(),
            "",
            null,
            List.of()
    );

    // when
    adminStaffService.updateStaff(request);
    entityManager.flush();
    entityManager.clear();

    Staff updatedStaff = entityManager.find(Staff.class, staff.getId());

    // then
    assertThat(updatedStaff.getName()).isEqualTo("updated-staff");
    assertThat(updatedStaff.getDepartment().getId()).isEqualTo(changedDepartment.getId());
    assertThat(updatedStaff.getPassword()).isEqualTo(originalPassword);
}
```

이 테스트는 직원 수정 시 비밀번호를 공란으로 두었을 때 기존 암호화된 비밀번호가 덮어써지지 않고 유지되어야 한다는 규칙을 고정한다.

## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 직원 정보를 고치는 건 학생 생활기록부에서 이름이나 반을 수정하는 것과 비슷하다.
- 그런데 비밀번호는 새 비밀 번호를 적었을 때만 바뀌어야 한다.
- 아무것도 안 적었는데 비밀번호가 바뀌거나 없어지면 안 된다.
- 이번 테스트는 “빈칸으로 제출하면 원래 비밀번호는 그대로 남는다”는 안전장치를 확인하는 검사다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 이 시나리오가 중요한가**: 수정 폼에서 비밀번호를 선택 입력으로 두는 경우, 가장 흔한 회귀는 “빈 문자열도 새 비밀번호처럼 처리해버리는 버그”다. 이 경우 사용자 계정이 의도치 않게 잠기거나 로그인 실패로 이어질 수 있다.
- **서비스 구현과 테스트의 관계**: [AdminStaffService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffService.java)의 `updateStaff(...)`는 `hasText(...)` 조건을 통해 비밀번호 입력이 있을 때만 `updatePassword(...)`를 호출한다. 이번 테스트는 이 조건 분기가 실제로 기대대로 동작하는지 검증하는 역할을 한다.
- **왜 기존 수정 테스트와 분리했는가**: 이미 있는 `updateStaff_updatesNameDepartmentAndPassword()`는 “비밀번호가 있을 때 바뀐다”를 보장한다. 이번 테스트는 그 반대편 경계인 “비밀번호가 없을 때는 유지된다”를 보장한다. 두 테스트를 나누면 의도가 명확하고, 실패 원인도 더 바로 드러난다.
- **인코딩 안정화 메모**: 이번 단계에서도 테스트 파일 한글이 흔들릴 수 있는 상황이 있었기 때문에, 파일 전체를 무리하게 편집하지 않고 필요한 블록만 추가하는 방식으로 리스크를 줄였다.
