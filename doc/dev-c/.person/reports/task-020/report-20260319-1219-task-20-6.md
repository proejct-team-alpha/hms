void updateStaff_blankPassword_keepsExistingPassword() {

    Staff staff = persistStaff("staff-keep-password", "S-011", "existing-staff", StaffRole.STAFF, originalDepartment, true);

            staff.getId(),
            "updated-staff",
            changedDepartment.getId(),
            "",
            null,

    );

    // when
    adminStaffService.updateStaff(request);

    // then
    Staff updatedStaff = entityManager.find(Staff.class, staff.getId());
    assertThat(updatedStaff.getPassword()).isEqualTo(originalPassword);
}

## 4. 기술 딥다이브 (Technical Deep-dive)

- **Gradle 테스트 실행 충돌**: 이번 단계에서 범위 테스트와 전체 테스트를 동시에 돌렸을 때 uild/test-results/test/binary/in-progress-results-*.bin 파일 충돌이 발생했다. 이건 코드 실패라기보다 Gradle 테스트 결과 파일을 같은 시점에 두 프로세스가 건드리며 생긴 문제였다. 그래서 순차 실행으로 다시 검증했고, 동일 코드 상태에서 전체 테스트까지 정상 통과함을 확인했다.
- **서비스 테스트 마무리 기준**: 서비스 단위 테스트를 보강할 때는 단순히 테스트 개수를 늘리는 것보다, 생성/수정/중복 방지/암호화/상태 전이 같은 핵심 규칙이 모두 닫혔는지가 중요하다. 이번 단계에서는 [AdminStaffServiceTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/staff/AdminStaffServiceTest.java)와 [AdminDepartmentServiceTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/department/AdminDepartmentServiceTest.java)가 그 기준을 만족하는지 최종 확인했다.
- **안전한 마무리 방식**: 인코딩 이슈가 있었던 테스트 파일은 과하게 재배열하면 리스크가 커질 수 있다. 그래서 이번 Task 20-6에서는 테스트 블록 사이 간격 같은 최소 정리만 하고, 검증은 실제 Gradle 실행 결과로 닫았다. 이런 방식이 지금 프로젝트 상태에서는 가장 안전한 마무리였다.
