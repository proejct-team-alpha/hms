## 2. 핵심 코드 (Core Logic)

```java
@Test
@DisplayName("activateDepartment rejects missing department")
void activateDepartment_throwsWhenDepartmentMissing() {
    // given
    given(adminDepartmentRepository.findById(91L)).willReturn(Optional.empty());

    // when
    // then
    assertThatThrownBy(() -> adminDepartmentService.activateDepartment(91L))
            .isInstanceOf(CustomException.class)
            .hasMessage("진료과를 찾을 수 없습니다.");
}
```

이 테스트는 활성화 상태 전이도 결국 “존재하는 진료과만” 대상으로 해야 한다는 규칙을 보장한다. 비활성화 쪽에서만 없는 ID를 막고 활성화 쪽에서는 놓치는 비대칭을 막는 역할을 한다.

## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 진료과를 켜고 끄는 건 교실 불을 켜고 끄는 것과 비슷하다.
- 그런데 실제로 없는 교실의 불을 켜라고 하면 당연히 할 수 없다.
- 이번 테스트는 “없는 교실 번호를 적으면 선생님이 바로 잘못됐다고 알려준다”는 규칙을 확인하는 검사다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 이 테스트가 필요했는가**: 상태 전이 로직은 성공 케이스보다 예외 케이스에서 더 자주 회귀가 생긴다. 특히 한쪽 전이(`deactivate`)에는 없는 ID 차단 테스트가 있는데 반대쪽 전이(`activate`)에는 없으면, 구현이 비대칭이더라도 테스트가 놓칠 수 있다.
- **상태 전이 테스트의 구조**: 좋은 상태 전이 테스트 세트는 보통 세 가지를 한 묶음으로 본다.
  1. 정상 전이 성공
  2. 이미 같은 상태인 중복 요청 차단
  3. 없는 대상 차단
  이번 단계로 진료과 상태 전이는 이 세 축이 모두 갖춰졌다.
- **왜 서비스 수정은 필요 없었는가**: [AdminDepartmentService.java](c:/workspace/Team/hms/src/main/java/com/smartclinic/hms/admin/department/AdminDepartmentService.java)의 `activateDepartment()`는 이미 `findDepartment()`를 통해 없는 ID를 `CustomException.notFound(...)`로 막고 있었다. 따라서 이번 작업은 구현 누락이 아니라 테스트 공백을 메우는 작업이었다.
- **문서와 테스트의 연결**: [task-020.md](c:/workspace/Team/hms/doc/dev-c/task/task-020.md)에서 Task 20-5를 완료 처리함으로써, 이제 진료과 서비스 테스트 쪽 남은 일은 사실상 `Task 20-6`의 구조 정리와 범위 검증뿐이라는 점이 분명해졌다.
