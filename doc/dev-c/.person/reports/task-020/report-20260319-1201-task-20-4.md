## 2. 핵심 코드 (Core Logic)

```java
@Transactional
public void createDepartment(String name, boolean active) {
    String normalizedName = normalizeName(name);
    validateDepartmentName(normalizedName);

    if (adminDepartmentRepository.existsByNameIgnoreCase(normalizedName)) {
        throw CustomException.conflict("DUPLICATE_DEPARTMENT_NAME", DUPLICATE_DEPARTMENT_NAME_MESSAGE);
    }

    adminDepartmentRepository.save(Department.create(normalizedName, active));
}
```

```java
@Test
@DisplayName("createDepartment rejects duplicate name")
void createDepartment_throwsWhenNameDuplicated() {
    // given
    given(adminDepartmentRepository.existsByNameIgnoreCase("내과")).willReturn(true);

    // when
    // then
    assertThatThrownBy(() -> adminDepartmentService.createDepartment("내과", true))
            .isInstanceOf(CustomException.class)
            .hasMessage("이미 존재하는 진료과명입니다.");
}
```

이 조합으로 “진료과 생성 시 이름 중복은 허용하지 않는다”는 규칙이 서비스와 테스트 양쪽에 같이 고정됐다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 서비스도 같이 바꿨는가**: 기존 Task 20-4는 테스트 보강이 목적이었지만, 점검해보니 생성 로직 자체에 중복 이름 차단이 없었다. 이 경우 테스트만 추가하면 당연히 실패하고, 그 실패는 단순 회귀가 아니라 실제 비즈니스 규칙 누락을 보여주는 신호였다. 그래서 이 단계는 테스트 보강과 서비스 보강이 함께 가는 게 맞았다.
- **왜 `existsByNameIgnoreCaseAndIdNot(...)`가 아니라 `existsByNameIgnoreCase(...)`를 추가했는가**: 수정(update)과 생성(create)은 중복 판단 기준이 다르다. 생성은 자기 자신을 제외할 필요가 없으므로 더 단순한 메서드가 맞다. 이 분리를 해두면 서비스 의도가 코드에서 바로 읽힌다.
- **왜 `normalizeName()`과 `validateDepartmentName()`를 생성에서도 재사용했는가**: 수정 로직에서 이미 쓰고 있는 이름 정규화/검증 규칙을 생성에도 동일하게 적용하면, 생성과 수정의 정책이 어긋나지 않는다. 같은 입력 규칙은 같은 코드 경로를 쓰는 편이 유지보수에 훨씬 안전하다.
- **왜 이번 단계는 여기서 멈췄는가**: Task 20-4 범위 안에서도 실제로 비어 있던 것은 “생성 중복 방지”가 핵심이었다. 수정 시나리오는 이미 충분히 갖춰져 있었기 때문에, 괜히 테스트를 더 늘리기보다 빈 구멍만 메우는 쪽이 더 깔끔했다.
