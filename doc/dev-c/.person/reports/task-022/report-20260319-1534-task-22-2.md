## 2. 핵심 코드/구조 (Core Logic)

```java
public record AdminPatientListResponse(
        List<AdminPatientSummary> patients,
        List<AdminPatientPageLinkResponse> pageLinks,
        String nameKeyword,
        String contactKeyword,
        long totalCount,
        int currentPage,
        int size,
        int totalPages,
        boolean hasPages,
        boolean hasPrevious,
        boolean hasNext,
        String previousUrl,
        String nextUrl

}
```

```java
@Query("""
        SELECT p
        FROM Patient p
        WHERE (:nameKeyword = '' OR LOWER(p.name) LIKE LOWER(CONCAT('%', :nameKeyword, '%')))
          AND (:contactKeyword = '' OR REPLACE(p.phone, '-', '') LIKE CONCAT('%', :contactKeyword, '%'))
        ORDER BY p.createdAt DESC, p.id DESC
        """)
Page<Patient> search(...)
```

핵심은 환자 목록을 “예약을 가볍게 보여주는 화면”으로 만들지 않고, 환자 기준 검색/페이징 화면으로 고정하는 것이다. 그래서 목록 응답에는 환자명, 연락처, 등록일, 상세 URL만 들어가고, 상세 화면에서만 예약 이력이 풀리도록 구조를 미리 나눴다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 스켈레톤을 먼저 만들었나**: `admin.patient`는 실제 자바 파일이 없던 상태라, 곧바로 구현에 들어가면 컨트롤러/서비스/리포지토리/DTO 책임이 뒤섞이기 쉽다. 그래서 먼저 컴파일 가능한 최소 뼈대를 만들고, 다음 단계에서 기능만 채워 넣는 방식이 더 안전했다.
- **왜 목록 응답에 pageLinks까지 먼저 넣었나**: 기존 `admin.department`와 `admin.staff`는 페이지네이션 UI가 서비스 응답 구조에 강하게 묶여 있다. 환자 목록도 같은 패턴을 따르면 템플릿 구현 시 고민이 줄어든다.
- **연락처 검색 정규화**: 현재 환자 재사용 기준이 전화번호 문자열이므로, 검색에서도 하이픈 포함/미포함 입력을 흡수해야 사용자 경험이 좋다. 그래서 리포지토리 쿼리에서 `REPLACE(p.phone, '-', '')`를 사용해 방향을 먼저 고정했다.
- **기본 정렬을 최신 등록순으로 둔 이유**: 환자 목록은 “찾기 위한 인덱스”지만, 검색 없이 들어왔을 때도 최신 환자가 먼저 보이는 편이 관리자 입장에서 더 자연스럽다. 그래서 `createdAt desc, id desc`를 기본 정렬 기준으로 잡았다.
- **BOM 인코딩 이슈**: PowerShell의 기본 저장 방식으로 새 자바 파일을 만들면 BOM이 붙어 Java 컴파일 오류가 날 수 있다. 이번에는 UTF-8 without BOM으로 다시 저장해 문제를 해소했다. 새 파일을 많이 만드는 작업에서는 이 점을 계속 조심해야 한다.
