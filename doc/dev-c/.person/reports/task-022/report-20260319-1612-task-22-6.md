```java
@Test
@DisplayName("get patient list returns empty result when no patient matches")
void getPatientList_returnsEmptyResultWhenNoPatientMatches() {
    // when
    var result = adminPatientService.getPatientList(1, 20, "박", "0109999");

    // then
    assertThat(result.totalCount()).isZero();
    assertThat(result.patients()).isEmpty();
    assertThat(result.hasPages()).isFalse();
```

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 빈 결과 시나리오를 꼭 추가했나**: 목록/검색 화면은 결과가 있을 때보다 없을 때 더 쉽게 어색해진다. `patients=[]`일 때 템플릿이 어떻게 보일지, 페이지네이션 정보가 어떻게 내려갈지를 테스트로 닫아두면 이후 템플릿을 다시 손볼 때 회귀 가능성이 크게 줄어든다.
- **왜 문서 갱신을 테스트와 같은 단계로 봤나**: 이번 흐름은 workflow/task 기반으로 세분화되어 있어서, 코드가 끝났더라도 문서가 옛 상태로 남아 있으면 실제 진행 상태를 오해하기 쉽다. 특히 리뷰 포인트는 PR에서 바로 옮겨 적을 수 있는 정보라서 마지막 단계에서 같이 정리하는 편이 효율적이다.
- **전체 테스트 재확인 이유**: `admin.patient` 범위만 통과해도 기능 개발은 된 셈이지만, PR 직전 기준으로는 전체 `./gradlew test`가 초록인지 확인하는 것이 훨씬 안전하다. 이번에도 실제 코드 문제는 없었고, 병렬 실행에 따른 Gradle 결과 파일 충돌만 한 번 확인되었다.
- **인코딩과 저장 방식**: 한글 템플릿과 테스트를 자주 다루다 보니 BOM이 끼어들면 컴파일이 바로 깨진다. 이번 작업에서는 최종적으로 UTF-8 without BOM 기준으로 정리해 이후 마찰을 줄였다.
