## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 환자 목록은 도서관의 카드 목록이다. 필요한 사람을 빠르게 찾는 것이 핵심이고, 모든 내용을 펼쳐 보여줄 필요는 없다.
- 환자 상세는 책 본문이다. 본문에서는 연락처, 메모, 예약 이력처럼 실제로 읽어야 하는 정보를 차분히 보여줘야 한다.
- 이번 Task 22-5는 카드 목록과 책 본문이 서로 헷갈리지 않게 표지와 섹션 제목을 더 잘 붙여준 작업이라고 보면 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 목록에 `환자 목록` 섹션 제목을 넣었나**: 페이지 H1이 `환자 관리`인 상태에서, 바로 검색 폼과 테이블만 보이면 인덱스 화면의 목적이 약하게 느껴질 수 있다. 섹션 제목을 추가하면 사이드바 메뉴 구조와도 더 잘 맞고, 화면을 처음 보는 사람도 지금 보고 있는 것이 목록이라는 점을 쉽게 이해할 수 있다.
- **왜 목록에 상세 정보 미리보기를 넣지 않았나**: 문서와 인터뷰에서 계속 합의한 방향은 “목록은 찾기, 상세는 보기”였다. 여기서 예약 이력 미리보기나 통계 위젯을 넣으면 목록 화면이 다시 무거워지고 책임이 흐려진다. 이번 단계에서는 오히려 덜 넣는 쪽이 더 중요했다.
- **왜 상세 화면에 보조 안내 카드를 추가했나**: 상세는 정보량이 많아질수록 좌우 구성이 딱딱하게 보일 수 있다. 짧은 안내 카드 하나를 두면 화면이 단순 표/카드 묶음이 아니라, 현재 페이지가 어떤 역할을 하는지 맥락을 더 잘 전달할 수 있다.
- **왜 테스트를 문구 기준으로 조금 더 보강했나**: Mustache 화면 리팩토링은 로직보다 문구와 구조 변화가 중심이라, 단순 view 이름 검증만으로는 회귀를 충분히 막기 어렵다. 이번에는 실제 화면 핵심 문구가 유지되는지까지 확인해서, 다음에 템플릿을 건드릴 때도 역할 분리가 깨지지 않도록 했다.
- **인코딩 정리 이유**: 이전 단계에서 환자 템플릿과 테스트 일부에 한글 깨짐이 다시 나타났기 때문에, 이번에는 템플릿과 테스트 파일을 UTF-8 without BOM 기준으로 다시 저장해 이후 작업 마찰을 줄였다.
