핵심은 환자 목록 화면이 “예약을 보여주기 위한 화면”이 아니라, 이름/연락처로 환자를 빨리 찾는 인덱스 화면이 되도록 구조를 고정한 것이다. 그래서 목록은 환자 1명당 1줄만 보여주고, 검색/페이징/정렬을 안정적으로 제공하는 데 집중했다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 환자 목록은 “병원 접수대의 환자 찾기 명단”에 가깝다. 여기서는 누구를 찾고 싶은지 빨리 찾는 것이 목적이라 이름, 연락처, 등록일 정도면 충분하다.
- 환자 상세는 “환자 차트”에 가깝다. 차트에는 예약 이력, 메모, 연락처 같은 많은 정보가 들어가지만, 접수대 명단에 그걸 다 펼쳐놓으면 오히려 더 복잡해진다.
- 이번 Task 22-3은 접수대 명단을 먼저 제대로 만든 단계라고 보면 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 검색 파라미터를 두 축으로 나눴나**: 이름과 연락처는 사용자 검색 행동이 가장 분명한 두 축이다. 검색창 하나에 섞기보다 `nameKeyword`, `contactKeyword`로 나누면 서비스와 테스트가 더 명확해진다.
- **왜 연락처를 정규화했나**: 현재 시스템은 전화번호로 같은 환자를 재사용한다. 사용자는 `01012345678`로도 입력하고 `010-1234-5678`로도 입력할 수 있기 때문에, 목록 검색도 같은 수준의 정규화가 없으면 체감 검색 품질이 떨어진다.
- **왜 기본 페이지 크기를 20으로 둔 이유**: 문서 기준과 맞추는 동시에, 환자 목록이 “많이 보여주는 메인”이 아니라 “찾는 인덱스 화면”이라는 성격에도 잘 맞는다. 너무 적으면 자주 넘겨야 하고, 너무 많으면 메인이 무거워진다.
- **왜 Mustache는 기본형만 넣었나**: 지금 단계는 UX 미세조정보다 “화면이 실제로 열리고 검색/목록/페이징이 연결된다”가 더 중요했다. 상세 정보와 예약 이력은 다음 Task의 책임으로 의도적으로 남겨두었다.
- **인코딩/테스트 패키지 이슈**: 새 파일을 만들 때 BOM이 들어가거나 테스트 import 패키지를 Spring Boot 4 현재 프로젝트 패턴과 다르게 잡으면 바로 컴파일이 깨진다. 이번 단계에서는 그 두 가지를 바로 잡아서 이후 작업의 마찰을 줄였다.
