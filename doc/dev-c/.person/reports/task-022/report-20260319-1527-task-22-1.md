## 2. 핵심 코드/구조 (Core Logic)

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "patient_id", nullable = false)
private Patient patient;
```

```java
public interface PatientRepository extends JpaRepository<Patient, Long> {
    Optional<Patient> findByPhone(String phone);
}
```

핵심은 현재 시스템이 “예약을 만들 때 환자를 새로 설계해서 연결하는 구조”가 아니라, 먼저 전화번호로 기존 환자를 찾고 그 `Patient`를 예약에 연결하는 구조라는 점이다. 그래서 같은 환자가 월요일/수요일 예약을 각각 만들면, 환자 레코드는 1개이고 예약 레코드만 2개가 된다. 이 구조 때문에 환자 목록을 예약 목록의 변형으로 구현하면 환자별 1줄 인덱스 화면이라는 목표와 어긋나게 된다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 예약 목록은 “택배 송장 목록”에 가깝다. 같은 사람이 택배를 두 번 보내면 송장은 두 장이다.
- 환자 목록은 “사람 주소록”에 가깝다. 같은 사람이 월요일/수요일 예약을 두 번 해도 주소록에는 한 사람으로 한 번만 있어야 한다.
- 이번 Task 22-1은 주소록 화면을 만들기 전에, 현재 시스템이 사람을 어떤 기준으로 구분하는지(전화번호), 송장과 주소록이 어떻게 연결되어 있는지(`patient_id`)를 먼저 확인한 단계라고 보면 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **환자 동일성 기준**: 현재 구현에서는 전화번호 문자열이 동일 환자 판별 기준이다. `findByPhone(...)`만 제공하므로, 하이픈 포함/미포함 차이는 검색과 동일성 판별에서 곧바로 리스크가 된다. 따라서 목록 검색에서는 반드시 정규화 전략을 고민해야 한다.
- **왜 `admin.patient` 전용 Repository가 필요한가**: 현재 `PatientRepository`는 생성/재사용을 위한 최소 기능만 있고, 이름/연락처 검색 + 20건 페이징 + 환자 기준 1줄 목록을 그대로 지원하지 않는다. 따라서 `admin.patient` 화면 전용 조회는 `admin.patient` 소유 리포지토리에서 책임을 분리하는 편이 맞다.
- **왜 예약 목록을 그대로 못 쓰는가**: 예약은 `Reservation` 중심 목록이라 같은 환자가 여러 건을 가지면 중복 row가 생긴다. 문서 기준 환자 목록은 “찾기 위한 인덱스”라서 환자 1명당 1줄이어야 한다. 결국 환자 목록은 예약 기반이 아니라 환자 기반으로 다시 설계해야 한다.
- **없는 patientId 처리**: 관리자 SSR 패턴상 없는 상세 대상은 404 또는 프로젝트 공통 에러 화면으로 처리하는 게 자연스럽다. 이 기준을 미리 정해두면 `Task 22-4`에서 상세 화면을 구현할 때 흐름이 깔끔해진다.
