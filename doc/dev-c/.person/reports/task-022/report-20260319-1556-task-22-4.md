## 2. 핵심 코드/구조 (Core Logic)

```java
@GetMapping("/detail")
public String detail(
        @RequestParam("patientId") Long patientId,
        HttpServletRequest req,
        HttpServletResponse response) {
    try {
        AdminPatientDetailResponse result = adminPatientService.getPatientDetail(patientId);
        return renderDetailPage(req, result);
    } catch (CustomException ex) {
        response.setStatus(HttpServletResponse.SC_NOT_FOUND);
        req.setAttribute("pageTitle", "페이지를 찾을 수 없습니다");
        req.setAttribute("errorMessage", ex.getMessage());
        req.setAttribute("path", req.getRequestURI());
        return "error/404";
    }
}
```

```java
public AdminPatientDetailResponse getPatientDetail(Long patientId) {
    Patient patient = adminPatientRepository.findById(patientId)
            .orElseThrow(() -> CustomException.notFound(PATIENT_NOT_FOUND_MESSAGE));

    List<AdminPatientReservationHistoryItemResponse> reservationHistories = adminPatientRepository
            .findReservationHistoriesByPatientId(patientId)
            .stream()
            .map(history -> new AdminPatientReservationHistoryItemResponse(
                    history.getReservationNumber(),
                    history.getReservationDate().toString(),
                    history.getTimeSlot(),
                    history.getDepartmentName(),
                    history.getDoctorName(),
                    STATUS_LABELS.getOrDefault(history.getStatus(), history.getStatus().name())
            ))
            .toList();

    return new AdminPatientDetailResponse(
            patient.getId(),
            patient.getName(),
            patient.getPhone(),
            defaultText(patient.getEmail()),
            defaultText(patient.getAddress()),
            defaultText(patient.getNote()),
            reservationHistories
    );
}
```

```java
@Query("""
        SELECT r.reservationNumber AS reservationNumber,
               r.reservationDate AS reservationDate,
               r.timeSlot AS timeSlot,
               department.name AS departmentName,
               staff.name AS doctorName,
               r.status AS status
        FROM Reservation r
        JOIN r.department department
        JOIN r.doctor doctor
        JOIN doctor.staff staff
        WHERE r.patient.id = :patientId
        ORDER BY r.reservationDate DESC, r.timeSlot DESC, r.id DESC
        """)
List<AdminPatientReservationHistoryProjection> findReservationHistoriesByPatientId(Long patientId);
```

핵심은 환자 상세 화면이 “환자 기본 정보 + 해당 환자의 전체 예약 이력”이라는 책임에만 집중하도록 만들고, 목록 화면에 있던 가벼운 인덱스 역할과 확실히 분리한 것이다.

## 3. 쉽게 이해하는 비유 (Easy Analogy)

- 환자 목록은 접수대에서 환자를 찾는 인덱스다. 이름과 연락처만 보고 빠르게 사람을 찾는 데 목적이 있다.
- 환자 상세는 환자 차트다. 차트에서는 기본 정보와 지금까지의 예약 이력을 차분히 읽어야 하므로, 목록 화면에서 보지 않던 정보까지 모아서 보여준다.
- 이번 Task 22-4는 “찾는 화면” 다음에 “읽는 화면”을 붙인 단계라고 보면 된다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 상세 응답에서 기본값 보정을 했나**: 이메일, 주소, 메모는 실제 데이터가 비어 있을 수 있다. 이 값을 그대로 템플릿에 넘기면 빈 공간처럼 보여 화면이 어색해질 수 있어서, 서비스에서 `-`로 보정해 템플릿 책임을 줄였다.
- **왜 예약 이력 조회를 admin.patient 리포지토리에 둔 이유**: 이력 데이터는 `Reservation`에서 오지만, 지금 화면 책임은 분명히 `admin.patient`다. 상세 화면을 위한 읽기 모델을 `admin.patient` 범위에서 가져가는 편이 컨트롤러/서비스 책임을 이해하기 쉽다.
- **왜 상태 라벨을 서비스에서 매핑했나**: 템플릿에서 enum 이름을 직접 해석하게 두면 화면이 도메인 세부 구현에 묶이게 된다. 서비스에서 `예약 / 접수 / 진료 중 / 완료 / 취소`로 확정해 주면 템플릿은 표시만 하면 된다.
- **왜 404 처리를 컨트롤러에서 바로 맞췄나**: 기존 관리자 상세 화면 패턴과 동일하게 맞추는 것이 중요했다. 상세 요청이 실패할 때 목록으로 되돌리기보다, 없는 대상을 요청했다는 사실을 명확한 404 상태와 화면으로 보여주는 편이 더 자연스럽다.
- **왜 patient-list.mustache까지 같이 정리했나**: 상세 구현과 직접 관련은 없지만, 목록 템플릿의 인코딩이 흔들린 상태로 두면 `admin.patient` 범위 렌더링 테스트가 계속 불안해질 수 있다. 이번 기회에 UTF-8 기준으로 같이 정리해 두는 것이 이후 Task 22-5 진행에도 도움이 된다.
