## 2. 핵심 전달 코드 (Core Logic)

```java
// staff-form.mustache
<input
  type="datetime-local"
  id="staff-retired-at"
  name="retiredAt"
  value="{{model.retiredAt}}"
  class="w-full ..."
  {{#model.retiredAtLocked}}disabled{{/model.retiredAtLocked}}
/>
```

현재 `퇴사 일시`는 `datetime-local` 한 칸 입력이다. 즉 화면에서 분까지 입력하게 되어 있고, 이번에 원하는 `날짜 + 시` 형태와는 아직 다르다.

```java
// UpdateAdminStaffRequest
@DateTimeFormat(pattern = "yyyy-MM-dd'T'HH:mm")
LocalDateTime retiredAt
```

SSR 수정 DTO도 이미 `HH:mm` 패턴을 기대한다. 등록 DTO와 API DTO도 같은 구조라서, 입력 형식 단순화는 화면만 바꾸는 게 아니라 DTO 계약도 같이 손봐야 한다.

```java
// AdminStaffService.updateStaff(...)
validateReactivation(staff, request.active());

LocalDateTime retiredAt = request.retiredAt();
boolean active = resolveActiveState(request.active(), retiredAt);

staff.update(request.name().trim(), null, active);
staff.updateRetiredAt(retiredAt);
```

중요한 점은 여기서 비활성 직원의 재활성화만 막고 있다는 점이다. 이미 비활성인 직원이라도 `name`, `password`, `retiredAt`, `departmentId` 같은 값은 여전히 수정할 수 있다. 즉 "비활성 직원은 조회만 가능" 규칙은 아직 서비스에 없다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **템플릿 공용 구조**: `AdminStaffController`는 등록과 수정 모두 `admin/staff-form`을 쓴다. 따라서 수정 화면에서 `재직 상태` 옆에 `퇴사 일시`를 배치하려면 등록 화면까지 같이 깨지지 않도록 `editMode` 기준 분기나 공용 grid 재배치가 필요하다.
- **입력 계약 문제**: `CreateAdminStaffRequest`, `UpdateAdminStaffRequest`, `UpdateAdminStaffApiRequest`가 모두 `LocalDateTime retiredAt` + `yyyy-MM-dd'T'HH:mm` 패턴을 전제로 한다. 즉 분 단위를 없애려면 날짜 input과 시간 select를 다시 조합해 `HH:00`으로 만들어 주는 공통 로직이 서비스나 컨트롤러 계층에 필요하다.
- **비활성 직원 수정 허용 범위**: 현재 `employmentStatusLocked`는 비활성 직원일 때 `재직 상태` select만 잠근다. 하지만 `retiredAtLocked`는 본인 계정에서만 잠기고, 서비스도 재활성화만 차단한다. 결과적으로 비활성 직원의 다른 정보 수정은 아직 허용되고 있다. 이번 작업에서 이 규칙을 SSR/API/서비스에 공통으로 올려야 한다.
- **SSR/API 동시 정리 필요**: 화면에서 저장 버튼을 숨기더라도 API나 SSR 저장 엔드포인트가 열려 있으면 규칙이 깨진다. 따라서 "비활성 직원 수정 불가"는 반드시 `AdminStaffService.updateStaff(...)` 기준의 공통 비즈니스 규칙으로 닫는 것이 안전하다.
