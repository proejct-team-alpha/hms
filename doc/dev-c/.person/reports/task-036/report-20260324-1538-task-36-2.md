## 2. 핵심 전달 코드 (Core Logic)

```java
@DateTimeFormat(pattern = "yyyy-MM-dd'T'HH:mm")
LocalDateTime retiredAt
```

현재 등록/수정 SSR DTO는 모두 `LocalDateTime retiredAt` 하나를 직접 받는다. 이 구조는 `datetime-local`에는 맞지만, 우리가 원하는 `날짜 + 시` 분리 입력과는 바로 연결되지 않는다.

```java
@JsonFormat(pattern = "yyyy-MM-dd'T'HH:mm")
LocalDateTime retiredAt
```

수정 API도 같은 형태다. 즉 화면만 바꾸고 API 계약을 그대로 두면, SSR과 API가 서로 다른 저장 규칙을 타게 될 가능성이 크다.

```java
LocalDateTime.of(date, hour, 0, 0)
```

이번 단계에서 고정한 핵심은 이 조합 규칙이다. 앞으로는 `retiredAtDate`와 `retiredAtHour`를 받아서, 서비스에서 항상 분 `00`으로 `LocalDateTime`을 조합한다.

## 3. 초등학생도 이해할 수 있게 비유 (Easy Analogy)

- 지금은 퇴사 시간을 적을 때 "몇 시 몇 분"까지 다 적게 하는 상태와 비슷하다.
- 그런데 우리한테는 "몇 시"까지만 있으면 충분하다.
- 그래서 앞으로는
  - 달력에서 날짜 하나 고르고
  - 시계에서 몇 시인지 하나 고르고
  - 시스템이 알아서 `00분`을 붙여 저장하게 만드는 거다.
- 이렇게 하면 사람이 적는 건 단순해지고, 컴퓨터 안에 저장되는 규칙은 더 일정해진다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **저장 모델 유지**: 도메인과 DB는 이미 `Staff.retiredAt: LocalDateTime` 기준으로 움직이고 있다. 여기까지 바꾸면 영향 범위가 너무 커지므로, 저장 모델은 유지하고 입력 계약만 바꾸는 것이 가장 안전하다.
- **입력 계약 분리**: SSR과 API 모두 `retiredAtDate`(`yyyy-MM-dd`) + `retiredAtHour`(`00~23`) 두 필드로 받는 방향이 자연스럽다. 이렇게 해야 등록 화면, 수정 화면, 수정 API가 같은 사용 경험과 검증 규칙을 공유할 수 있다.
- **검증 규칙 필요**: 두 필드가 모두 비어 있으면 `null`로 볼 수 있지만, 하나만 들어오면 모호하다. 따라서 이 경우는 검증 에러로 막는 것이 맞다. 이 규칙을 지금 설계 단계에서 못 박아두면 다음 단계 템플릿 작업이 흔들리지 않는다.
- **서비스 기준 조합**: 날짜와 시간 조합은 컨트롤러에서 각자 처리하면 SSR과 API가 다시 갈라질 가능성이 있다. `LocalDateTime.of(date, hour, 0, 0)` 조합은 서비스 공통 헬퍼에서 수행하고, 분은 항상 `00`으로 고정하는 것이 가장 일관적이다.
- **응답 DTO 정리 방향**: `AdminStaffFormResponse`와 수정 API 응답도 화면 렌더링/응답 표시를 위해 `retiredAtDate`, `retiredAtHour`를 제공하는 쪽이 좋다. 그래야 템플릿과 프런트가 다시 `LocalDateTime` 문자열을 파싱하지 않아도 된다.
