# AI 작업 보고서 - Task 36-5 비활성 직원 조회 전용 처리

- **작업 일시**: 2026-03-24
- **진행 상태**: 완료

## 1. 전체 작업 흐름 (Workflow)

1. 직원 수정 화면에서 비활성 직원이 어느 수준까지 수정 가능한지 다시 점검했습니다.
2. `AdminStaffService.updateStaff(...)` 시작 지점에 비활성 직원 수정 차단 검증을 넣어 SSR 저장과 관리자 수정 API가 같은 규칙을 타도록 맞췄습니다.
3. 수정 화면 응답 모델에서 `readOnly` 상태를 계산하도록 정리해, 비활성 직원은 화면 진입은 가능하지만 조회 전용으로 렌더링되게 바꿨습니다.
4. `staff-form.mustache`에서 주요 입력 필드를 disabled로 렌더링하고, 저장 버튼은 숨기고 `비활성화된 직원입니다. 조회만 가능합니다.` 안내 문구만 보이도록 정리했습니다.
5. `AdminStaffControllerTest`, `AdminStaffApiControllerTest`, `AdminStaffServiceTest`를 현재 규칙 기준으로 복구하고, 인코딩(BOM) 때문에 깨진 테스트 파일도 UTF-8 no BOM으로 다시 정리했습니다.
6. `./gradlew cleanTest test --tests 'com.smartclinic.hms.admin.staff.*'`를 실행해 범위 테스트가 통과하는지 확인했습니다.

## 2. 핵심 코드 (Core Logic)

```java
private void validateInactiveStaffUpdate(Staff staff) {
    if (!staff.isActive()) {
        throw CustomException.badRequest(
                "VALIDATION_ERROR",
                INACTIVE_STAFF_UPDATE_NOT_ALLOWED_MESSAGE
        );
    }
}

private AdminStaffFormResponse buildEditFormResponse(
        Staff staff,
        Doctor doctor,
        UpdateAdminStaffRequest request,
        String currentUsername
) {
    boolean selfEdit = currentUsername != null && staff.getUsername().equals(currentUsername);
    boolean readOnly = !staff.isActive();
    boolean employmentStatusLocked = selfEdit || readOnly;
    boolean retiredAtLocked = selfEdit || readOnly;

    return new AdminStaffFormResponse(
            // ...중간 값 생략...
            employmentStatusLocked,
            retiredAtLocked,
            readOnly,
            staff.getRole() == StaffRole.DOCTOR,
            // ...옵션 리스트...
    );
}
```

- `validateInactiveStaffUpdate(...)`가 비활성 직원 수정 자체를 서비스 레벨에서 막습니다.
- 이 검증을 `updateStaff(...)` 초입에 두어서 SSR 저장과 관리자 수정 API가 같은 규칙을 공유하게 했습니다.
- `buildEditFormResponse(...)`에서 `readOnly = !staff.isActive()`를 계산해, 화면 렌더링도 같은 기준을 쓰도록 맞췄습니다.

## 3. 쉬운 설명(비유)

- 이번 작업은 `비활성 직원 카드에 수정용 펜 대신 투명 보호 커버를 씌운 것`과 비슷합니다.
- 카드는 계속 볼 수 있지만, 더 이상 내용을 고치거나 저장할 수는 없습니다.
- 그리고 혹시 다른 경로로 억지로 수정하려고 해도 서비스와 API에서 한 번 더 막아 두어, 화면과 실제 저장 규칙이 어긋나지 않게 했습니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **서비스 기준 차단**: 화면에서만 disabled 처리하면 API나 다른 저장 경로로 우회할 수 있습니다. 그래서 `AdminStaffService.updateStaff(...)` 초입에서 비활성 직원 수정 차단을 먼저 수행해 SSR/API 규칙을 한 군데로 모았습니다.
- **SSR 읽기 전용 렌더링**: `AdminStaffFormResponse`에 `readOnly`를 추가해 템플릿이 비활성 직원 상태를 직접 렌더링하도록 했습니다. 이 값으로 입력 필드 disabled, 저장 버튼 숨김, 안내 문구 노출을 일관되게 처리했습니다.
- **테스트 복구와 인코딩 정리**: 작업 중 `AdminStaffApiControllerTest`, `AdminStaffServiceTest`에 BOM이 들어가 `compileTestJava`가 깨졌습니다. 두 파일을 UTF-8 no BOM으로 다시 저장하고, 현재 read-only 규칙과 메시지(`비활성화된 직원은 수정할 수 없습니다.`) 기준으로 테스트를 맞춰 안정화했습니다.
