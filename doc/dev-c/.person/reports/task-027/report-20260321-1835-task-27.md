## 1. 🌊 전체 작업 흐름 (Workflow)

1. 기존 `admin.rule` 코드를 점검해 보니 규칙 관리에는 목록과 등록만 있었고, 문서에 있던 `GET /admin/rule/detail`, `POST /admin/rule/update` 흐름은 아직 비어 있었다.
2. 먼저 수정 요구사항을 `detail 화면 안에서 수정 폼을 함께 보여주는 구조`로 고정하고, `UpdateAdminRuleRequest`, `AdminRuleDetailResponse`를 추가해 조회 모델과 수정 요청 모델을 분리했다.
3. `AdminRuleService`에는 `getRuleDetail(...)`, `updateRule(...)`를 추가해 규칙 조회와 전체 필드 수정이 서비스에서 끝나도록 맞췄다.
4. `AdminRuleController`에는 상세 조회, 수정 저장, same-view 재렌더링, not found 404, 성공 플래시 리다이렉트까지 한 흐름으로 붙였다.
5. 템플릿은 등록 화면을 그대로 복붙하지 않고 `admin/_rule-form.mustache` 공통 partial을 만들고, `rule-new.mustache`와 새 `rule-detail.mustache`가 같은 폼 블록을 재사용하도록 정리했다.
6. 목록에서 상세로 들어갈 수 있도록 `rule-list.mustache` 제목 링크를 상세 URL로 연결했다.
7. 마지막으로 컨트롤러/서비스 테스트를 보강하고 `task-027.md`, `workflow-027.md`를 현재 구현 상태에 맞게 완료 처리했다.

## 2. 🧩 핵심 코드 (Core Logic)

```java
@PostMapping("/update")
public Object update(
        @Valid @ModelAttribute("model") UpdateAdminRuleRequest request,
        BindingResult bindingResult,
        RedirectAttributes redirectAttributes,
        Model model,
        HttpServletRequest req) {
    // 입력 검증 실패면 같은 상세 화면을 다시 그린다
    if (bindingResult.hasErrors()) {
        return renderUpdateValidationFailure(req, model, request, bindingResult, redirectAttributes);
    }

    try {
        // 저장 성공 시 상세 화면으로 PRG 리다이렉트
        String successMessage = adminRuleService.updateRule(request);
        redirectAttributes.addFlashAttribute("successMessage", successMessage);
        return redirectToDetail(request.ruleId());
    } catch (CustomException ex) {
        // 대상이 없으면 목록으로, 그 외는 상세로 돌려보낸다
        redirectAttributes.addFlashAttribute("errorMessage", ex.getMessage());
        if (ex.getHttpStatus() == HttpStatus.NOT_FOUND) {
            return redirectTo("/admin/rule/list");
        }
        return redirectToDetail(request.ruleId());
    }
}
```

## 4. 📚 기술 딥다이브 (Technical Deep-dive)

- **SSR same-view 재렌더링**: 수정 실패 시 새 페이지로 보내지 않고 같은 상세 화면을 다시 그려 사용자가 입력하던 값을 유지하는 방식이다. 관리자 폼 UX에서 입력 유실을 줄이는 데 중요하다.
- **PRG 패턴**: 수정 성공 후 바로 뷰를 반환하지 않고 상세 화면으로 리다이렉트한다. 새로고침 시 중복 저장을 막고, `successMessage` 플래시를 안정적으로 보여줄 수 있다.
- **Mustache partial 재사용**: 등록과 수정의 필드 구성이 거의 같아서 폼 블록을 partial로 분리했다. 이렇게 하면 제목, action URL, 버튼 문구, 초기값만 바꿔도 두 화면을 같이 유지할 수 있다.
- **DTO 분리**: `UpdateAdminRuleRequest`와 `AdminRuleDetailResponse`를 나눠서 입력 검증 책임과 화면 출력 책임을 분리했다. 등록/수정/상세가 섞이지 않아 이후 유지보수가 쉬워진다.
