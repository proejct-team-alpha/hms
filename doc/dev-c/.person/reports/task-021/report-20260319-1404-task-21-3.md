4. 상태 필터 영역 앞에 `상태 필터` 라벨을 추가해, 상단 필터 버튼 묶음의 의미가 더 명확하게 보이도록 정리했다. 빈 목록 메시지도 `조회된 예약이 없습니다.`에서 `조회된 내역이 없습니다.`로 바꿔, 예약/접수 통합 흐름과 더 잘 맞도록 맞췄다.
5. 마지막으로 [AdminReservationControllerTest.java](c:/workspace/Team/hms/src/test/java/com/smartclinic/hms/admin/reservation/AdminReservationControllerTest.java)에 실제 HTML 렌더링 문자열 검증을 추가해, 설명 문구/필터 라벨/빈 목록 메시지/`접수` 라벨이 다음 변경에서 다시 흔들리지 않도록 고정했다.

## 2. 핵심 코드 (Core Logic)

```mustache
<div>
  <h1 class="text-2xl font-bold text-slate-800">{{pageTitle}}</h1>
  <p class="text-slate-500 mt-1">예약과 접수 현황을 상태별로 확인하고 관리합니다.</p>
</div>

<div class="p-4 border-b border-slate-200 flex flex-wrap items-center gap-2 bg-slate-50">
  <span class="text-xs font-semibold text-slate-500 uppercase tracking-wide mr-1">상태 필터</span>
  {{#model.statusOptions}}
    <a href="{{url}}">{{label}}</a>
  {{/model.statusOptions}}
</div>
```

이 변경은 UI 구조를 갈아엎는 것이 아니라, 기존 예약 목록이 `접수`를 포함한 관리 화면으로 자연스럽게 읽히도록 문장과 안내 포인트를 바로잡는 데 집중한다.

## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 원래 이 화면은 마치 “예약만 보는 책상”처럼 써 있었는데, 실제로는 접수도 같이 보고 있었다.
- 그래서 책상 위 안내판을 “예약과 접수를 같이 보는 책상”으로 다시 붙여준 셈이다.
- 버튼이 여러 개 있을 때 그 앞에 `상태 필터`라고 적어둔 건, 서랍 이름표를 붙여서 덜 헷갈리게 한 것과 비슷하다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 문구 수정이 중요한가**: 현재 방향은 별도 `/admin/reception/list`를 만드는 것이 아니라, `/admin/reservation/list` 안에서 `접수`를 함께 관리하는 것이다. 그렇다면 화면 설명 문구도 그 책임을 반영해야 한다. 문구가 여전히 “전체 예약 내역”만 말하고 있으면, 사용자는 왜 `접수` 필터가 있는지 직관적으로 이해하기 어렵다.
- **왜 상태 필터 라벨을 추가했는가**: 기존에도 버튼 묶음은 있었지만, 화면 상단 정보와 바로 붙어 있어서 첫 진입 시 “이 버튼 묶음이 상태 탭인지”가 덜 분명할 수 있었다. 얇은 라벨 하나만 추가해도 정보 구조가 훨씬 또렷해진다.
- **빈 목록 메시지를 바꾼 이유**: `조회된 예약이 없습니다.`는 `접수` 필터 상태에서 보면 약간 어색해질 수 있다. `조회된 내역이 없습니다.`는 예약/접수 어느 필터 상태에도 자연스럽게 들어맞는다.
- **왜 컨트롤러 테스트에 렌더 문자열을 넣었는가**: 이번 단계는 서비스보다 템플릿 문구가 핵심이기 때문에, 단순히 request attribute만 보는 테스트로는 회귀를 막기 어렵다. 실제 HTML에 원하는 문구가 들어가는지까지 한 번 확인해두면, 후속 템플릿 변경에서 훨씬 안정적이다.
