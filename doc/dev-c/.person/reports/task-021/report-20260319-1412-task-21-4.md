## 2. 핵심 코드 (Core Logic)

```java
private String normalizeStatus(String status) {
    if (status == null || status.isBlank()) {
        return "ALL";
    }

    String normalized = status.trim().toUpperCase(Locale.ROOT);
    return ALLOWED_STATUSES.contains(normalized) ? normalized : "ALL";
}
```

이 메서드는 예약 취소 후 목록으로 돌아갈 때 `status` 파라미터를 한 번 정리해 준다. 즉 `RECEIVED` 복귀는 그대로 살리고, 잘못된 값은 안전하게 `ALL`로 보내는 역할을 한다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 컨트롤러에서 정규화했는가**: 이번 문제는 서비스 조회 자체보다 “리다이렉트 URL 파라미터” 문제에 가깝다. 사용자가 취소 버튼을 눌렀을 때 어떤 URL로 되돌아가느냐는 컨트롤러 책임이므로, 서비스보다는 컨트롤러에서 안전하게 정규화하는 편이 더 자연스럽다.
- **왜 `toUpperCase()`만으로는 부족했는가**: 소문자를 대문자로 바꾸는 것만으로는 `INVALID` 같은 잘못된 상태를 걸러내지 못한다. 결국 허용된 집합으로 한 번 더 검증해야 “접수 복귀는 유지하되, 엉뚱한 값은 퍼지지 않게” 만들 수 있다.
- **이번 단계가 중요한 이유**: 접수 필터를 별도 화면으로 분리하지 않고 `/admin/reservation/list` 안에서 함께 관리하기로 했기 때문에, 취소 후에도 사용자가 같은 필터 상태로 다시 돌아오는 경험이 중요하다. 여기서 흐름이 흔들리면 접수 필터 UX가 금방 어색해진다.
- **테스트 전략**: 이번 테스트는 성공 메시지 자체보다 `Location` 헤더를 더 중요하게 본다. 왜냐하면 사용자가 실제로 체감하는 건 “어디로 돌아갔는가”이기 때문이다. 그래서 `status=RECEIVED` 유지, `status=INVALID -> ALL`, `status=received -> RECEIVED`를 각각 URL 수준에서 검증했다.
