## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 Task 21-6이 중요한가**: 많은 작업이 기능 구현에서 끝나면, 이후 리뷰나 후속 작업에서 의사결정 맥락이 사라진다. 이번 단계에서는 “왜 `/admin/reception/list`를 만들지 않았는가”, “왜 `RECEIVED` 내부값을 유지했는가”, “왜 접수 필터를 예약 목록에 통합했는가” 같은 판단 근거를 문서에 남겨두는 것이 핵심이었다.
- **문서와 코드의 정합성**: [workflow-021.md](c:/workspace/Team/hms/doc/dev-c/workflow/workflow-021.md)는 요구사항과 수용 기준을 담고 있고, [task-021.md](c:/workspace/Team/hms/doc/dev-c/task/task-021.md)는 실제 실행 체크리스트다. 최종 단계에서 둘 다 완료 상태로 갱신하지 않으면, 코드가 맞더라도 문서상으로는 미완료처럼 보일 수 있다.
- **왜 전체 테스트를 다시 돌렸는가**: 앞선 단계들에서는 `admin.reservation` 범위 테스트를 중심으로 확인했다. 하지만 PR 직전에는 “이번 브랜치 전체 상태”가 더 중요하므로, `./gradlew test`를 다시 돌려 전체 초록 상태를 확보하는 것이 필요했다.
- **후속 분리 기준을 왜 남겼는가**: 지금은 접수 전용 화면을 따로 만들지 않는 판단이 맞다. 다만 이후 실시간 모니터링, 접수 통계, 보드형 레이아웃 같은 요구가 붙으면 판단이 바뀔 수 있다. 이 기준을 문서에 남겨두면, 나중에 다시 같은 토론을 처음부터 반복하지 않아도 된다.
