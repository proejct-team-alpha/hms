## 2. 핵심 코드 (Core Logic)

```java
@Test
@DisplayName("예약 목록은 접수 상태 예약 row를 접수 배지로 렌더링한다")
void list_rendersReceivedReservationRowWithReceptionBadge() throws Exception {
    // given
    AdminReservationItemResponse receivedItem = new AdminReservationItemResponse(
            12L,
            "RES-20260319-012",
            "2026-03-19",
            "10:30",
            "홍길동",
            "010-1234-5678",
            "내과",
            "김의사",
            "RECEIVED",
            "접수",
            true,
            false,
            true,
            false,
            false
    );

    // when
    // then
    mockMvc.perform(get("/admin/reservation/list")
                    .param("status", "RECEIVED")
                    .with(user("admin").roles("ADMIN"))
                    .with(csrf()))
            .andExpect(status().isOk())
            .andExpect(content().string(containsString("RES-20260319-012")))
            .andExpect(content().string(containsString("홍길동")))
            .andExpect(content().string(containsString(">접수</span>")));
}
```

이 테스트는 상태 필터 라벨만 보는 데서 끝나지 않고, 실제 목록 row에서 사용자가 보게 되는 `접수` 배지까지 확인한다. 즉 필터/서비스/목록 행 표현이 서로 어긋나는 회귀를 막아준다.

## 3. 초등학생도 이해할 수 있는 비유 (Easy Analogy)

- 필터 버튼에 `접수`라고 써 있어도, 표 안의 줄에서는 다른 말이 나오면 이상하다.
- 이번 테스트는 상단 이름표와 표 안 이름표가 둘 다 `접수`로 맞는지 확인하는 검사다.
- 즉 문 앞 간판과 교실 안 칠판 이름이 서로 같은지 확인하는 것과 비슷하다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **왜 row 렌더링 테스트를 추가했는가**: 지금까지의 검증은 크게 두 갈래였다. 하나는 서비스 레이어의 `RECEIVED -> 접수` 라벨 계약, 다른 하나는 컨트롤러의 필터 선택/리다이렉트 흐름이었다. 그런데 사용자가 진짜 보는 건 목록 row의 상태 배지이기도 하다. 따라서 화면 계약을 닫으려면 row 배지까지 한 번은 보는 게 더 안전하다.
- **왜 `admin.reservation.*` 범위를 한 번 더 돌렸는가**: Task 21-2 ~ 21-5는 서비스 테스트, 컨트롤러 테스트, 템플릿 렌더링 테스트가 각각 조금씩 다른 관점으로 쌓여 있다. 이런 경우 개별 파일 테스트만 보면 부분적으로는 초록이어도 묶어서 돌릴 때 다른 회귀가 튈 수 있다. 그래서 마지막에 모듈 범위 테스트를 다시 한 번 확인하는 게 중요했다.
- **이번 단계의 의미**: 이 작업은 구현 추가보다 “기능 이름과 표현을 시스템 전체에서 일관되게 맞추는 작업”의 마침표에 가깝다. 서비스는 `RECEIVED`, UI는 `접수`, 리다이렉트는 `status=RECEIVED`, 템플릿은 `접수` 배지라는 식으로 층마다 역할이 달라도 사용자 경험은 하나로 맞아야 한다.
- **Task 21-6로 자연스럽게 이어지는 이유**: 이제 남은 건 실제 기능 추가라기보다 workflow/task 문서 정리와 최종 리뷰 포인트 정리다. 즉 회귀 방지 테스트는 충분히 닫혔고, 남은 단계는 문서와 최종 점검 중심으로 넘어갈 수 있다.
