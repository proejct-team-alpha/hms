## 1. 전체 작업 흐름 (Workflow)
1. 관리자 대시보드 환자 그래프가 현재 어떤 데이터 계약과 렌더링 경로를 가지는지 다시 확인했다.
2. 원무과 대시보드의 `시간대별 예약 및 접수 현황` 그래프를 읽고, 우리가 실제로 가져와야 하는 요소가 데이터가 아니라 시각 스타일이라는 점을 분리했다.
3. 관리자 환자 그래프는 `dailyPatients` 계약을 그대로 유지하고, 스타일 변경만 하기로 범위를 고정했다.
4. 막대/축/카드 톤은 원무과 그래프를 따르되, 일자별 환자 수라는 현재 의미는 유지해야 하므로 line chart 계약으로 바꾸지 않는 방향을 정리했다.
5. 다음 단계에서 실제 구현이 필요한 파일 범위를 `admin/dashboard.mustache`, `admin-dashboard.js`로 좁히고, 테스트는 SSR 렌더링과 기존 API/서비스 계약 재사용 기준으로 닫기로 정리했다.

## 4. 기술 딥다이브 (Technical Deep-dive)
- **데이터 계약 유지**: 관리자 환자 그래프는 `AdminDashboardChartResponse.dailyPatients`를 그대로 사용한다. 이 계약을 유지하면 `AdminDashboardApiController`, `AdminDashboardStatsService`의 집계 로직과 기존 zero-fill 보장은 건드리지 않아도 된다.
- **스타일 차용 범위**: 원무과 그래프는 Chart.js line chart이지만, 핵심은 `인디고 중심 색감`, `옅은 슬레이트 계열 격자선`, `작은 축 라벨`, `여백이 안정적인 카드 레이아웃`이다. 이 스타일 요소만 관리자 그래프에 이식한다.
- **표현 방식 유지**: 관리자 그래프는 현재 일자별 환자 수를 세로 막대로 표현하고 있다. 일자별 합계를 보여주는 문맥에서는 막대 표현이 자연스럽기 때문에, 그래프 타입까지 원무과와 동일하게 line chart로 바꾸는 것은 이번 범위에서 제외했다.
- **변경 파일 범위 고정**: 실제 구현은 `admin/dashboard.mustache`의 카드 마크업과 `admin-dashboard.js`의 `renderDailyPatientChart(...)` 중심으로 끝내는 것이 맞다. 이 범위를 지키면 Task 34-3의 리스크가 작고, Task 34-4의 물품 그래프 계약 변경 작업과도 충돌이 적다.
- **테스트 전략**: 스타일 변경은 숫자 계산보다는 렌더링 구조가 중요하다. 그래서 테스트는 SSR 렌더링에서 그래프 카드 제목과 `canvas`/에러 영역 id가 유지되는지 확인하고, 기존 API/서비스 테스트로 데이터 계약이 그대로인지 간접적으로 보장하는 방식이 적절하다.
