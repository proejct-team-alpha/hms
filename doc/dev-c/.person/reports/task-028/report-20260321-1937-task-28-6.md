## 4. 기술 딥다이브 (Technical Deep-dive)

- **API 컨트롤러 테스트 분리**: [AdminRuleApiControllerTest.java](/c:/Users/HSystem/hms/src/test/java/com/smartclinic/hms/admin/rule/AdminRuleApiControllerTest.java)는 [AdminControllerTestSecurityConfig.java](/c:/Users/HSystem/hms/src/test/java/com/smartclinic/hms/common/AdminControllerTestSecurityConfig.java)를 사용해 `/admin/**` 보안 규칙을 그대로 적용했다. 이 덕분에 규칙 삭제 API도 기존 관리자 API와 같은 방식으로 403 차단을 검증할 수 있다.
- **성공 응답 구조 검증**: 성공 케이스에서 `$.status`, `$.body.ruleId`, `$.body.message`를 검증해 `Resp.ok(...)` 래퍼와 [AdminRuleDeleteResponse.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/dto/AdminRuleDeleteResponse.java)의 조합이 실제 응답에서 맞게 직렬화되는지 고정했다.
- **404 공통 실패 응답 검증**: not found 케이스는 서비스에서 `CustomException.notFound(...)`를 던지게 한 뒤, 컨트롤러 테스트에서 `status=404`와 `msg` 안의 `RESOURCE_NOT_FOUND`를 확인했다. 이로써 삭제 API가 별도 실패 바디를 만들지 않고 공통 `Resp.fail(...)` 흐름을 탄다는 점이 테스트로 고정됐다.
- **물리 삭제 서비스 검증**: 서비스 테스트는 [HospitalRuleRepository.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/HospitalRuleRepository.java)의 `delete(rule)` 호출 여부를 직접 검증해, 현재 삭제 구현이 soft delete가 아니라 실제 엔티티 삭제라는 점을 명시적으로 보장한다.
