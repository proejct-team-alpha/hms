## 2. 핵심 코드 (Core Logic)

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/admin/api/rules")
public class AdminRuleApiController {

    private final AdminRuleService adminRuleService;

    @PostMapping("/{id}")
    public ResponseEntity<Resp<AdminRuleDeleteResponse>> deleteRule(
            @PathVariable("id") Long ruleId) {
        AdminRuleDeleteResponse response = adminRuleService.deleteRule(ruleId);
        return Resp.ok(response);
    }
}
```

```java
@Transactional
public AdminRuleDeleteResponse deleteRule(Long ruleId) {
    HospitalRule rule = findRule(ruleId);
    hospitalRuleRepository.delete(rule);
    return AdminRuleDeleteResponse.success(ruleId);
}
```

## 4. 기술 딥다이브 (Technical Deep-dive)

- **기존 관리자 JSON API 패턴 유지**: 새 컨트롤러 [AdminRuleApiController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleApiController.java)는 기존 [AdminReservationApiController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/reservation/AdminReservationApiController.java), [AdminStaffApiController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/staff/AdminStaffApiController.java)와 같은 형식을 따른다. 이 덕분에 응답 래퍼, 예외 처리, 테스트 방식이 기존 관리자 API와 자연스럽게 맞는다.
- **Task 28-3과 28-4의 결합 이유**: 원래 체크리스트상 28-3은 컨트롤러, 28-4는 서비스 삭제 로직으로 분리돼 있었지만, 컨트롤러만 먼저 추가하면 실제 요청이 동작하지 않는다. 그래서 이번 단계에서는 API 엔드포인트를 진짜 사용 가능한 상태로 만들기 위해 최소한의 서비스 삭제 로직까지 같이 붙였다. 문서에는 이 범위를 숨기지 않고 그대로 반영했다.
- **not found 규칙 재사용**: 새 삭제 메서드는 별도 예외 분기를 새로 만들지 않고 기존 `findRule(...)`을 재사용한다. 따라서 존재하지 않는 규칙 삭제 요청은 기존과 동일하게 `CustomException.notFound(...)`로 연결되고, 이후 [GlobalExceptionHandler.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/common/exception/GlobalExceptionHandler.java)의 공통 JSON 실패 응답 규칙을 그대로 탄다.
- **물리 삭제 유지**: 사용자 결정대로 삭제는 `active=false` 전환이 아니라 `hospitalRuleRepository.delete(rule)` 기반의 실제 엔티티 삭제다. 이번 단계에서는 soft delete 규칙을 새로 만들지 않았고, 현재 도메인/리포지토리 구조에도 가장 자연스럽게 맞춘 구현이다.
