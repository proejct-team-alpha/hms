## 4. 기술 딥다이브 (Technical Deep-dive)

- **공통 실패 처리 재사용**: [AdminRuleApiController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleApiController.java)는 `@RestController`이므로, 서비스에서 `CustomException.notFound(...)`가 발생하면 [GlobalExceptionHandler.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/common/exception/GlobalExceptionHandler.java)의 `handleBusiness(...)`가 이를 받아 `Resp.fail(...)` 형식 JSON으로 내려준다. 따라서 삭제 API가 별도 실패 바디를 직접 만들 필요가 없다.
- **404 흐름 고정**: [AdminRuleService.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleService.java)의 `deleteRule(...)`는 기존 `findRule(...)`을 재사용한다. 이 메서드는 규칙이 없으면 `CustomException.notFound(...)`를 던지므로, 삭제 API의 not found도 규칙 조회/상세/수정과 같은 예외 규칙을 공유한다.
- **보안 규칙 재사용**: [SecurityConfig.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/config/SecurityConfig.java)에서 `/admin/**` 전체가 `ROLE_ADMIN` 전용으로 묶여 있으므로 `POST /admin/api/rules/{id}`도 별도 예외 없이 관리자 전용이다. 기존 관리자 API 테스트들도 같은 방향으로 `non-admin -> 403`을 검증하고 있다.
- **이번 단계의 범위 제한**: 현재 메인 보안 설정은 접근 거부 시 `accessDeniedPage("/error/403")`를 사용한다. 하지만 기존 관리자 API들도 같은 구조를 공유하고 있고, 이번 task 범위는 삭제 API를 그 공통 패턴에 맞추는 데 있으므로 별도 JSON 인증/인가 핸들러 확장까지는 포함하지 않았다. 즉, 이번 단계는 “새 예외 체계 추가”가 아니라 “기존 공통 규칙 적용 확인” 단계다.
