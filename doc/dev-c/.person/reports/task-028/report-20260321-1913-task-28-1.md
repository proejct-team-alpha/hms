## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR 컨트롤러와 JSON API 컨트롤러 분리**: 현재 규칙 모듈은 [AdminRuleController.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/AdminRuleController.java) 하나로 목록/등록/상세/수정 SSR만 처리한다. 반면 기존 관리자 JSON API는 `AdminReservationApiController`, `AdminStaffApiController`, `AdminPatientApiController`처럼 별도 `@RestController`로 분리되어 있다. 그래서 삭제 API도 기존 규칙을 따르려면 `AdminRuleApiController`를 새로 두는 편이 구조적으로 가장 맞다.
- **Resp 공통 래퍼 패턴**: [Resp.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/common/util/Resp.java)는 `status`, `msg`, `body`를 감싸고, 성공 시 `Resp.ok(body)`로 `200 + 성공` 응답을 내려준다. 따라서 규칙 삭제 성공 응답 body는 `ruleId`, `message`만 담고, 성공 여부나 상태 코드는 `Resp` 래퍼에 맡기는 방향이 가장 일관적이다.
- **물리 삭제 가능성**: [HospitalRuleRepository.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/admin/rule/HospitalRuleRepository.java)는 `JpaRepository`를 상속하므로 별도 커스텀 쿼리 없이도 물리 삭제 메서드를 바로 사용할 수 있다. [HospitalRule.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/domain/HospitalRule.java)에 `active` 필드는 있지만 soft delete 전용 정책은 없어서, 이번 요구는 `active=false` 전환보다 실제 삭제로 보는 해석이 자연스럽다.
- **보안 적용 범위**: [SecurityConfig.java](/c:/Users/HSystem/hms/src/main/java/com/smartclinic/hms/config/SecurityConfig.java)에서 `/admin/**`는 `hasRole("ADMIN")`으로 묶여 있다. 따라서 `POST /admin/api/rules/{id}`를 추가해도 별도 보안 예외를 두지 않는 한 자동으로 관리자 전용 API가 된다.
