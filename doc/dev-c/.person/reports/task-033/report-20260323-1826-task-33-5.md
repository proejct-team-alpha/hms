## 3. 쉬운 비유로 설명하면 (Easy Analogy)

- 이번 작업은 검색 기능의 엔진은 이미 달아둔 상태에서, 운전석에 날짜 스위치와 버튼을 실제로 붙여 사용자가 바로 조작할 수 있게 만든 것과 비슷합니다.
- 엔진만 있으면 내부적으로는 움직일 수 있지만, 사용자는 어디를 눌러야 하는지 모릅니다. 이번 단계는 그 조작부를 화면에 올리고, “오늘 기준으로 시작”하는 기본 상태도 같이 맞춘 작업입니다.
- 특히 출고 화면은 기록을 보는 자리이면서 출고를 실행하는 자리이기도 해서, 날짜 스위치를 맞춘 채 작업을 해도 계속 그 날짜 맥락이 유지되도록 손잡이까지 같이 묶어둔 셈입니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **템플릿 UTF-8 정리**: 기존 `item-history.mustache`, `item-use.mustache`는 한글이 일부 깨져 있고 input/placeholder 문자열도 흔들려 있었다. 날짜 검색 UI를 얹는 김에 템플릿을 UTF-8 기준으로 다시 써서 이후 단계에서 문구 비교 테스트나 유지보수가 덜 흔들리게 만들었다.
- **SSR 기본값 노출**: 서버는 이미 `fromDate`, `toDate`를 모델에 내려주고 있었기 때문에, UI 단계에서는 `<input type="date" value="{{fromDate}}">` 식으로 그대로 연결해 `오늘 ~ 오늘` 기본값이 첫 렌더링부터 보이도록 했다.
- **서버 날짜 필터 + 클라이언트 보조 필터 분리 유지**: `item-history`는 서버가 날짜 범위로 데이터 집합을 먼저 좁히고, 화면에서는 그 안에서만 물품명/입출고 구분 필터가 동작한다. `item-use`도 같은 패턴으로, 출고 로그는 서버 날짜 범위를 따르고 물품 카드 검색/카테고리 필터는 클라이언트에서 계속 처리한다.
- **POST 이후 날짜 맥락 유지**: 출고 화면은 조회와 실행이 동시에 일어나는 화면이라, 출고 후 로그를 갱신할 때 날짜 범위를 잃으면 UX가 갑자기 오늘 기준으로 튈 수 있다. 그래서 hidden field와 JS `URLSearchParams`를 둘 다 써서 현재 날짜 범위를 POST 요청에 같이 실어 보내도록 보완했다.
