## 1. 전체 작업 흐름 (Workflow)

1. `item-history`, `item-use` 화면이 서버 날짜 검색과 클라이언트 필터를 함께 쓰는 구조라는 점을 다시 확인했다.
2. 날짜 검색 폼 제출 시 클라이언트 필터 상태가 사라지지 않도록 hidden field를 추가했다.
3. 각 화면의 JavaScript에서 현재 필터 상태를 URL query string과 동기화하도록 정리했다.
4. 클라이언트 필터가 바뀌면 페이지 번호를 1로 되돌리도록 보정했다.
5. 컨트롤러 테스트에 상태 유지용 hidden field 렌더링 검증을 추가하고, `admin.item` 범위 테스트를 다시 돌려 회귀가 없는지 확인했다.

## 2. 핵심 전달 코드 (Core Logic)

```html
<form method="get" action="/admin/item/history" id="history-date-form">
  <input type="hidden" name="keyword" id="history-hidden-keyword" value="">
  <input type="hidden" name="type" id="history-hidden-type" value="">
</form>
```

```javascript
function syncStateToUrl() {
  const params = new URLSearchParams(window.location.search);

  if (currentQuery) params.set('keyword', currentQuery);
  else params.delete('keyword');

  if (currentType) params.set('type', currentType);
  else params.delete('type');

  const nextUrl = `${window.location.pathname}${params.toString() ? `?${params.toString()}` : ''}`;
  window.history.replaceState({}, '', nextUrl);
}
```

- 날짜 검색은 서버가 처리하고,
- 물품명/구분/카테고리 필터는 클라이언트가 처리하되,
- URL과 hidden field를 통해 두 상태를 다시 만날 수 있게 만든 것이 이번 단계의 핵심이다.

## 3. 쉬운 비유 (Easy Analogy)

- 이번 작업은 책갈피를 꽂아둔 상태로 페이지를 다시 펼치는 것과 비슷하다.
- 날짜를 바꿔 다시 조회해도, 사용자가 방금 보고 있던 검색어와 필터 위치를 잃지 않게 해서 같은 맥락으로 바로 이어서 볼 수 있게 했다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **SSR + 클라이언트 필터 역할 분리**: 두 화면은 서버 페이지네이션이 아니라 클라이언트 필터/페이지네이션을 같이 쓰고 있다. 그래서 날짜 범위만 서버에서 처리하고, 나머지 검색 상태는 URL과 JS 동기화로 이어주는 방식이 가장 덜 흔들린다.
- **history.replaceState 활용**: 필터가 바뀔 때마다 페이지 새로고침 없이 현재 URL query string을 갱신했다. 덕분에 새로고침이나 날짜 재조회 후에도 같은 상태를 복원할 수 있다.
- **페이지 번호 보정**: 클라이언트 필터가 바뀌면 결과 개수가 줄어들 수 있으므로, 기존 페이지 번호를 유지하면 빈 페이지가 보일 수 있다. 이 문제를 막기 위해 필터 변경 시 항상 1페이지로 되돌렸다.
- **테스트 안정화**: 한국어 문자열은 인코딩 환경에 따라 테스트가 흔들릴 수 있어, 이번 단계는 hidden field id와 폼 id 같은 DOM 고정 포인트 중심으로 렌더링 검증을 보강했다.
