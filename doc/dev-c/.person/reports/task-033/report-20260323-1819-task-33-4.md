## 2. 핵심 전달 코드 (Core Logic)

```java
@GetMapping("/use")
public String itemUsePage(@RequestParam(name = "fromDate", required = false) String fromDate,
                          @RequestParam(name = "toDate", required = false) String toDate,
                          Model model) {
    DateRange dateRange = resolveDateRange(fromDate, toDate);
    List<ItemUsageLogDto> usageLogs = List.of();
    long totalUsedAmount = 0L;

    if (!dateRange.hasError()) {
        usageLogs = itemManagerService.getStaffUsageLogs(dateRange.parsedFromDate(), dateRange.parsedToDate());
        totalUsedAmount = itemManagerService.getTotalStaffUsageAmount(
                dateRange.parsedFromDate(),
                dateRange.parsedToDate());
    }

    model.addAttribute("todayLogs", usageLogs);
    model.addAttribute("todayLogCount", usageLogs.size());
    model.addAttribute("todayTotalUsedAmount", totalUsedAmount);
    model.addAttribute("fromDate", dateRange.fromDate());
    model.addAttribute("toDate", dateRange.toDate());
    if (dateRange.hasError()) {
        model.addAttribute("dateError", dateRange.errorMessage());
    }
    return "admin/item-use";
}
```

```java
public List<ItemUsageLogDto> getStaffUsageLogs(LocalDate fromDate, LocalDate toDate) {
    return usageLogRepository.findByReservationIdIsNullAndUsedAtGreaterThanEqualAndUsedAtLessThanOrderByUsedAtDesc(
                    fromDate.atStartOfDay(),
                    toDate.plusDays(1).atStartOfDay())
            .stream()
            .map(ItemUsageLogDto::new)
            .toList();
}
```

## 3. 쉬운 비유로 설명하면 (Easy Analogy)

- 이번 작업은 출고 창고 문 앞에 날짜 출입증을 붙인 것과 비슷합니다.
- 전에는 출고 화면이 무조건 “오늘 기록만” 다시 보여줬다면, 이제는 사용자가 어떤 날짜 범위를 보고 있었는지 같은 출입증을 붙여서 그 범위 안의 기록만 다시 보여주게 된 상태입니다.
- 그래서 나중에 날짜 검색 UI를 붙여도 화면이 갑자기 오늘 기록으로 튀지 않고, 사용자가 보고 있던 기간 맥락을 유지할 수 있습니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **공통 날짜 해석 로직**: `/admin/item/history`와 `/admin/item/use`가 같은 검증 규칙을 가져야 하므로, 컨트롤러 내부에 `DateRange` record와 `resolveDateRange()` 헬퍼를 두고 기본값/한쪽 누락/역전/파싱 실패를 한 군데에서 처리하도록 정리했다.
- **반개구간 날짜 범위 조회**: 출고 로그도 입출고 이력과 동일하게 `fromDate.atStartOfDay()` 이상, `toDate.plusDays(1).atStartOfDay()` 미만 범위로 조회하도록 통일했다. 이 방식은 시/분/초가 있는 로그에서 경계값 실수를 줄여준다.
- **GET 화면 + POST 갱신 일관성**: 출고 화면은 조회 화면이면서 동시에 실행 화면이기 때문에 GET에서만 날짜 범위를 적용하면 POST 후 로그 갱신이 오늘 기준으로 돌아가는 문제가 생긴다. 그래서 POST `/admin/item/use`도 같은 날짜 범위를 선택적으로 받도록 열어두고, 템플릿 JS가 현재 URL 쿼리를 함께 보내도록 보완했다.
- **서비스 재사용 구조**: 기존 `getTodayStaffUsageLogs()`를 없애지 않고 새 범위 메서드 위에 얹어 두어, 다른 화면은 기존 메서드를 계속 써도 되고 admin 쪽만 날짜 범위를 확장해도 되게 만들었다.
