## 2. 핵심 전달 코드 (Core Logic)

```text
입력 계약
- fromDate: yyyy-MM-dd
- toDate: yyyy-MM-dd

기본 규칙
- 두 값이 모두 없으면 오늘 ~ 오늘 기본 세팅
- 한쪽만 있으면 검증 에러
- fromDate > toDate 면 검증 에러

조회 경계
- 시작: fromDate.atStartOfDay()
- 끝: toDate.plusDays(1).atStartOfDay() 직전

화면 적용 방식
- item-history: 서버 날짜 필터 + 기존 JS 검색/구분 필터 유지
- item-use: 서버 날짜 필터로 todayLogs 범위 변경 + 기존 JS 물품 검색/카테고리 필터 유지
```

## 4. 기술 딥다이브 (Technical Deep-dive)

- **파라미터 계약 고정**: 날짜는 `yyyy-MM-dd` 문자열로 받고 컨트롤러에서 `LocalDate`로 다루는 방향으로 정리했습니다. 이 방식이 SSR 폼과 가장 자연스럽게 맞습니다.
- **기간 경계 처리**: 로그는 시분초를 가진 datetime이므로, `toDate`를 같은 날 23:59:59로 임의 계산하기보다 `toDate.plusDays(1).atStartOfDay()` 직전까지로 보는 반개구간 방식이 더 안전합니다.
- **서버 필터와 클라이언트 필터의 분리**: `item-history`와 `item-use` 모두 이미 화면 안에 JS 검색/필터가 있습니다. 그래서 날짜는 서버에서 로그 집합을 먼저 줄이고, 이후 물품명/구분/카테고리 필터는 현재 클라이언트 흐름을 유지하는 쪽으로 설계를 고정했습니다.
- **초기화 UX 기준**: 이번 요구에서는 전체 비움보다 오늘 내역이 더 실무적이므로, 초기화도 빈 값으로 되돌리는 대신 `오늘 ~ 오늘`로 복귀하는 방향으로 정리했습니다.
