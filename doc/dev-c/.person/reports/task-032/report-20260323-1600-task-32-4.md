## 2. 핵심 전달 코드 (Core Logic)

```java
public void updateStaff(UpdateAdminStaffRequest request) {
    Staff staff = adminStaffRepository.findById(request.staffId())
            .orElseThrow(() -> CustomException.notFound("직원을 찾을 수 없습니다."));

    Department department = resolveDepartment(request.departmentId());
    validateDoctorDepartment(staff.getRole(), department);

    Department staffDepartment = staff.getRole() == StaffRole.DOCTOR ? department : null;
    staff.update(request.name().trim(), staffDepartment, request.active());

    if (hasText(request.password())) {
        staff.changePassword(passwordEncoder.encode(request.password().trim()));
    }

    if (staff.getRole() == StaffRole.DOCTOR) {
        Doctor doctor = getDoctor(staff);
        String availableDays = joinAvailableDays(request.availableDays());
        String specialty = resolveDoctorSpecialty(department);
        doctor.updateProfile(department, availableDays, specialty);
    }
}
```

이 코드의 핵심은 `전문분야`를 더 이상 화면 자유입력에 맡기지 않고, 수정 시 선택한 `부서`를 기준으로 의사 정보 전체를 다시 정리한다는 점입니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **화면 책임 정리**
  - 공통 직원 정보와 의사 전용 정보를 분리해, `부서`는 의사 전용 정보에서만 다루도록 맞췄습니다.
  - 이로써 화면 구조가 도메인 책임과 더 잘 맞게 정리됐습니다.

- **DTO/API 계약 정리**
  - 수정 요청 DTO와 admin API 요청/응답에서 `specialty`를 제거했습니다.
  - 화면과 API가 서로 다른 계약을 들고 가면 이후 유지보수가 어려워지기 때문에, 이번 단계에서 함께 정리해 두는 편이 안전했습니다.

- **서비스 보정**
  - 수정 서비스는 여전히 `Doctor` 엔티티가 `department`와 `specialty`를 함께 관리하므로, 선택한 부서를 기준으로 `specialty`도 같이 계산하도록 맞췄습니다.
  - 즉, 입력은 단순해졌지만 저장 시점의 일관성은 서비스에서 계속 보장합니다.

- **same-view 검증 UX 유지**
  - 의사 수정 시 부서가 빠진 경우 컨트롤러가 `departmentIdError`를 다시 주입하도록 정리했습니다.
  - 그래서 수정 화면에서 잘못 입력해도 redirect 없이 같은 화면에서 바로 원인을 확인할 수 있습니다.
