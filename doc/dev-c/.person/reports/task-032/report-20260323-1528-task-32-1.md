# 이번 작업 보고서: 직원 등록/수정 화면 구조 점검

- **작업 일시**: 2026-03-23 15:28 (Asia/Seoul)
- **진행 상태**: 완료

## 1. 전체 작업 흐름 (Workflow)

1. 직원 등록/수정 화면이 현재 어떤 엔티티 책임을 가정하고 있는지 먼저 확인했다.
2. `AdminStaffController`, `AdminStaffService`, 등록/수정 DTO를 읽어 `departmentId`, `specialty`, `employmentStatus`가 어디서 처리되는지 추적했다.
3. `staff-form.mustache`를 열어 공통 정보 영역과 의사 전용 정보 영역의 실제 배치를 확인했다.
4. `AdminStaffControllerTest`, `AdminStaffServiceTest`, `AdminStaffApiControllerTest`에서 현재 필드 구조를 어떤 식으로 기대하고 있는지 함께 점검했다.
5. 점검 결과를 `task-032` 메모에 반영해, 다음 단계에서 DTO/서비스 책임을 바꿀 때 어디를 함께 손대야 하는지 기준을 남겼다.

## 2. 핵심 전달 코드 (Core Logic)

```java
Staff staff = Staff.create(
        request.username().trim(),
        request.employeeNumber().trim(),
        passwordEncoder.encode(request.password().trim()),
        request.name().trim(),
        role,
        department);

if (role == StaffRole.DOCTOR) {
    Doctor doctor = Doctor.create(
            staff,
            department,
            joinAvailableDays(request.availableDays()),
            normalizeNullableText(request.specialty()));
    doctorRepository.save(doctor);
}
```

- 이 부분이 현재 문제를 가장 잘 보여준다.
- 직원 공통 엔티티를 만들 때도 `department`를 넣고, 의사 전용 엔티티를 만들 때도 다시 같은 `department`를 넣고 있다.
- 즉 화면에서 공통 정보와 의사 전용 정보가 섞여 있는 이유가 서비스 로직에도 그대로 반영돼 있다는 뜻이다.

## 3. 쉬운 비유 (Easy Analogy)

- 지금 구조는 직원 인사카드와 의사 자격카드에 같은 부서 도장을 둘 다 찍는 것과 비슷하다.
- 한 곳에서만 관리돼야 할 정보를 두 장의 카드가 동시에 들고 있으니, 화면에서도 어디에 둬야 할지 애매해진 상태다.
- 이번 점검은 먼저 도장이 어디에 중복돼 있는지 찾아서, 다음 단계에서 한 장의 카드로 책임을 정리하려는 작업이었다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **컨트롤러/폼 모델 구조**: `AdminStaffController`는 등록/수정 모두 같은 `staff-form.mustache`를 사용한다. 즉 화면 구조 변경은 등록과 수정을 따로 놀게 만들기보다 같은 폼 모델을 기준으로 함께 정리해야 한다.
- **DTO 검증 구조**: `CreateAdminStaffRequest`, `UpdateAdminStaffRequest` 모두 현재 `departmentId`를 공통 필수값으로 선언하고 있다. 따라서 단순히 템플릿에서 필드를 옮기는 것으로는 부족하고, DTO 레벨 검증 조건도 `DOCTOR` 전용 의미에 맞게 재설계가 필요하다.
- **서비스 책임 중복**: `AdminStaffService`는 `Staff.create/update`와 `Doctor.create/updateProfile` 양쪽에서 모두 부서를 사용한다. 이번 리팩토링은 사실상 화면 수정이 아니라 서비스 책임 재정렬을 포함한 구조 변경이다.
- **테스트 영향 범위**: `AdminStaffControllerTest`, `AdminStaffServiceTest`, `AdminStaffApiControllerTest` 모두 `departmentId` 공통 필수와 `specialty` 자유입력 왕복을 전제로 작성돼 있다. 다음 단계에서 테스트를 빠뜨리면 회귀가 아니라 테스트 설계 자체가 현재 구조를 붙잡는 상황이 생긴다.
- **재직 상태 위치 이슈**: `employmentStatus`는 현재 등록 화면에서만 보이고, 수정 화면에는 없다. 또한 단독 큰 select처럼 배치돼 있어 사용자가 보기엔 공통 정보보다 화면 하단 보조 섹션처럼 느껴질 수 있다. 이번 변경에서 작은 공통 필드 톤으로 옮기는 이유가 여기 있다.
