## 4. 기술 딥다이브 (Technical Deep-dive)

- **범위 테스트와 전체 테스트 분리**
  - `admin.staff` 범위 테스트는 이번 리팩토링 변경분을 직접 검증하는 데 충분했고, 이 결과는 성공했습니다.
  - 하지만 PR 직전에는 전체 `./gradlew test`를 함께 보는 편이 안전해서, 범위 밖 실패도 이번 단계에서 드러났습니다.

- **문서 완료 체크 보류 판단**
  - 구현 자체는 거의 닫혔더라도, 전체 테스트가 초록이 아니면 완료 체크를 무조건 닫아버리는 건 위험합니다.
  - 그래서 `workflow-032`, `task-032`에 “구현은 완료에 가깝지만 전체 테스트 초록은 아님”이라는 상태를 그대로 반영했습니다.

- **실패 범위 분리**
  - 이번에 남은 실패는 `admin.department`와 `domain.reservation` 쪽으로, 현재 `admin.staff` 리팩토링 파일과는 직접 연결되지 않습니다.
  - 이 구분을 문서에 남겨야 다음 작업자가 원인을 빠르게 좁힐 수 있습니다.
