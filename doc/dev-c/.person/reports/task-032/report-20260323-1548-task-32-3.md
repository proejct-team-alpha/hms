## 1. 전체 작업 흐름 (Workflow)

1. 직원 등록 화면과 생성 DTO, 생성 서비스가 `departmentId`를 공통 필드로 전제하고 있는 구조를 다시 확인했다.
2. 등록 화면만 먼저 닫기 위해 `CreateAdminStaffRequest`에서 `specialty`를 제거하고 `departmentId`를 조건부 필드로 완화했다.
3. `AdminStaffService.createStaff(...)`를 수정해 비의사 등록 시에는 `Staff.department`를 비우고, 의사 등록 시에만 `Doctor.department`와 `Doctor.specialty`를 선택한 부서 기준으로 저장하도록 정리했다.
4. `staff-form.mustache`는 create/edit 분기를 명확히 나눠서, 등록 화면에서는 공통 정보의 `부서`를 제거하고 `재직 상태`를 같은 크기의 작은 필드로 재배치했다.
5. 의사 전용 정보 영역에는 등록 화면일 때만 `부서 select`를 보여주고, 전문분야 자유입력은 제거했다.
6. 컨트롤러에서는 의사 등록 시 부서 누락이 서비스 검증으로 넘어오는 경우 `departmentIdError`를 다시 같은 화면에 주입하도록 보완했다.
7. 마지막으로 `AdminStaffControllerTest`, `AdminStaffServiceTest`를 현재 생성 계약 기준으로 정리하고 `./gradlew test --tests 'com.smartclinic.hms.admin.staff.*'`로 검증했다.

## 2. 핵심 전달 코드 (Core Logic)

```java
@Transactional
public String createStaff(CreateAdminStaffRequest request) {
    validateDuplicateUsername(request.username());
    validateDuplicateEmployeeNumber(request.employeeNumber());
    validatePassword(request.password());

    StaffRole role = resolveRequiredRole(request.role());
    Department department = resolveDepartment(request.departmentId());
    validateDoctorDepartment(role, department);

    Staff staff = Staff.create(
            request.username().trim(),
            request.employeeNumber().trim(),
            passwordEncoder.encode(request.password().trim()),
            request.name().trim(),
            role,
            null);

    if (!request.active()) {
        staff.update(staff.getName(), null, false);
    }

    adminStaffRepository.save(staff);

    if (role == StaffRole.DOCTOR) {
        Doctor doctor = Doctor.create(
                staff,
                department,
                joinAvailableDays(request.availableDays()),
                resolveDoctorSpecialty(department));
        doctorRepository.save(doctor);
    }

    return STAFF_CREATED_MESSAGE;
}
```

## 3. 초등학생도 이해할 수 있는 쉬운 비유 (Easy Analogy)

- 이번 작업은 "학교 전체 학생 카드"에 동아리 정보를 적어두던 방식을 바꿔서, "동아리 학생 카드"에만 동아리 정보를 적게 만든 것과 비슷하다.
- 모든 직원에게 부서를 먼저 적게 하면 의사가 아닌 직원도 의미 없는 값을 들고 있게 된다.
- 그래서 공통 직원 카드에는 진짜 공통 정보만 남기고, 의사일 때만 따로 열리는 카드에서 부서를 고르게 바꿨다.
- 전문분야도 직접 손으로 아무 말이나 쓰게 하지 않고, 이미 병원에 존재하는 부서 중 하나를 고르게 해서 엉뚱한 값이 들어가지 않게 만들었다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **조건부 검증**: `departmentId`는 더 이상 생성 DTO의 unconditional `@NotNull`이 아니다. 대신 서비스에서 역할이 `DOCTOR`일 때만 `validateDoctorDepartment(...)`로 확인한다. 이 방식은 "의사만 부서가 필요하다"는 도메인 규칙을 더 정확하게 반영한다.
- **SSR 단일 템플릿 분기**: 등록과 수정이 같은 `staff-form.mustache`를 쓰기 때문에, 화면을 둘로 쪼개지 않고 `{{^model.editMode}}`, `{{#model.editMode}}` 분기로 등록 쪽 레이아웃만 먼저 바꿨다. 덕분에 Task 32-3은 등록 화면을 먼저 닫고, 수정 화면은 다음 Task 32-4에서 같은 방식으로 이어서 맞출 수 있다.
- **저장 책임 보정**: 등록 서비스에서 `Staff.create(..., null)`로 바꾸고, 의사일 때만 `Doctor.create(..., department, ..., department.getName())`를 사용하도록 정리했다. 즉 등록 단계에서는 부서 책임을 의사 엔티티 쪽으로 먼저 당겨왔다.
- **테스트 계약 재정렬**: 기존 테스트는 "모든 직원이 공통 부서를 가진다"와 "전문분야를 자유입력한다"를 전제로 하고 있었다. 이번 단계에서는 생성 테스트를 다시 써서, 일반 직원 등록 시 `Staff.department == null`, 의사 등록 시 `Doctor.department`와 `Doctor.specialty`가 선택한 부서 기준으로 저장되는 흐름을 고정했다.
