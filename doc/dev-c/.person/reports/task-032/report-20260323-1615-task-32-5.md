## 1. 전체 작업 흐름 (Workflow)

1. 직원 등록과 수정 서비스가 실제로 같은 부서 책임 규칙을 쓰고 있는지 다시 확인했습니다.
2. 확인 결과 등록은 이미 `Staff.department = null` 기준으로 정리돼 있었지만, 수정은 의사일 때 `Staff.department`에 다시 부서를 넣고 있었습니다.
3. 수정 서비스와 비활성화 서비스에서 `Staff` 쪽 부서 저장을 제거하고, 공통 직원 엔티티는 이름/재직 상태만 관리하도록 정리했습니다.
4. 직원 목록 조회 쿼리도 `Staff.department` 대신 `Doctor.department`를 보도록 바꿔, 화면이 새로운 저장 규칙과 어긋나지 않게 맞췄습니다.
5. 서비스 테스트를 다시 정리해 비의사 수정 시 부서 값 무시, 의사 수정 시 `Doctor.department` 기준 갱신, 목록 조회 시 의사 부서 표시까지 확인했습니다.

## 2. 핵심 전달 코드 (Core Logic)

```java
public String updateStaff(UpdateAdminStaffRequest request) {
    Staff staff = getStaff(request.staffId());
    Department department = resolveDepartment(request.departmentId());
    validateDoctorDepartment(staff.getRole(), department);

    staff.update(request.name().trim(), null, request.active());

    if (hasText(request.password())) {
        validatePassword(request.password());
        staff.updatePassword(passwordEncoder.encode(request.password().trim()));
    }

    if (staff.getRole() == StaffRole.DOCTOR) {
        Doctor doctor = doctorRepository.findByStaffId(staff.getId())
                .orElseThrow(() -> CustomException.notFound(DOCTOR_NOT_FOUND_MESSAGE));

        doctor.updateProfile(
                department,
                joinAvailableDays(request.availableDays()),
                resolveDoctorSpecialty(department)
        );
    }

    return STAFF_UPDATED_MESSAGE;
}
```

핵심은 `Staff`에는 더 이상 부서를 저장하지 않고, 의사인 경우에만 `Doctor`가 부서와 전문분야를 함께 책임지도록 한 점입니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **엔티티 책임 정렬**
  - 등록은 이미 `Staff.department = null` 기준이었는데 수정이 그 규칙을 다시 깨고 있었습니다.
  - 이번 단계에서 수정과 비활성화까지 같은 규칙으로 맞춰, 서비스가 등록/수정에서 서로 다른 모델을 저장하지 않도록 정리했습니다.

- **조회 쿼리 보정**
  - 저장 로직만 바꾸면 직원 목록의 부서 표시가 비어 버릴 수 있어서, 목록 조회 쿼리를 `Doctor.department` 기준으로 바꿨습니다.
  - 즉 저장 규칙과 조회 규칙을 함께 움직여야 화면 회귀가 생기지 않습니다.

- **비의사 방어 로직 확인**
  - 비의사 수정 시 `departmentId`가 들어와도 실제 저장에는 영향을 주지 않도록 서비스 테스트를 강화했습니다.
  - 화면에서는 이미 의사 전용 영역이 숨겨지지만, 서비스에서도 한 번 더 안전하게 닫아두는 게 유지보수에 유리합니다.
