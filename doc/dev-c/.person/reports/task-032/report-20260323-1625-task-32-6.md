## 2. 핵심 전달 코드 (Core Logic)

```java
mockMvc.perform(get("/admin/staff/new")
                .with(user("admin").roles("ADMIN"))
                .with(csrf()))
        .andExpect(status().isOk())
        .andExpect(view().name("admin/staff-form"))
        .andExpect(content().string(containsString("name=\"departmentId\"")))
        .andExpect(content().string(not(containsString("name=\"specialty\""))));
```

핵심은 화면 구조가 바뀌었다고 말로만 남기지 않고, 실제 렌더링 결과에서 `부서 select는 남고 전문분야 입력은 사라졌다`는 점을 테스트로 고정한 것입니다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **렌더링 회귀 방지**
  - SSR 화면은 DTO만 맞아도 얼핏 동작해 보일 수 있어서, 실제 HTML에 `departmentId`가 남아 있고 `specialty`가 사라졌는지를 직접 확인하는 테스트가 중요했습니다.
  - 이 검증이 있어야 템플릿 리팩토링 이후에도 화면 구조가 뒤로 돌아가지 않습니다.

- **API 계약 고정**
  - admin API는 프런트와의 약속이기 때문에, 응답에서 제거한 필드가 다시 살아나지 않도록 `doesNotExist()` 검증을 추가했습니다.
  - 화면 계약과 API 계약을 같이 묶어두면, 한쪽만 예전 구조로 되돌아가는 문제를 줄일 수 있습니다.

- **서비스 계약 유지 확인**
  - 서비스 테스트는 의사 생성/수정, 비의사 수정, 목록 조회까지 이어지는 흐름을 다루고 있어서 이번 구조 변경의 중심축 역할을 합니다.
  - 그래서 Task 32-6에서는 새 테스트를 과하게 늘리기보다, 이미 중요한 경로가 잘 닫혀 있는지 다시 확인하는 방식으로 정리했습니다.
