## 2. 핵심 전달 코드 (Core Logic)

```java
UpdateAdminStaffRequest serviceRequest = new UpdateAdminStaffRequest(
        staffId,
        request.name(),
        request.departmentId(),
        request.password(),
        request.specialty(),
        request.availableDays()
);
```

- 이 코드는 현재 admin API도 SSR과 똑같이 `departmentId`와 `specialty`를 동시에 넘기고 있다는 걸 보여준다.
- 즉 화면만 고치면 끝나는 작업이 아니라, API 계약과 서비스 요청 객체도 같이 정리해야 일관성이 생긴다.
- 이번 단계에서 중요한 건 "어디까지가 공통 정보이고 어디부터가 의사 전용 정보인가"를 코드 단위로 고정한 점이다.

- 아직 서류 양식을 완전히 바꾸진 않았지만, 다음 단계에서 어디를 지워야 하고 어디를 남겨야 하는지는 명확해진 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)

- **조건부 검증 필요성**: 현재 `CreateAdminStaffRequest`, `UpdateAdminStaffRequest`는 `departmentId`를 무조건 필수로 두고 있다. 하지만 도메인상 `departmentId`는 `DOCTOR`일 때만 의미 있으므로, 다음 단계에서 unconditional `@NotNull`을 제거하고 역할 기반 검증으로 옮겨야 한다.
- **API 계약 포함 범위**: `AdminStaffApiController`와 `UpdateAdminStaffApiRequest/Response`도 `specialty`와 `departmentId`를 전제로 한다. 따라서 이번 리팩토링은 SSR 폼만의 문제가 아니라 admin API 계약까지 포함한다고 보는 게 맞다.
- **폼 응답 DTO 영향**: `AdminStaffFormResponse`는 현재 `selectedDepartmentId`, `specialty`, `active`를 함께 들고 있다. 다음 단계에서는 `selectedDepartmentId`를 의사 전용 필드로만 해석하고, `specialty`는 제거 대상이라는 전제를 갖고 움직여야 한다.
- **서비스 책임 재정렬**: `AdminStaffService`는 아직 `Staff.create/update`와 `Doctor.create/updateProfile` 양쪽에서 부서를 받는다. 화면 변경 전에 이 중복을 인식하고, 이후 구현에서 `Staff`와 `Doctor` 책임을 어떻게 나눌지 먼저 고정한 것이 이번 단계의 핵심이다.
- **재직 상태 위치**: `employmentStatus`는 여전히 `Staff` 공통 정보의 일부다. 따라서 `active` boolean 표현은 유지하되, 화면에서는 등록/수정 모두 공통 정보 안의 작은 필드로 옮기는 것이 가장 자연스럽다는 결론을 냈다.
