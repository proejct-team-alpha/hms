## 2. 핵심 전달 코드 (Core Logic)
```java
@Test
@DisplayName("page links preserve status and keyword")
void pageLinks_preserveStatusAndKeyword() {
    AdminReservationListResponse result = adminReservationService.getReservationList(2, 5, "RECEIVED", "Kim");

    assertThat(result.previousUrl()).contains("page=1").contains("status=RECEIVED").contains("keyword=Kim");
    assertThat(result.nextUrl()).contains("page=3").contains("status=RECEIVED").contains("keyword=Kim");
    assertThat(result.pageLinks()).allSatisfy(link -> {
        assertThat(link.url()).contains("size=5");
        assertThat(link.url()).contains("status=RECEIVED");
        assertThat(link.url()).contains("keyword=Kim");
    });
}
```

## 3. 쉽게 설명하기 (Easy Analogy)
- 이번 작업은 검색창을 만든 뒤에 “다음 페이지로 넘어가면 방금 입력한 검색 조건을 잊어버리지 않는지”를 확인하는 점검 작업과 비슷했다.
- 예약 목록과 환자 목록 모두 검색어를 들고 다음 페이지, 이전 페이지, 페이지 번호 링크를 이동해야 하므로, 이번 테스트는 마치 엘리베이터에서 층을 바꿔도 손에 들고 있던 메모가 계속 같이 따라오는지 확인하는 역할을 한다.
- 중간에 예약 취소 API 테스트가 함께 깨진 것은, 점검하러 갔더니 옆문이 아예 막혀 있던 상황과 비슷해서, 이번 기회에 그 문도 다시 열어둔 셈이다.

## 4. 기술 딥다이브 (Technical Deep-dive)
- **검색 조건 유지 페이지네이션 검증**: 검색형 목록 화면에서 가장 자주 놓치는 부분은 조회 결과 자체보다 링크 유지다. 이번 단계에서는 `previousUrl`, `nextUrl`, `pageLinks`, `statusOptions`에 `keyword`와 `status`가 계속 실리는지를 서비스 레벨에서 직접 검증해, 화면 템플릿이 아니라 URL 생성 로직 자체를 안전하게 닫았다.
- **기존 테스트 자산 재사용**: Task 31-7의 일부 항목은 이미 Task 31-3, 31-5, 31-6 과정에서 채워진 상태였다. 그래서 컨트롤러 테스트를 또 새로 늘리기보다, 남아 있던 공백인 페이지네이션 유지 시나리오를 중심으로 보강해 중복 없이 마무리했다.
- **범위 테스트를 막는 선행 결함 정리**: `admin.reservation.*` 범위 테스트가 검색 작업과 직접 무관한 이유로 실패하고 있었는데, 원인은 `AdminReservationApiController`의 취소 엔드포인트가 주석 처리되어 있던 것이었다. 이런 경우 테스트를 좁혀 피하는 대신, 범위 테스트 자체가 신뢰 가능하도록 선행 결함을 같이 바로잡는 편이 이후 작업에도 훨씬 도움이 된다.
- **문서 체크리스트와 실제 커버리지 동기화**: Task 문서를 체크할 때는 단순히 “오늘 만든 테스트”만이 아니라, 이전 단계에서 이미 들어간 테스트까지 합쳐 현재 기준으로 충족되는지를 보는 편이 더 정확하다. 이번에도 Task 31-7은 새 테스트 추가 + 기존 커버리지 재확인 + 범위 테스트 통과까지 기준으로 완료 처리했다.
