## 1. 전체 작업 흐름 (Workflow)
1. 환자 목록이 현재 `nameKeyword`, `contactKeyword` 두 파라미터를 기준으로 움직이고 있는지 컨트롤러, 서비스, 리포지토리, 템플릿, 테스트를 다시 확인했다.
2. 목록 요청 계약을 `keyword` 하나로 단순화하고, 컨트롤러가 더 이상 이름/연락처를 따로 받지 않도록 정리했다.
3. 서비스는 `keyword` 하나를 받아서 이름 검색용 값과 연락처 비교용 값을 내부에서 각각 정규화하도록 바꿨다.
4. 리포지토리 쿼리는 `이름 LIKE OR 정규화된 연락처 LIKE` 구조로 정리해 단일 입력으로 두 검색을 모두 처리하게 만들었다.
5. 템플릿은 두 개의 입력창을 하나의 검색창으로 줄이고, 검색 요약 영역도 `검색어` 하나만 표시하도록 최소 변경했다.
6. 컨트롤러/서비스 테스트를 현재 계약에 맞게 다시 정리하고, 이름 검색과 연락처 검색이 각각 동작하는지 확인했다.

public AdminPatientListResponse getPatientList(int page, int size, String keyword) {
    int safePage = page < 1 ? DEFAULT_PAGE : page;
    int safeSize = size < 1 ? DEFAULT_SIZE : size;
    String normalizedKeyword = normalizeKeyword(keyword);
    String normalizedContactKeyword = normalizeContact(keyword);

    Pageable pageable = PageRequest.of(safePage - 1, safeSize);
    Page<Patient> pageResult = adminPatientRepository.search(
            normalizedKeyword,
            normalizedContactKeyword,

}

## 4. 기술 딥다이브 (Technical Deep-dive)
- **검색 계약 단순화**: 컨트롤러와 응답 DTO에서 `nameKeyword`, `contactKeyword`를 없애고 `keyword` 하나로 고정했다. 이 작업으로 이후 UI 통일과 URL 유지 규칙이 훨씬 단순해진다.
- **서비스 내부 정규화 분리**: 외부 계약은 단순하지만, 서비스 내부에서는 이름 검색용 `trim()` 값과 연락처 비교용 하이픈 제거 값을 따로 만든다. 사용자에게는 하나의 입력창만 보여주고, 구현 내부에서 복합 처리를 흡수한 구조다.
- **Repository OR 검색 전환**: 기존의 `name AND contact` 성격에서 `name OR phone` 성격으로 바꾸어, 사용자가 한 입력값으로도 충분히 찾을 수 있게 했다. 환자 목록은 찾기용 인덱스 화면이라는 목적에 더 잘 맞는다.
- **UI 최소 변경 전략**: 이번 단계에서는 환자 목록을 단일 검색창으로 먼저 바꾸는 데 집중하고, 전체 검색 바 톤을 예약 목록/규칙 관리와 완전히 맞추는 일은 다음 Task 31-6으로 남겨두었다. 이렇게 나누면 계약 변경과 UI 리팩토링을 분리해서 안정적으로 진행할 수 있다.
