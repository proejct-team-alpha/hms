## 1. 전체 작업 흐름 (Workflow)
1. 예약 목록의 현재 파라미터 구조를 기준으로 `keyword`를 추가해도 기존 `status`, `page`, `size` 흐름이 깨지지 않는지 먼저 점검했다.
2. 컨트롤러에서 `GET /admin/reservation/list`와 취소 후 redirect 모두 `keyword`를 받을 수 있도록 연결했다.
3. 서비스에서 검색어를 이름 검색용과 연락처 검색용으로 나눠 정규화하고, 상태 필터와 조합되는 URL 생성 규칙까지 함께 확장했다.
4. Repository 쿼리에 `patient.name LIKE`와 `REPLACE(patient.phone, '-', '') LIKE` 조건을 추가해 단일 `keyword`로 이름/연락처 검색이 가능하게 만들었다.
5. 예약 취소 후에도 사용자가 보던 검색 상태를 잃지 않도록 hidden field와 redirect 파라미터 유지까지 반영했다.
6. 컨트롤러/서비스 테스트를 보강하고, 리다이렉트 URL 인코딩 차이로 흔들리던 테스트는 ASCII keyword 기준으로 안정화했다.

public AdminReservationListResponse getReservationList(int page, int size, String statusParam, String keyword) {
    int safePage = page < 1 ? DEFAULT_PAGE : page;
    int safeSize = size < 1 ? DEFAULT_SIZE : size;
    String normalizedKeyword = normalizeKeyword(keyword);
    String normalizedPhoneKeyword = normalizePhoneKeyword(keyword);

    ReservationStatus status = resolveStatus(statusParam);
    Pageable pageable = PageRequest.of(
            safePage - 1,
            safeSize,
            Sort.by(Sort.Order.desc("reservationDate"), Sort.Order.desc("timeSlot")));

    Page<AdminReservationRepository.AdminReservationListProjection> pageResult =
            adminReservationRepository.findReservationListPage(
                    status,
                    normalizedKeyword,
                    normalizedPhoneKeyword,
                    pageable);
}

## 3. 쉽게 설명하기 (Easy Analogy)
- 이번 작업은 도서관에서 책을 찾을 때 "분야 필터"와 "검색창"을 같이 쓰는 것과 비슷하다.
- 예전에는 "상태"라는 책장만 고를 수 있었는데, 이제는 그 책장 안에서 "홍길동"이나 "0101234"처럼 환자 이름이나 연락처 일부를 같이 검색할 수 있게 된 셈이다.
- 중요한 점은 책장을 넘겨도, 즉 페이지를 이동하거나 취소 후 목록으로 돌아와도 방금 입력했던 검색어를 계속 들고 다니게 만든 것이다.

## 4. 기술 딥다이브 (Technical Deep-dive)
- **Controller 파라미터 확장**: `AdminReservationController`가 `keyword`를 목록 조회와 취소 redirect 모두에서 유지하도록 바뀌었다. 이로써 사용자가 검색한 상태에서 액션을 해도 화면 문맥이 유지된다.
- **Service 정규화 책임**: 서비스가 `trim()` 기준 이름 검색어와 하이픈 제거 기준 연락처 검색어를 각각 만들도록 분리했다. 덕분에 UI는 검색어 하나만 보내도 되고, 검색 로직의 복잡도는 서비스 내부에서 흡수된다.
- **Repository OR 검색 조합**: JPA 쿼리에 `name LIKE OR normalized phone LIKE` 조건을 추가해 단일 `keyword`로 환자명과 연락처를 동시에 찾도록 만들었다. 상태 필터는 그대로 AND 조건으로 묶어서 의도치 않은 넓은 조회를 막았다.
- **테스트 안정화**: redirect URL은 실제 실행 시 인코딩이 개입되므로, 컨트롤러 테스트는 ASCII keyword 기준으로 맞춰서 환경 차이로 흔들리지 않게 정리했다. 서비스 테스트는 이름 검색, 연락처 검색, 상태+검색 조합을 각각 따로 검증해 회귀 지점을 분리했다.
