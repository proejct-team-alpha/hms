# Task 31-1 예약/환자 목록 검색 구조 점검 보고서

- 작업 일시: 2026-03-23 (Asia/Seoul)

- 상태: 완료

## 1. 작업 배경
예약 목록에 검색 기능을 추가하고, 환자 목록 검색 방식도 단일 `keyword`로 통일하려면 두 화면의 현재 파라미터 구조와 템플릿 패턴을 먼저 정리할 필요가 있었다. 이번 단계에서는 예약 목록, 환자 목록, 병원 규칙 관리 검색 바를 함께 비교해 다음 단계 설계 기준을 고정했다.

## 3. 점검 결과
### 3-1. 예약 목록
- 현재 `GET /admin/reservation/list`는 `page`, `size`, `status`만 받고 있다.
- 서비스와 URL 생성 로직도 `status`만 유지하고 있어 `keyword`를 추가하려면 컨트롤러, 서비스, 리포지토리, 템플릿을 함께 수정해야 한다.
- 리포지토리 projection에는 이미 `patient.name`, `patient.phone`가 포함되어 있어 검색 조건 추가 여지는 충분하다.
- 템플릿은 상태 필터만 있고 검색 바가 없어, 병원 규칙 관리 패턴으로 상단 레이아웃을 재구성해야 한다.

### 3-2. 환자 목록
- 현재 `GET /admin/patient/list`는 `nameKeyword`, `contactKeyword`를 분리해서 받는다.
- 서비스도 이름/연락처를 별도로 정규화하고, URL 역시 두 파라미터를 각각 유지한다.
- 리포지토리는 이름 검색과 연락처 정규화 검색을 이미 지원하므로, 단일 `keyword`로 바꿀 때는 OR 검색으로 합치는 방향이 자연스럽다.
- 템플릿은 입력창이 두 개라서, 단일 검색창과 간결한 검색 요약 구조로 정리할 필요가 있다.

### 3-3. 기준 패턴
- 병원 규칙 관리 검색 바는 `검색 입력 1개 + 필터 + 조회/초기화 버튼 + 결과 요약` 구조가 이미 정리되어 있다.
- 예약 목록은 여기에 상태 필터를 결합하는 방식으로, 환자 목록은 단일 검색창 중심의 가벼운 형태로 맞추는 것이 적절하다고 판단했다.

## 4. 다음 단계 연결 포인트
- 예약 목록은 `keyword` 검색 계약을 먼저 고정한 뒤 백엔드 검색 구현으로 들어간다.
- 환자 목록은 예약 목록 패턴이 정리된 후 `nameKeyword/contactKeyword -> keyword` 통합으로 이어간다.

## 5. 검증 결과

- 별도 테스트 실행은 없음

## 6. 메모
- 예약 목록을 먼저 기준 화면으로 정리하고, 환자 목록은 같은 패턴으로 이어서 맞추는 순서가 가장 안정적이다.
