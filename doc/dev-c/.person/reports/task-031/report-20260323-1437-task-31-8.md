## 4. 기술 딥다이브 (Technical Deep-dive)
- **문서와 구현 상태 동기화**: workflow/task 문서는 단순 기록이 아니라 다음 작업자와 리뷰어가 현재 상태를 이해하는 기준점이다. 이번에는 Task 31-1부터 31-8까지의 진행 결과를 실제 코드/테스트 상태와 다시 맞추면서, 완료 기준이 추상적 체크리스트가 아니라 현재 프로젝트에서 재현 가능한 사실이 되도록 정리했다.
- **범위 테스트와 전체 테스트 분리 검증**: 먼저 `admin.reservation`, `admin.patient` 범위 테스트를 돌려 직접 관련된 검색 작업이 안전한지 확인하고, 그 다음 전체 테스트를 다시 돌리는 순서로 검증했다. 이렇게 하면 문제 발생 시 원인 범위를 더 빨리 좁힐 수 있고, 마지막에는 전체 회귀도 확인할 수 있다.
- **검색 통합 작업의 마무리 기준 정리**: 예약 목록은 `status + keyword`, 환자 목록은 `keyword` 단일 검색이라는 서로 다른 계약을 갖고 있지만, 화면 UX는 같은 검색형 목록 패턴으로 통일했다. 문서에서도 이 차이와 공통점을 함께 명시해, 이후 누가 봐도 왜 두 화면이 비슷하면서도 완전히 같지는 않은지 이해할 수 있게 했다.
- **부수 수정 포함 여부 명시**: Task 31-7 과정에서 예약 취소 API 엔드포인트를 복구한 것도 이번 작업 문서에 남겼다. 이런 부수 수정은 기능 요약에서 빠지기 쉬운데, 범위 테스트를 통과시키는 데 실제로 영향을 준 부분이라서 문서에도 남겨두는 편이 안전하다.
