# Task 31-2 예약 목록 keyword 검색 설계 보고서

- 작업 일시: 2026-03-23 (Asia/Seoul)
- 작업 범위: 예약 목록 단일 `keyword` 검색 계약 설계
- 상태: 완료

## 1. 작업 배경
예약 목록 검색 기능을 구현하기 전에 `keyword`의 의미, 검색 대상, 상태 필터와의 조합 방식, URL 유지 규칙을 먼저 고정할 필요가 있었다. 이번 단계에서는 실제 구현 전에 계약을 문서로 못 박아 다음 단계 변경 범위를 줄이는 데 집중했다.

## 2. 설계 결정
### 2-1. 요청 파라미터
- `page=1`
- `size=10`
- `status=ALL`
- `keyword=''`

### 2-2. 검색 대상
- `patient.name`
- `patient.phone`

### 2-3. 검색 규칙
- `status=ALL`이면 상태 조건 없이 조회
- `keyword`가 비어 있으면 검색 조건 없이 조회
- 값이 있으면 `상태 AND (이름 OR 연락처)` 조합으로 조회

### 2-4. 정규화 규칙
- 이름 검색용 `keyword`: trim만 적용
- 연락처 비교용 `keyword`: 하이픈 제거 후 비교

### 2-5. URL 유지 규칙
- `previousUrl`
- `nextUrl`
- `pageLinks`
- `statusOptions`
- 취소 후 redirect

위 모든 링크/이동 흐름이 현재 `keyword`를 유지하도록 설계 방향을 고정했다.

## 3. 구현 영향 범위
- `AdminReservationController`
  - `keyword` 파라미터 추가
- `AdminReservationService`
  - `getReservationList(page, size, status, keyword)`로 시그니처 확장
  - `buildListUrl(page, size, status, keyword)` 형태로 URL 생성 확장
- `AdminReservationRepository`
  - 상태 + 이름/연락처 검색을 함께 처리하는 쿼리 확장
- `reservation-list.mustache`
  - 검색창, 조회 버튼, 초기화 버튼, hidden field 추가
- 취소 폼
  - 현재 검색 상태 유지를 위한 `keyword` hidden field 추가

## 4. 설계 판단 이유
- 상태는 이미 필터로 분리되어 있으므로, 검색은 환자명/연락처 중심의 단일 `keyword`가 가장 단순하고 운영 UX에 잘 맞는다.
- 이름/연락처를 별도 입력으로 나누지 않으면 입력 비용이 줄고, 병원 규칙 관리 검색 바 패턴과도 톤이 맞아진다.
- 페이지 이동과 상태 탭 이동에서 `keyword`를 함께 유지해야 검색형 목록 화면으로서 일관성이 생긴다.
