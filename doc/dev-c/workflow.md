# AdminDashboardStatsService 리팩터링 워크플로우

## 문서 목적
`AdminDashboardStatsService`의 가독성을 높이고, 서비스 계층에 섞인 집계 책임(특히 카테고리별 아이템 개수 계산)을 DB 집계로 이전하기 위한 작업 기준과 실행 순서를 정의한다.

## 범위
- 대상: `AdminDashboardStatsService` 및 관련 DTO/Repository/테스트
- 목표: 메서드 분리 + 읽기 쉬운 구조 + DB 중심 집계
- 실행 방식: 요청마다 실시간 계산
- 허용 범위: 작은 동작 수정(버그성 의심 로직 정리) 허용

## 리팩터링 원칙
1. 서비스는 오케스트레이션 중심으로 유지한다.
2. 단일 도메인 집계(예: 카테고리별 COUNT)는 DB(`GROUP BY`)로 처리한다.
3. 서비스 public 메서드는 흐름이 보이도록 단계형 private 메서드로 분리한다.
4. 쿼리 조건/필터/정렬 기준은 서비스와 DB 간 불일치가 없도록 명시한다.
5. 테스트는 Given-When-Then 구조로 보강한다.

## DB vs 서비스 처리 기준
- DB에서 처리할 것
  - 카운트/합계/평균/최소/최대
  - 그룹화 집계(`GROUP BY`)
  - 정렬/페이징/필터링 등 데이터 축소 연산
  - 대량 데이터에서 루프 기반 후처리를 대체 가능한 경우
- 서비스에서 처리할 것
  - 유스케이스 흐름 제어
  - 여러 리포지토리 결과 조합
  - 권한/검증/예외 변환
  - 표현 계층 DTO 매핑 및 비즈니스 규칙 적용

## 실행 순서
1. 현재 `AdminDashboardStatsService`에서 메서드별 책임을 식별하고, 집계/조합/표현 로직을 분류한다.
2. 카테고리별 아이템 개수 계산 로직을 DB 집계 쿼리로 이전한다.
3. 서비스 public 메서드를 "조회 → 집계 결과 수집 → 응답 조립" 단계로 분리한다.
4. 중복 조건문/매직값/분기 로직을 private 메서드 또는 상수로 추출한다.
5. 관련 DTO를 점검해 결과 구조를 명확히 하고, 필요 시 record/정적 팩토리로 정리한다.
6. Repository 계층에 집계 메서드를 추가하고 네이밍을 도메인 기준으로 맞춘다.
7. 테스트(기존/신규)를 작성·수정해 집계 정합성과 예외 시나리오를 검증한다.
8. 리팩터링 전후 결과 비교(주요 케이스)로 동작 동일성 + 허용된 소규모 수정 반영 여부를 확인한다.

## 수용 기준
- [ ] 서비스 메서드가 책임 단위로 분리되어 가독성이 개선된다.
- [ ] 카테고리별 아이템 개수 계산이 DB 집계로 처리된다.
- [ ] 요청마다 실시간 계산 동작이 유지된다.
- [ ] 기존 기능 회귀가 없고, 허용된 범위 내 소규모 동작 수정만 반영된다.
- [ ] 관련 테스트가 통과한다.
