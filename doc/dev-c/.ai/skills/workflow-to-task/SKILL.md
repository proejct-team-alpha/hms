# 워크플로우 분해 스킬

## 개요

`workflow` 문서의 큰 구현 범위를 실제 개발 가능한 작은 작업 묶음으로 나누고, 체크리스트 중심의 `task` 문서로 저장한다.
산출물은 항상 `doc/dev-c/task/task-{workflow 번호}.md` 형식으로 작성하며, 이 문서는 실제 구현 순서를 잡는 실행 문서로 사용한다.

## 입력 해석 규칙

1. 대상 workflow를 먼저 특정한다.
   - 사용자가 번호를 주면 해당 번호를 그대로 사용한다.
   - 사용자가 파일 경로를 주면 해당 파일을 기준으로 작업한다.
   - 사용자가 명확히 지정하지 않았지만 대화 맥락상 특정 workflow가 분명하면 그 파일을 추론해 사용한다.
2. 입력 문서는 우선 `doc/dev-c/workflow/workflow-{번호}.md`에서 찾는다.
3. 출력 문서는 항상 `doc/dev-c/task/task-{번호}.md`로 만든다.
4. 같은 번호의 task 문서가 이미 있으면 새 파일을 또 만들지 말고, 현재 workflow 기준으로 업데이트하거나 재작성한다.

## 작업 절차

1. 대상 workflow 문서를 읽고 다음 정보를 추출한다.
   - 작업 주제 또는 문서 목적
   - 현재 상태 점검 결과
   - 보완 구현 범위 또는 실행 순서
   - 수용 기준
   - 우선순위
2. 큰 작업을 실제 구현 가능한 작은 작업 묶음으로 분해한다.
3. 각 작업 묶음을 체크 가능한 세부 step으로 다시 나눈다.
4. 최종 결과를 `task-{번호}.md` 형식으로 저장한다.
5. 저장 후에는 어떤 workflow를 기준으로 어떤 task 파일을 만들었는지 사용자에게 짧게 알려준다.

## 분해 원칙

- 한 덩어리 작업이 너무 크면 반드시 더 작게 나눈다.
- 각 Task 묶음은 "집중해서 한 번에 구현하고 검증 가능한 수준"이 되도록 만든다.
- 하나의 Task 안에 서로 다른 큰 책임이 섞이면 분리한다.
- 화면 복구, 책임 정리, API/서비스 구현, 템플릿 반영, 테스트 추가처럼 의존성이 보이는 단위로 나눈다.
- 세부 체크리스트는 실제로 완료 여부를 확인할 수 있는 문장으로 쓴다.
- `구현한다`, `정리한다` 같은 큰 표현만 두지 말고, 무엇을 어떻게 바꾸는지 드러나게 쪼갠다.
- 테스트나 검증이 중요한 기능이면 별도 Task로 분리하거나 마지막 단계에 명시적으로 포함한다.
- 현재 상태 복구가 선행되어야 하면 복구 작업을 가장 앞 Task로 둔다.

## 세부 작업 크기 기준

- 각 Task 묶음은 보통 3개에서 8개 사이의 체크 항목으로 구성한다.
- 세부 체크 항목은 보통 1개 파일 수정, 1개 책임 정리, 1개 검증 포인트처럼 명확한 단위로 쓴다.
- 한 체크 항목 안에 `및`, `그리고`, `전체`, `전부`가 반복되면 너무 큰 항목일 가능성이 높으니 다시 나눈다.
- "기능 전체 구현"처럼 끝이 큰 문장은 피하고, "파라미터 추가", "뷰 모델 계산", "렌더링 반영", "테스트 작성"처럼 행동이 보이게 쓴다.

## 문서 구조 규칙

출력 문서는 아래 구조를 기본으로 사용한다.

````markdown
# task-017

## 목적
- [ ] `workflow-017`의 범위를 구현/검증 단위로 나눈다.
- [ ] ...

## Task 17-1. 첫 번째 작업 묶음 제목
- [ ] ...
- [ ] ...

## Task 17-2. 두 번째 작업 묶음 제목
- [ ] ...
- [ ] ...

## 완료 기준
- [ ] ...

## 추천 진행 순서
- [ ] Task 17-1 ...
- [ ] Task 17-2 ...

## 메모
- [ ] `workflow-017`은 보완 명세, `task-017`은 실제 구현 단위 분할 문서로 사용
````

문서 제목의 번호는 `task-017`처럼 3자리 형식을 유지한다.
세부 Task 제목은 `Task 17-1.`처럼 상위 번호에서는 앞의 0을 제거한 형식을 사용한다.

## 섹션 매핑 규칙

- workflow의 `문서 목적`, `추가 작업 주제`, `보완 구현 범위 핵심 요약`은 `## 목적`으로 옮긴다.
- workflow의 `현재 상태 점검 결과`, `판단`, 선행 이슈는 필요 시 `## 메모`에 녹인다.
- workflow의 `보완 구현 범위`, `실행 순서`, 레이어별 구현 항목은 여러 `## Task N-M.` 섹션으로 분해한다.
- workflow의 `수용 기준`은 `## 완료 기준`으로 옮긴다.
- workflow의 `우선순위`가 있으면 `## 추천 진행 순서`로 바꾼다.
- workflow에 우선순위가 없더라도 의존성을 기준으로 추천 순서를 반드시 작성한다.

## 품질 기준

- 각 체크 항목은 완료 여부를 예/아니오로 판단할 수 있어야 한다.
- Task 제목만 읽어도 전체 구현 순서가 보이도록 구성한다.
- 한 Task는 선행 조건이 너무 많지 않게 유지한다.
- 가능하면 마지막에는 테스트 또는 검증 Task를 둔다.
- workflow의 수용 기준이 누락되어 있으면 작업 내용을 바탕으로 측정 가능한 완료 기준을 보강한다.
- task 문서는 구현자가 바로 따라갈 수 있는 실행 계획이어야 하며, 추상적인 요약문으로 끝나면 안 된다.

## 프로젝트 기준 참고

- 형식이 애매하면 `doc/dev-c/task/task-017.md`의 스타일을 우선 참고한다.
- workflow 문서 예시는 `doc/dev-c/workflow/workflow-017.md`처럼 현재 프로젝트 문서 패턴을 따른다.
- 이 스킬의 목적은 workflow를 다시 요약하는 것이 아니라, 실제 작업 가능한 체크리스트 문서로 바꾸는 것이다.
