# 작업 보고서 작성 스킬

## 개요

현재 세션에서 방금 끝난 작업을 분석해, 의도와 과정, 결과, 배운 점이 드러나는 한글 보고서로 정리한다.
보고서는 task 기반 작업이면 `doc/dev-c/.person/reports/task-{번호}/` 아래에 저장하고, task 번호를 알 수 없을 때만 `doc/dev-c/.person/reports/{feature-name}/`를 사용한다. 채팅 답변보다 저장된 파일을 최종 산출물로 취급한다.

## 작업 절차

1. 먼저 현재 세션에서 보고서에 필요한 맥락을 모은다.
   - 최근 사용자 요청, 변경된 파일, 실행한 테스트, 구현 메모를 우선 확인한다.
   - 세션 내용만으로 변경 이유를 설명하기 어렵다면 코드나 diff를 직접 확인한다.
2. 저장 폴더 이름을 정한다.
   - 현재 작업이 `doc/dev-c/task/task-{번호}.md`를 기준으로 진행 중이면 저장 폴더는 반드시 `task-{번호}`로 정한다.
   - 현재 작업이 `Task 17-1`, `Task 17-2`처럼 세부 step 중심이더라도 상위 task 번호를 추출해 `task-017` 같은 폴더명을 사용한다.
   - task 번호를 확실히 알 수 없을 때만 기존 방식대로 `{feature-name}` 폴더를 만든다.
   - `{feature-name}`이 필요한 경우 사용자가 기능명을 명확히 준 이름을 우선 사용하고, 없으면 중심 기능이나 모듈 기준으로 짧은 kebab-case 이름을 만든다.
   - 예: `task-017`, `staff-login-fix`, `reservation-status-api`
3. 보고서 상태를 정한다.
   - 구현과 검증이 끝났으면 `완료`
   - 코드 변경은 있었지만 검증이 부족하면 `검증 필요`
   - 작업이 아직 끝나지 않았으면 `진행 중`
4. 아래에 정의된 섹션 순서를 그대로 지켜 한국어 보고서를 작성한다.
5. 대상 디렉터리가 없으면 생성한 뒤, 같은 폴더 안에 새 리포트 파일을 추가한다.
6. 저장 후에는 사용자에게 저장 위치와 보고서 핵심 내용을 짧게 알려준다.

## Task 우선 저장 규칙

- 세션이나 사용자 요청에 `task-017`, `task-028`, `doc/dev-c/task/task-017.md` 같은 식별자가 보이면 해당 번호를 최우선으로 사용한다.
- `Task 17-1`, `Task 17-2`, `17-3 단계`처럼 세부 작업만 언급되어도 상위 task 번호는 `task-017`로 통일한다.
- 같은 task 번호를 여러 번 작업하더라도 새 기능 폴더를 만들지 말고, 같은 task 폴더에 계속 누적 저장한다.
- task 번호가 분명한데 feature 이름도 함께 보인다면 폴더명은 feature 이름이 아니라 task 번호를 우선한다.
- task 번호를 전혀 특정할 수 없을 때만 `{feature-name}` 폴더 저장 규칙으로 fallback 한다.

## 필수 출력 형식

보고서는 아래 구조를 그대로 따라 Markdown으로 작성한다.

````markdown
# 🚩 작업 보고서: [작업명]

- **작업 일시**: YYYY-MM-DD
- **진행 단계**: [진행 중 / 완료 / 검증 필요]

## 1. 🌊 전체 작업 흐름 (Workflow)

1. ...
2. ...

## 2. 🧩 핵심 코드 (Core Logic)

```language
// 짧은 한글 주석을 포함한 핵심 코드
```

## 3. 🍦 상세비유 쉬운 예시를 들어서 (Easy Analogy)

- "이번 작업은 ... 와 같습니다."

## 4. 📚 기술 딥다이브 (Technical Deep-dive)

- **기술명**: 쉬운 설명
````

이모지 섹션 제목과 전체 순서는 그대로 유지한다.

## 작성 원칙

- 작업 내용을 단순 나열하지 말고, 시작부터 끝까지 흐름으로 설명한다.
- 초보자도 읽을 수 있는 쉬운 한국어를 우선하되, 기술적 정확성은 유지한다.
- 무엇을 바꿨는지만 쓰지 말고 왜 그렇게 했는지도 함께 설명한다.
- 이해에 도움이 된다면 실제 파일명, 클래스명, 메서드명, 사용자 영향 범위를 구체적으로 적는다.
- 함수나 클래스가 20줄 이상이면 전체를 길게 붙이지 말고 핵심 부분만 발췌해 보여준다.
- 핵심 코드에는 짧은 한글 주석을 넣어 각 줄의 의미를 설명한다.
- 긴 원본 코드를 그대로 붙이는 것보다 요약된 코드가 더 잘 전달되면 요약본을 사용한다.
- 코드 변경이 없었다면 분석, 리뷰, 설계 중심 작업이었다는 점을 명시한다.
- 어려운 기술 개념은 별도 설명 항목으로 분리해 독립적으로 이해할 수 있게 한다.

## 핵심 코드 작성 가이드

- 변경의 본질이 드러나는 가장 짧은 코드 조각을 고른다.
- 비즈니스 규칙, 분기 처리, 검증, 상태 변경, API 계약이 드러나는 코드를 우선한다.
- 로직이 바로 읽히지 않으면 코드 안에 짧은 한글 주석을 직접 달아준다.
- 여러 파일이 중요하더라도 대표 코드 1개를 중심으로 보여주고, 나머지는 문장으로 설명한다.

## 쉬운 비유 작성 가이드

- 실제 코드 동작과 맞는 일상 비유를 사용한다.
- 초등학생도 따라올 수 있을 만큼 단순하고 친숙한 비유를 고른다.
- 비유로 끝내지 말고, 실제 시스템에서 무엇에 대응되는지도 연결해서 설명한다.

## 기술 딥다이브 작성 가이드

- 이번 작업에 실제로 사용된 어려운 기술 개념만 설명한다.
- 예: 트랜잭션, 검증, Mock, AJAX, BCrypt, AOP, REST, 템플릿 렌더링
- 설명 순서는 "이 기술이 무엇인지 -> 이번 작업에서 왜 필요했는지 -> 어떤 효과가 있었는지"를 따른다.
- 모든 키워드를 나열하기보다 2개에서 4개 정도의 핵심 개념만 고른다.

## 파일 저장 규칙

- task 기반 작업이면 항상 `doc/dev-c/.person/reports/task-{번호}/` 경로를 우선 사용한다.
- task 번호를 알 수 없을 때만 `doc/dev-c/.person/reports/{feature-name}/` 경로를 사용한다.
- 디렉터리가 없으면 먼저 생성한다.
- 리포트 파일명은 덮어쓰기 대신 새 파일로 저장한다.
- 기본 파일명은 `report-YYYYMMDD-HHmm.md` 형식을 사용한다.
- 세부 step이 분명하면 `report-YYYYMMDD-HHmm-task-17-1.md`처럼 step 정보를 뒤에 붙일 수 있다.
- 같은 `task-017`을 여러 번 작업했다면 모든 리포트 파일은 `doc/dev-c/.person/reports/task-017/` 폴더 안에 모여 있어야 한다.
- 채팅에만 출력하지 말고 저장된 파일을 기준 산출물로 삼는다.
