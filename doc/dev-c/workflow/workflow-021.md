# 예약 관리 화면 내 접수 필터 통합 명세서 (workflow-021)

## 문제 정의

전체 접수 목록(19) 기능을 새 화면으로 분리하지 않고, 기존 관리자 예약 관리 화면(`/admin/reservation/list`) 안에서 상태 필터 기반으로 함께 관리한다.

## 현재 판단

- 현재도 `/admin/reservation/list?page=1&size=10&status=RECEIVED` 형태로 접수 상태를 확인할 수 있다.
- 접수는 업무적으로 당일 현황 확인에 가깝지만, 사용자 입장에서는 예약과 접수를 한 화면에서 함께 관리하는 편이 더 직관적이다.
- 따라서 이번 단계에서는 별도 `/admin/reception/list` 화면을 만들지 않고, 기존 예약 목록 화면을 확장하는 방향이 더 적절하다.

## 대상 사용자

- 관리자(`ROLE_ADMIN`)

## 확정 방향

1. 별도 `AdminReceptionController`는 이번 단계에서 만들지 않는다.
2. 접수 현황은 기존 `GET /admin/reservation/list` 안에서 상태 필터로 관리한다.
3. 상태 필터 라벨은 다른 화면/문서와 용어를 맞추기 위해 `접수`로 사용한다.
4. UI 라벨 `접수`는 내부 상태값 `RECEIVED`에 매핑한다.
5. 예약 관리 화면은 아래 필터 구성을 사용한다.
   - 전체
   - 예약
   - 접수
   - 완료
   - 취소

## 전달 요구사항

1. 예약 목록 상태 필터에 `접수` 라벨을 명확히 반영한다.
2. `접수` 선택 시 내부적으로 `RECEIVED` 상태 목록을 보여준다.
3. 페이징 구조는 기존 예약 목록 패턴을 유지한다.
4. 접수 상태에서도 기존 페이지네이션과 성공/실패 메시지 흐름이 깨지지 않도록 유지한다.
5. 취소 처리 후에도 현재 선택한 상태 필터와 페이지 정보가 유지되도록 한다.
6. 예약 목록 화면 설명 문구와 필터 문구가 사용자 관점에서 자연스럽게 읽히도록 정리한다.

## 이번 단계에서 하지 않는 것

- `GET /admin/reception/list` 신규 URL 생성
- `AdminReceptionController` 신규 생성
- 접수 전용 Mustache 화면 분리
- 접수 전용 DTO/서비스 별도 분리
- 날짜 필터 추가

## 제약 조건

- 기존 URL prefix(`/admin/**`)를 유지한다.
- 기존 `AdminReservationController`, `AdminReservationService`, `reservation-list.mustache` 구조를 최대한 재사용한다.
- 컨트롤러는 `model + pageTitle` 패턴을 유지한다.
- 접수 필터 도입으로 예약 관리 화면의 기존 상태 필터 동작이 깨지지 않아야 한다.
- 현재 프로젝트의 관리자 목록 페이지네이션 패턴과 일관성을 유지한다.

## 엣지 케이스

- `status=RECEIVED` 직접 요청 시에도 동일하게 접수 목록이 보여야 한다.
- 잘못된 상태 파라미터는 기존과 동일하게 안전하게 무시되거나 기본값으로 처리한다.
- 접수 필터에서 취소 가능한 항목과 불가능한 항목 표시는 기존 규칙을 유지한다.
- 접수 필터 상태에서 취소 처리 후 목록 복귀 시 page/size/status가 유지되는지 확인한다.

## 수용 기준

- [ ] `/admin/reservation/list` 상태 필터에 `접수`가 자연스럽게 노출된다.
- [ ] `접수` 필터 선택 시 `RECEIVED` 상태 예약 목록이 조회된다.
- [ ] 별도 `/admin/reception/list` 없이 기존 예약 관리 화면에서 접수 현황 확인이 가능하다.
- [ ] 페이지네이션과 필터 선택 상태가 함께 유지된다.
- [ ] 관련 컨트롤러/서비스/템플릿 테스트가 통과한다.

## 구현 순서

1. 현재 예약 목록 상태 필터/상태 라벨 구조 점검
2. `RECEIVED` 상태 라벨을 `접수` 기준으로 정리
3. `reservation-list.mustache` 설명 문구와 상태 필터 UI 정리
4. 필요 시 컨트롤러/서비스 테스트에서 `RECEIVED` 필터 기대값 정리
5. 범위 테스트 및 전체 테스트 재확인

## 향후 분리 기준 메모

아래 요구가 추가되면 이후 단계에서 `/admin/reception/list` 전용 화면 분리를 다시 검토한다.

- 접수 전용 통계 카드
- 실시간 모니터링/자동 새로고침
- 대기 순번 강조
- 접수 완료 중심 액션 강화
- 당일 접수 전용 보드형 레이아웃
