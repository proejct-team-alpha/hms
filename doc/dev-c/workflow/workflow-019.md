# 유효성 검증 및 SSR 에러 표시 명세서 (workflow-019)

## 문제 정의
직원과 진료과의 관리자 입력 폼에 기본 유효성 검증을 추가한다. `@NotBlank`, `@NotNull` 기반으로 빈 필드와 공백 입력을 막고, 검증 실패 시 SSR 폼 화면에서 에러 메시지와 기존 입력값이 유지되도록 처리한다.

## 대상 사용자
- 관리자(`ROLE_ADMIN`)

## 대상 범위
1. 직원 등록 폼
2. 직원 수정 폼
3. 진료과 목록 인라인 등록 폼
4. 진료과 상세/수정 폼

## 전달 요구사항
1. 입력 DTO에 `@Valid` 또는 `@Validated` 검증을 적용한다.
2. 기본 검증 범위는 빈 필드와 공백 입력 방지 중심으로 제한한다.
3. 검증 실패 시 같은 SSR 폼 화면을 다시 렌더링한다.
4. 검증 실패 시 필드별 에러 메시지를 표시한다.
5. 검증 실패 시 사용자가 입력한 기존 값을 유지한다.

## 화면별 검증 대상
### 직원 등록
- 이름 필수
- 부서 필수
- 로그인 아이디 필수
- 비밀번호 필수

### 직원 수정
- 이름 필수
- 부서 필수
- 비밀번호는 선택 입력

### 진료과 등록
- 진료과명 필수

### 진료과 수정
- 진료과명 필수

## 제약 조건
- 이번 단계에서는 `@NotBlank`, `@NotNull` 중심으로만 적용한다.
- 길이 제한과 패턴 검증은 다음 단계로 미룬다.
- API와 SSR을 동시에 확장하지 않고, 우선 SSR 입력 UX에 집중한다.
- 기존 URL 구조와 PRG 흐름은 유지하되, 검증 실패 시에는 redirect보다 폼 재렌더링을 우선 검토한다.
- 직원/진료과 폼 모두 에러 메시지와 입력값 유지 방식이 일관되도록 정리한다.

## 엣지 케이스
- 공백만 입력한 이름 제출
- 직원 등록 시 로그인 아이디 미입력
- 직원 등록 시 비밀번호 미입력
- 직원 등록/수정 시 부서 미선택
- 진료과 등록 인라인 폼에서 이름 미입력
- 진료과 수정 화면에서 이름 미입력

## 기대 효과
- 빈 값 제출 시 서버에서 일관되게 차단한다.
- 사용자가 입력한 값이 유지되어 재입력이 편해진다.
- 직원과 진료과 폼의 검증 UX 패턴이 통일된다.
- 다음 단계의 길이 제한/패턴 검증 확장 기반을 마련한다.

## 수용 기준
- [ ] 직원 등록 폼에서 필수값 누락 시 에러 메시지가 표시된다.
- [ ] 직원 수정 폼에서 필수값 누락 시 에러 메시지가 표시된다.
- [ ] 진료과 등록 폼에서 진료과명 누락 시 에러 메시지가 표시된다.
- [ ] 진료과 수정 폼에서 진료과명 누락 시 에러 메시지가 표시된다.
- [ ] 검증 실패 시 기존 입력값이 유지된다.
- [ ] 관련 SSR 컨트롤러와 폼 테스트가 통과한다.

## 구현 순서
1. 직원 등록/수정 입력 DTO의 필수 필드 검증 어노테이션 추가
2. 진료과 등록/수정 입력 DTO 또는 요청 객체의 필수 필드 검증 정리
3. SSR 컨트롤러에서 `BindingResult` 처리 추가
4. 검증 실패 시 폼 재렌더링과 기존 입력값 유지 구조 적용
5. Mustache 템플릿에 필드별 에러 메시지 영역 추가
6. 직원/진료과 컨트롤러 테스트 보강
