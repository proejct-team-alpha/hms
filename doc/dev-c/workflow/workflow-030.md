# 규칙 등록/수정 DTO 기본 검증 및 SSR 오류 메시지 정리 명세서 (workflow-030)

## 문제 정의
관리자 규칙 등록/수정 화면의 유효성 검증과 오류 메시지 계약을 문서 기준으로 코드와 화면에 일치시킨다.  
이번 작업은 새로운 복잡 검증을 추가하는 것이 아니라, 문서에 이미 정의된 기본 검증 규칙과 메시지를 DTO 어노테이션, 컨트롤러 SSR 흐름, Mustache 화면에 맞춰 정리하는 것이 목적이다.

## 대상 URL
- `GET /admin/rule/new`
- `POST /admin/rule/new`
- `GET /admin/rule/detail?ruleId={id}`
- `POST /admin/rule/update`

## 비채택 범위 / 제외 사항
- 커스텀 validator 도입
- 메시지 리소스 분리 또는 국제화 구조 변경
- 삭제/목록/상세 UX 개편
- 추가 비즈니스 규칙 검증
- REST API 계약 변경

## 구현 범위
1. 등록 DTO와 수정 DTO에 동일한 기본 검증 규칙 적용
2. `title`, `content`, `category` 필드 기준 문서 계약 반영
3. 등록/수정 화면에 동일한 오류 메시지 계약 반영
4. 검증 실패 시 SSR same-view 재렌더링과 입력값 유지 보장
5. 카테고리 목록 등 폼 재구성에 필요한 옵션 데이터 재세팅
6. 컨트롤러/서비스 테스트에 빈 값, 길이 초과, category 누락 케이스 보강

## 검증 계약

### 대상 필드
- `title`
- `content`
- `category`

### 규칙
- `title` 필수
- `title` 최대 200자
- `content` 필수
- `content` 최대 3000자
- `category` 필수

### 메시지 계약
- `제목을 입력해 주세요.`
- `내용을 입력해 주세요.`
- `제목은 200자 이하로 입력해 주세요.`
- `내용은 3000자 이하로 입력해 주세요.`
- `카테고리를 선택해 주세요.`

## DTO 설계 기준
- 등록 DTO와 수정 DTO 모두 같은 검증 규칙을 가진다.
- 이번 범위에서는 메시지 문자열을 DTO 어노테이션에 직접 선언한다.
- 메시지 소스 분리보다 계약의 명확성과 재작업 감소를 우선한다.

## SSR 실패 처리 계약

### 등록 실패
- 반환 뷰: `admin/rule-new`
- same-view 재렌더링
- 입력값 유지
- 필드별 오류 메시지 표시
- 카테고리 목록 등 옵션 데이터 재세팅

### 수정 실패
- 반환 뷰: `admin/rule-detail`
- same-view 재렌더링
- 입력값 유지
- 필드별 오류 메시지 표시
- 카테고리 목록 등 옵션 데이터 재세팅

### 공통 원칙
- `@Valid` + `BindingResult` 실패 시 원래 폼 뷰를 다시 반환한다.
- SSR 화면은 `errors`, `inputData`, 필요 시 `errorMessage`를 함께 사용한다.
- 화면 분리는 현재 구조를 유지한다.
  - 등록: 별도 화면
  - 수정: 상세/수정 통합 화면

## 제약 조건
- 등록 실패는 반드시 등록 화면으로 돌아가야 한다.
- 수정 실패는 반드시 상세/수정 통합 화면으로 돌아가야 한다.
- 기존 규칙 삭제/목록/상세 진입 흐름과 충돌하지 않아야 한다.
- invalid category 별도 친절 메시지 처리 규칙이 이미 있으면, 이번 기본 검증 계약과 충돌하지 않도록 유지한다.

- `content` 빈 문자열 또는 공백
