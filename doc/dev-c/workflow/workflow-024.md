# 환자 상세 화면 인라인 수정 연동 명세서 (workflow-024)

## 문제 정의
이미 구현된 관리자 환자 수정 API를 실제 관리자 화면에서 사용할 수 있도록 환자 상세 화면에 인라인 수정 UI를 추가한다.

## 대상 화면
- `GET /admin/patient/detail?patientId={id}`

## 연동 API
- `POST /admin/api/patients/{id}/update`

## 수정 가능 필드
- 이름
- 연락처
- 메모(`note`)

## 읽기 전용 필드
- 환자 ID
- 이메일
- 주소

## 구현 방향
1. 환자 상세 화면의 `기본 정보` 카드 안에 `정보 수정` 버튼을 둔다.
2. 버튼 클릭 시 별도 페이지 이동 없이 같은 카드 안에서 읽기 모드와 수정 모드를 전환한다.
3. 저장은 `fetch` 기반 비동기 호출로 처리한다.
4. 성공 시 화면의 표시값을 즉시 갱신하고 성공 메시지를 보여준다.
5. 실패 시 필드 에러 또는 카드 상단 에러 메시지를 보여준다.

## 에러 처리 규칙
- `VALIDATION_ERROR`
  - `name` 공백: 이름 필드 아래 에러 표시
  - `phone` 공백: 연락처 필드 아래 에러 표시
- `DUPLICATE_PATIENT_PHONE`
  - 연락처 필드 아래 에러 표시
- `RESOURCE_NOT_FOUND`
  - 카드 상단 에러 메시지 표시
- 그 외
  - 일반 실패 메시지 표시

## 제약 조건
- 목록 화면은 검색 중심 인덱스 역할을 유지한다.
- 수정 UI는 상세 화면에서만 제공한다.
- 기존 SSR 상세 조회와 예약 이력 렌더링은 유지한다.
- JavaScript는 `const` 우선, `async/await` 중심으로 작성한다.

## 수용 기준
- [x] 환자 상세 화면에서 `정보 수정` 버튼으로 인라인 수정 모드 전환이 가능하다
- [x] 이름, 연락처, 메모만 수정 가능하다
- [x] 저장 시 기존 `POST /admin/api/patients/{id}/update` API를 호출한다
- [x] 성공 시 상세 화면 값이 즉시 갱신되고 성공 메시지가 노출된다
- [x] 이름/연락처 검증 실패와 전화번호 중복이 화면에서 바로 표시된다
- [x] 관련 `admin.patient` 범위 테스트가 통과한다

## 구현 점검 결과
- 환자 상세 화면에 읽기/수정 전환 UI 추가
- 수정 성공 후 화면 표시값 즉시 갱신
- `note` 편집값은 원본 값을 사용하고, 표시용 `-`와 분리
- `admin.patient` 범위 렌더링 테스트 통과
