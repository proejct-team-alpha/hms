# 환자 관리 목록·상세 화면 명세서 (workflow-022)

## 문제 정의
관리자 환자 관리 기능을 구현한다. 환자 관리의 메인 화면은 `GET /admin/patient/list` 환자 목록 화면으로 두되, 전체를 많이 보여주는 대시보드가 아니라 이름/연락처 기반으로 환자를 찾는 인덱스 화면으로 구성한다. 환자 상세 정보와 전체 예약 이력은 `GET /admin/patient/detail?patientId={id}` 상세 화면에서 별도로 제공한다.

## 현재 판단
- 예약 목록 화면에서도 환자 이름과 연락처를 확인할 수 있지만, 예약 화면은 예약 건 기준 목록이라 같은 환자가 여러 예약을 가진 경우 중복 표시된다.
- 환자 목록은 환자 1명당 1줄로 보여주는 환자 기준 화면이어야 한다.
- 따라서 환자 목록과 환자 상세는 별도 `AdminPatientController`에서 구현하는 것이 적절하다.
- 예약 목록에서는 환자 상세로 이동하는 진입점(환자명 클릭 또는 상세보기 링크)을 추가할 수 있지만, 예약 목록 자체가 환자 목록을 대체하지는 않는다.

## 대상 사용자
- 관리자(`ROLE_ADMIN`)

## 확정 방향
1. 환자 관리 메인 화면은 `GET /admin/patient/list` 환자 목록 화면으로 둔다.
2. 환자 목록 화면은 검색 중심 인덱스 화면으로 구성한다.
3. 환자 상세는 `GET /admin/patient/detail?patientId={id}`에서 별도 제공한다.
4. 역할은 아래처럼 분리한다.
   - 목록: 환자 찾기
   - 상세: 환자 기본 정보 보기 + 전체 예약 이력 확인
5. 환자 목록 기본 페이지 크기는 20건으로 한다.
6. 기본 정렬은 최신 등록순(`createdAt desc`)을 우선 검토한다.
7. 검색 조건은 이름과 연락처를 우선 제공한다.

## 전달 요구사항
1. `AdminPatientController`를 추가한다.
2. `GET /admin/patient/list`
   - 이름/연락처 검색 제공
   - 기본 size=20 페이징 제공
   - 환자 1명당 1줄 목록 렌더링
3. `GET /admin/patient/detail?patientId={id}`
   - 환자 기본 정보 표시
   - 해당 환자의 전체 예약 이력 표시
4. 환자 목록 화면은 아래 중심으로 구성한다.
   - 페이지 제목
   - 설명 문구
   - 검색 영역(이름, 연락처)
   - 결과 요약(총 건수, 현재 검색 조건)
   - 환자 목록 테이블
   - 페이지네이션
5. 환자 목록 테이블 컬럼은 최소 아래를 포함한다.
   - 환자명
   - 연락처
   - 등록일
   - 관리(상세보기)
6. 환자 상세 화면은 최소 아래를 포함한다.
   - 이름
   - 연락처
   - 이메일
   - 주소
   - 메모
   - 전체 예약 이력
7. 예약 목록 화면에서는 이후 단계에서 환자명 클릭을 환자 상세 페이지로 연결할 수 있도록 고려한다.

## 이번 단계에서 하지 않는 것
- 환자 목록 화면에 대시보드형 통계 카드 추가
- 환자 목록 화면에 예약 이력 미리보기 추가
- 환자 목록 화면에 상세 정보 패널 동시 노출
- 환자 수정/삭제 기능 추가
- 환자 전용 REST API 추가

## 제약 조건
- 기존 URL prefix(`/admin/**`)를 유지한다.
- 로컬 구조 문서 기준으로 `admin.patient` 범위에서 구현한다.
- 환자 목록은 예약 목록의 변형이 아니라 환자 기준 목록이어야 한다.
- 목록 화면은 무겁지 않게 검색과 페이징 중심으로 유지한다.
- 상세 화면에서 전체 예약 이력을 제공하되, 목록 화면에 이력을 과도하게 노출하지 않는다.
- 연락처 검색은 하이픈 포함/미포함 입력 모두 고려할 수 있도록 설계한다.
- 기존 관리자 목록 화면의 `model + pageTitle` 패턴과 페이지네이션 UI 패턴을 최대한 따른다.

## 엣지 케이스
- 같은 환자가 여러 예약을 가진 경우 목록에는 한 번만 보여야 한다.
- 예약 이력이 없는 환자가 존재하는 경우에도 목록/상세가 동작해야 한다.
- 연락처 검색 시 `01012345678`와 `010-1234-5678` 입력 차이를 어떻게 처리할지 정리해야 한다.
- 존재하지 않는 `patientId` 상세 요청은 404 또는 프로젝트 규칙에 맞는 에러 화면으로 처리해야 한다.
- 검색 결과가 없을 때 빈 목록 메시지가 자연스럽게 보여야 한다.

## 수용 기준
- [ ] `/admin/patient/list`에서 이름/연락처 검색이 가능하다.
- [ ] `/admin/patient/list`에서 기본 20건 페이징이 적용된다.
- [ ] 환자 목록은 환자 1명당 1줄로 렌더링된다.
- [ ] `/admin/patient/detail?patientId={id}`에서 환자 기본 정보가 보인다.
- [ ] 환자 상세 화면에서 해당 환자의 전체 예약 이력이 보인다.
- [ ] 환자 목록은 인덱스 화면 역할에 집중하고, 상세 정보는 상세 화면으로 분리된다.
- [ ] 관련 컨트롤러/서비스/템플릿 테스트가 통과한다.

## 구현 순서
1. 현재 환자/예약 엔티티 및 조회 가능 구조 점검
2. `AdminPatientController`와 목록/상세 서비스 응답 구조 설계
3. 환자 목록 검색 + 페이징 구현
4. 환자 상세 + 전체 예약 이력 구현
5. 목록/상세 템플릿 구현
6. 컨트롤러/서비스 테스트 보강
7. 예약 목록에서 환자 상세 진입점 추가 여부 후속 검토

## 향후 확장 메모
아래 요구가 생기면 이후 단계에서 별도 확장 검토가 가능하다.
- 환자 수정/메모 편집
- 최근 방문/최근 접수 요약 카드
- 환자별 통계(누적 예약 수, 최근 방문일)
- 예약 목록에서 환자 상세 바로가기 강화
- 환자 검색 고도화(이메일, 메모, 복합 검색)
