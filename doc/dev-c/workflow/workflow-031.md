# 예약 목록 검색 추가 및 환자 목록 검색 통합 명세서 (workflow-031)

## 문제 정의
관리자 예약 목록 화면에 환자명/연락처 기준 단일 `keyword` 검색을 추가하고, 환자 목록 화면도 같은 검색 UX로 통일한다. 두 화면의 검색/필터 UI는 병원 규칙 관리 화면의 검색 바 패턴을 공통 기준으로 맞춘다.

## 대상 화면
- `GET /admin/reservation/list`
- `GET /admin/patient/list`

## 현재 판단
- 예약 목록은 상태 필터만 있어 특정 환자의 예약/접수 건을 빠르게 찾기 어렵다.
- 환자 목록은 `nameKeyword`, `contactKeyword`를 분리해서 받아 검색 UX가 관리자 다른 목록 화면과 어긋난다.
- 관리자 검색형 목록 화면의 패턴을 맞추려면 예약 목록과 환자 목록 모두 단일 `keyword` 검색으로 정리하는 편이 자연스럽다.
- 구현 순서는 예약 목록을 먼저 기준 화면으로 정리하고, 같은 패턴을 환자 목록에 이어서 적용하는 방식이 가장 안정적이다.

## 확정 방향
1. 예약 목록에 단일 `keyword` 검색을 추가한다.
2. 예약 목록 검색 대상은 아래 두 필드로 제한한다.
   - 환자명
   - 연락처
3. 상태는 기존처럼 별도 필터로 유지한다.
4. 환자 목록은 기존 `nameKeyword`, `contactKeyword`를 제거하고 단일 `keyword` 검색으로 통일한다.
5. 예약 목록과 환자 목록의 상단 검색/필터 UI는 병원 규칙 관리 화면 패턴으로 맞춘다.
6. 조회 / 초기화 버튼, 검색 입력창 배치, 페이지네이션 조건 유지 방식도 같은 흐름으로 정리한다.

## 예약 목록 구현 범위
1. `GET /admin/reservation/list`
2. 요청 파라미터
   - `page` (기본값 `1`)
   - `size` (기본값 `10`)
   - `status` (기본값 `ALL`)
   - `keyword` (optional)
3. 검색 대상
   - `patient.name LIKE`
   - `patient.phone LIKE`
4. 상태 필터와 `keyword`는 함께 조합 가능해야 한다.
5. 페이지 이동 시 `status`, `keyword`를 유지해야 한다.
6. 상단 검색 UI는 병원 규칙 관리 검색 바 패턴을 따른다.

## 환자 목록 구현 범위
1. `GET /admin/patient/list`
2. 요청 파라미터
   - `page` (기본값 `1`)
   - `size` (기본값 `20`)
   - `keyword` (optional)
3. 기존 `nameKeyword`, `contactKeyword`는 제거한다.
4. 검색 대상
   - 환자명
   - 연락처
5. 연락처는 하이픈 포함/미포함 입력 차이를 고려해 정규화 조회를 지원한다.
6. 상단 검색 UI는 병원 규칙 관리와 예약 목록과 같은 패턴으로 정리한다.

## UI 방향
- 검색 입력창 placeholder는 두 화면 모두 아래처럼 정리한다.
  - 예약 목록: `환자명 또는 연락처 검색`
  - 환자 목록: `환자명 또는 연락처 검색`
- 상태 필터는 예약 목록에서만 유지한다.
- 환자 목록은 상태 변경 팝업 없이 검색형 인덱스 화면에 집중한다.
- 조회 버튼과 초기화 버튼은 병원 규칙 관리와 동일한 위치와 흐름으로 맞춘다.
- 검색 결과 요약과 페이지네이션도 현재 관리자 목록 패턴을 유지한다.

## 제약 조건
- 기존 관리자 URL prefix(`/admin/**`)를 유지한다.
- SSR 목록 화면 기준으로 구현한다.
- 예약 목록은 기존 상태 필터 동작이 깨지지 않아야 한다.
- 환자 목록 상세 진입 흐름은 그대로 유지한다.
- 서비스는 `@Transactional(readOnly = true)` 기본 원칙을 유지한다.
- 테스트는 Given-When-Then 구조를 유지한다.

## 엣지 케이스
- `keyword`가 비어 있거나 공백만 있으면 전체 조회로 처리한다.
- 예약 목록에서 `status`가 잘못된 값이면 `ALL`로 정규화한다.
- 환자 목록 연락처 검색은 `01012345678`, `010-1234-5678` 입력 모두 동일하게 조회되도록 지원한다.
- 검색 결과가 없어도 검색 UI와 빈 목록 메시지가 자연스럽게 보여야 한다.
- 페이지 이동 시 현재 검색 조건이 유실되지 않아야 한다.

## 수용 기준
- [x] 예약 목록에서 `keyword` 검색이 가능하다.
- [x] 예약 목록에서 환자명으로 검색할 수 있다.
- [x] 예약 목록에서 연락처로 검색할 수 있다.
- [x] 예약 목록에서 상태 필터와 검색어를 함께 적용할 수 있다.
- [x] 환자 목록은 단일 `keyword` 검색으로 동작한다.
- [x] 환자 목록에서 기존 `nameKeyword`, `contactKeyword` 없이도 동일한 검색 경험을 제공한다.
- [x] 예약 목록과 환자 목록의 검색 UI가 병원 규칙 관리 패턴과 유사한 흐름으로 정리된다.
- [x] 관련 컨트롤러/서비스/리포지토리 테스트가 통과한다.

## 구현 순서
1. 예약 목록 현재 구조 점검
2. 예약 목록 `keyword` 검색 서비스/리포지토리 추가
3. 예약 목록 템플릿 검색 바를 병원 규칙 관리 패턴으로 정리
4. 환자 목록 파라미터를 `keyword` 단일 검색으로 통합
5. 환자 목록 서비스/리포지토리/템플릿을 같은 패턴으로 정리
6. 컨트롤러/서비스 테스트 보강
7. 문서와 검증 결과 정리

## 구현 점검 결과
- 예약 목록은 `status + keyword` 조합 검색, 페이지네이션, 취소 후 목록 복귀에서 `keyword`를 유지하도록 정리했다.
- 환자 목록은 `keyword` 하나로 이름/연락처 검색을 처리하고, 상단 검색 바도 예약/규칙 관리 목록과 같은 패턴으로 맞췄다.
- 예약/환자 검색형 목록의 페이지 링크 유지 시나리오를 서비스 테스트로 닫았다.
- 범위 테스트를 다시 확인하는 과정에서 주석 처리돼 있던 예약 취소 API 엔드포인트도 복구해 `admin.reservation` 범위 테스트가 안정적으로 통과하도록 맞췄다.

## 리뷰 포인트
- 예약 목록 검색 기준을 환자명/연락처 단일 `keyword`로 단순화한 방향이 운영 UX에 적절한지
- 환자 목록 `nameKeyword` + `contactKeyword`를 `keyword` 하나로 합친 것이 검색 경험을 충분히 유지하는지
- 병원 규칙 관리 검색 바 패턴을 관리자 검색형 목록의 공통 UI 기준으로 삼아도 자연스러운지
