# 직원 수정 REST API 구현 명세서 (workflow-016)

## 문제 정의
직원 수정 기능을 JSON 기반 REST API로 제공한다. `POST /api/staff/{id}` 요청으로 직원 정보를 수정하고, 성공 시 `Resp.ok(...)` 공통 응답 형식으로 결과를 반환한다.

## 대상 사용자
- 내부 시스템 사용자
- 권한 정책은 기존 보안 규칙에 따름

## 전달 요구사항
1. `POST /api/staff/{id}` 직원 수정 API 구현
2. JSON 요청/응답 기반 처리
3. 성공 응답은 `Resp.ok(...)` 공통 래퍼 사용
4. 수정 가능 항목
   - 이름
   - 부서
   - 비밀번호(입력 시만 변경)
   - `DOCTOR`인 경우 전문 분야
   - `DOCTOR`인 경우 진료 가능 요일
5. 수정 불가 항목
   - 로그인 아이디
   - 사번
   - 역할
   - 재직 상태
6. SSR 수정 기능과 동일한 비즈니스 규칙 유지

## 제약 조건
- 기존 SSR 수정 기능은 유지
- 서비스 로직은 최대한 재사용
- 예외는 기존 `CustomException` + `GlobalExceptionHandler` 포맷 유지
- 응답 DTO 네이밍은 `...Request`, `...Response` 규칙 준수
- API Controller와 SSR Controller는 분리

## 엣지 케이스
- 존재하지 않는 직원 ID
- 잘못된 부서 ID
- 비밀번호 공백 입력
- `DOCTOR`가 아닌데 의사 전용 필드가 들어오는 경우
- `DOCTOR`인데 연결된 `Doctor` 엔티티가 없는 경우

## 수용 기준
- [ ] `POST /api/staff/{id}` 요청으로 직원 수정이 가능하다
- [ ] 성공 시 `Resp.ok(...)` 응답이 반환된다
- [ ] 이름/부서/선택적 비밀번호 변경이 동작한다
- [ ] `DOCTOR` 직원은 전문 분야/진료 가능 요일 수정이 가능하다
- [ ] 수정 불가 항목은 API에서 변경되지 않는다
- [ ] 관련 API 테스트가 통과한다

## 구현 순서
1. 기존 SSR 직원 수정 로직과 DTO를 검토해 재사용 가능한 서비스 메서드 범위를 정리한다
2. API 요청/응답 DTO를 설계한다
3. `AdminStaffApiController`를 추가하거나 기존 API 컨트롤러 구조에 맞춰 수정 엔드포인트를 구현한다
4. 서비스 레이어에서 SSR과 동일한 수정 규칙을 재사용하도록 연결한다
5. API 테스트를 작성해 성공/예외 흐름을 검증한다
